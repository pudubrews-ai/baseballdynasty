## Executive Verdict

**NOT READY.**

The v0.2.0 spec and the Architect's eval-0 leave a foundational unit-of-time ambiguity unresolved that propagates into nearly every new subsystem (waivers, call-ups, send-downs, firings, service time, trade posture, prospect dev). The engine tracks **two completely different "game" counters** — a league-wide `current_game_number` (1–500) and a per-team `games_played` (0–50) — and the spec's repeated phrases "game 30," "game 35," "every 5 games," "every 10 games," "3 games of sim time," and "172 games" do not say which one they mean. The Architect's binding rulings (D5, D8, D17, G6, G8, G13) silently assume `current_game_number`, which produces wrong cadences and a waiver window that can expire in a single tick. Until this is nailed down, the Developer cannot build a correct call-up/waiver/firing/service-time system, and at least three test groups (G4, G5, G8) cannot deterministically pass.

In addition, there is a hard prerequisite the spec assumes but the codebase does not provide: **minor league games are never simulated**, so minor leaguers have zero `season_stats`. Multiple spec triggers ("OPS < .580 over last 15 AB" for a AAA replacement, hot streaks, "minors live stats") and test G6 ("minors returns live stats: games, ab, avg or ip, era") are unsatisfiable against the current data model.

Total findings below: **4 Critical, 9 High, 7 Medium, 4 Low.**

## CRITICAL FINDINGS

### AB-01 (Critical) — The "game N" unit is ambiguous and the Architect's resolution is wrong for half the systems

**Root cause:** Two independent counters exist and the spec/eval conflate them.

**Evidence (code):**
- `season.ts:208-216` `getNextGame` returns the next game whose `gameNumber > league.current_game_number`. Game numbers are **league-wide and sequential, 1..500** (`season.ts:179-184`: `game.gameNumber = i + 1` over all 500 games). Each league game advances `current_game_number` by exactly 1 (`game.ts:452`).
- `teams.games_played` is the **per-team** counter, incremented +1 for each team in a game (`game.ts:439-447`). After the full 500-game season, each team's `games_played` = 50.
- `season.ts:229-255` `shouldFireTradeDeadline` deliberately computes per-team game counts and fires when "10 teams have played >= 35 games" — i.e. it uses the **per-team** scale, NOT `current_game_number`.

**Evidence (spec/eval conflict):**
- v0.2.0 spec "Trade Deadline Logic" says "evaluate trade posture at game 30" and "trades happen at game 33-37." Architect D9 says "`teams.trade_posture` populated at game 30" and "trade execution runs once per game tick in window [30,37]." Window [30,37] of **what**? If `current_game_number`, that is the 30th–37th league game out of 500 — i.e., the first ~7% of the season, when every team has played 3–4 games. If per-team `games_played`, that is mid-season (game 35 of 50). These differ by a factor of ~10.
- Architect G8/D5 say call-up eval runs when "`current_game_number - last_call_up_check_game >= 5`." That is every 5 **league** games = roughly every half-round of the league. Across 20 teams a given team plays its own 5th game only around league game ~50. So "evaluated every 5 games" per the spec ("MLB roster drops below 23... evaluated every 5 games") is intended per-team but the eval fires league-wide — a team gets ~10× more eval passes than the spec implies, OR (if a team only gets evaluated on ticks where it played) the cadence is undefined because the per-team-played count is never tracked on the league clock.

**Impact:** Trade posture/deadline, call-up cadence, send-down cadence, firing cadence, prospect dev cadence, service-time accrual, and waiver windows are ALL specified in this ambiguous unit. Picking the wrong interpretation silently breaks every one. The existing `shouldFireTradeDeadline` already uses per-team scale, so a new `trade_posture` keyed off `current_game_number=30` would fire posture evaluation BEFORE most teams have played 3 games and ~5 league-games before the existing deadline marker is even close — the two trade systems would be on different clocks. **This must be resolved with one explicit rule (recommend: all "game N" thresholds are per-team `games_played`, all "every N games" cadences are per-team) before any build.**

### AB-02 (Critical) — Minor league games are not simulated; minor-leaguer stats are always empty, so call-up triggers and test G6 are unsatisfiable

**Root cause:** Only MLB-roster players produce box scores. There is no minor-league game sim.

**Evidence (code):**
- `game.ts:57-60` `selectLineup` and `:97-100` `selectStartingPitcher` both filter `is_on_mlb_roster = 1`. `simulateGame` only generates batter/pitcher lines for these players. `upsertBatterStats`/`upsertPitcherStats` (`game.ts:849-893`) are called only from those lines.
- There is no code path anywhere that simulates AAA/AA/A/Rookie games or writes `season_stats` rows for `is_on_mlb_roster = 0` players.

**Evidence (spec/test):**
- Call-up trigger: "A starter has OPS < .580 over last 15 AB **and AAA replacement is rated higher**" and "Starting pitcher ERA > 7.00... **and AAA arm available**." Send-down: "better option in AAA." Prospect dev: "Hot streaks (5+ game hit streak in minors)." All of these compare or read minor-league statistical performance.
- Test G6: "GET /api/teams/:id/minors returns **live stats** (games, ab, avg or ip, era) not just depth chart." Test spec D12 minors response shape includes `season_stats: {games_played, at_bats, hits, home_runs, battingAvg}`.

**Impact:** The Architect's G9 rolling-window plan (`recent_ab`, `recent_hits` on `season_stats`) only addresses MLB players. There is no source of `games`, `ab`, `avg`, `ip`, `era` for minor leaguers. Either (a) the Developer must build a minor-league sim (large, unscoped, not in any phase), or (b) minors stats are synthesized/faked, in which case "AAA replacement rated higher" reduces to a pure rating comparison (ignoring the OPS/ERA clause) and G6 "live stats" cannot show real `avg`/`era`. The spec promises statistical depth the engine cannot produce. **Resolve before build:** either descope minors live-stats to rating-only triggers (and weaken G6) or scope a minor-league sim explicitly.

### AB-03 (Critical) — Waiver "3 games of sim time" can expire (or never evaluate) in a single turbo burst; the claim-at-expiry model races the tick loop

**Root cause:** Turbo sims many games per macrotask; the waiver window is measured in `current_game_number` increments but claims are evaluated only at the exact expiry tick.

**Evidence (code):**
- `engine.ts:247-273`: turbo uses `setImmediate` and yields only every 5 games (`turboGameCount % 5`). Between yields, 5 league games (5 `current_game_number` increments) run with no interleaving. There is NO per-tick transaction wrapping game + maintenance (contrary to Architect R6/D17 which assume one) — `runGameTick` (`engine.ts:368-405`) calls `simulateGame` (which opens its OWN transaction at `game.ts:409`) and does nothing else transactional.
- Architect G6/D4: waiver entry sets `claim_game_window_end = current_game + 3`; "Claims are evaluated EXACTLY ONCE at expiration." Architect D5 ordering runs waiver-expiration as step 2 of `runRosterMaintenance` after each game.

**Evidence (spec):** "Claim window: 3 games of sim time." "DFA'd player appears on waiver wire within same sim tick" (test G4).

**Impact / attack:**
1. **Window in league games vs per-team games:** "3 games of sim time" almost certainly means "3 days a player sits on waivers." In MLB that is real days. Here, 3 `current_game_number` increments = ~0.3% of a season, often a few milliseconds in turbo. A claim window of "3 league games" is essentially instantaneous — the worst-team's GM may not have a maintenance pass scheduled in that span. If it instead means "3 per-team games," there is no per-team game clock to compare against (same root issue as AB-01).
2. **Exactly-once-at-expiry race:** If `claim_game_window_end` is set when `current_game_number = X`, the expiry tick fires when `current_game_number = X+3`. But `current_game_number` only advances inside `simulateGame`, and a game can be **skipped** without writing a claim evaluation (the missing-SP / box-score-fail paths at `game.ts:253-258` and `:392-400` advance `current_game_number` and `return` BEFORE any roster maintenance hook). On a skipped game, the maintenance hook never runs for that tick → the exact expiry tick can be missed → the entry sits forever as `waiver_state='dfa'` and shows on `/api/waivers` with a negative `claim_window_games_remaining`. Test G4 ("After 3 games unclaimed: player moves to free agent pool, disappears from /api/waivers") then fails.
3. **Spec's "DFA appears within same sim tick"** (test G4) conflicts with batched maintenance: Architect defers some maintenance to "every 10 games" (service time, prospect dev). If DFA is itself a consequence of a send-down evaluated on a 5-game cadence, the DFA may not appear in the *same* tick it was triggered.

**Recommended resolution:** Evaluate waiver claims on **every** maintenance pass for any entry with `claim_game_window_end <= current_game_number` (range check, not equality), and move the maintenance hook to run even on skipped/failed games (or advance the window only when a real game commits). Define the window unit explicitly.

### AB-04 (Critical) — Call-up / send-down / DFA deadlock when the 40-man is full and no player has options

**Root cause:** The spec's overflow handling is circular and the Architect's G4 ruling does not close the loop.

**Evidence (spec):**
- Send-down §: "Player must have option remaining — if no options left, **must DFA instead**."
- Call-up §: "If no minor leaguer qualifies: check waiver wire, then free agents." Edge case: "Call-up when no AAA players available: ... If truly none available, leave roster short temporarily."
- DFA §: "When a team needs 40-man space mid-season: DFA a player." Architect G4: "Call-up needs 40-man slot and team is already at 40 (rare — call-up logic must pick a DFA candidate)."

**Evidence (code):** Under C2, `is_on_mlb_roster=1` is the 40-man flag. There is no current concept of a 40-man cap enforced anywhere; v0.1.0 just sets `is_on_mlb_roster=1` for the top 25 drafted and FA signings (`draft.ts:352`, `offseason.ts:191`). A team can already exceed 40 today (no cap check). v0.2.0 introduces the cap implicitly.

**Impact / deadlock scenarios:**
1. **Injury call-up, 40-man full, every 40-man player out of options:** Call-up needs a 40-man slot → must DFA someone to make room. DFA candidate selection per G4 ("analytics picks lowest projected WAR") picks a player → that player goes to waivers (still `is_on_mlb_roster=1` per D4 state machine!). DFA does NOT immediately free a 40-man slot — the architect's D4 says DFA sets `is_on_25man=0` but **keeps `is_on_mlb_roster=1`** (player is still on the 40-man while on waivers). So DFA-to-make-40-man-room does not actually create 40-man room until the 3-game waiver window expires. The call-up that triggered the DFA still has nowhere to go for 3 games. The spec's "DFA to create space" is a no-op for the immediate need.
2. **Spring cuts overflow with no options and 40-man already at 40:** Architect G2 rule 4 "force-DFA the next-lowest player" — but force-DFA also keeps them on the 40-man (D4), so the 25-man can still not be reduced to 25 if every surplus player is mid-waiver. The "force to 25" loop (G2: "Continue until 25-man = 25 exactly") can fail to converge.
3. **Roster invariant check (R7) will then fire autoBalance every tick**, churning.

**Recommended resolution:** Define DFA as *immediately* vacating the 40-man slot (set `is_on_mlb_roster=0` on DFA, hold the player in a separate waiver-limbo state not counted against the 40-man), OR explicitly allow the 40-man to temporarily exceed 40 while waiver claims resolve. The current D4 state machine guarantees the deadlock.

## HIGH FINDINGS

### AB-05 (High) — Service time math: a player can never reach free agency within any realistic playthrough, and the offseason does not advance the service-time-year clock

**Root cause:** 172 games/year × 6 years = 1,032 MLB games for free agency, but a player accrues at most 50 games/season.

**Evidence (spec):** "172 games = 1 full year." "6 years service time = free agency eligibility." App-spec: 50-game seasons.

**Math:** 1,032 MLB games ÷ 50 games/season ≈ **20.6 full MLB seasons** for a single player to reach free agency by service time alone. Players retire at age 40 (`offseason.ts:73`), entering ~age 22, so a career is ~18 seasons max — and only if they spend every game on the 25-man. **No player can ever reach 6 years of service time.** Meanwhile v0.1.0 already releases players when `contract_years_remaining <= 0` (`offseason.ts:142`), and worldgen sets contracts of 1–3 years and draft prospects to 6 years (`draft.ts:563`). Free agency in practice is driven entirely by contract expiry, never by the new `service_time_days`/`free_agent_eligible` flag. The Architect's D8 `free_agent_eligible = 1 WHERE service_time_days >= 6*172` flag will **never** be set. Test G5 ("After 172 games service time: player's free agency eligibility flag set") requires reaching just 172 games (~3.4 seasons) for the *flag math at 1 year* — that is testable in isolation — but the spec's actual "6 years = FA" promise is unreachable and creates a dead feature.

**Secondary bug:** Architect C6 says `service_time_years` is derived `FLOOR(service_time_days/172)` and D8 accrues `service_time_days += 10` every 10 games for `is_on_25man` players. But the **service-time manipulation tactic** (G12: delay call-up when `service_time_days in [157,172)`) targets a band a player only crosses once every ~3.4 seasons; combined with the unreachable 6-year threshold, manipulation "to delay FA by 1 year" is meaningless because FA-by-service-time never happens. **Resolve:** either drastically lower the games-per-year constant for this 50-game world (e.g., 1 year = 30 MLB games, FA at ~5 years = 150 games) or explicitly state that contract-expiry remains the real FA driver and service time is cosmetic.

### AB-06 (High) — Test G4 "small market analytics GM > large market old-school" requires a league composition that worldgen does not guarantee

**Root cause:** GM philosophy/risk/focus are drawn independently and uniformly at random, uncorrelated with market size.

**Evidence (code):** `worldgen.ts:230-234`: `philosophy`, `riskTolerance`, `focus`, `ownerPersonality` are each `[randInt(rng,0,2)]` — independent of `city.market_size` (chosen separately at `:210-228`). The Architect's C1 derives `gm_archetype` from these flat fields.

**Evidence (test):** G4: "Small market **analytics** GM teams have higher waiver claim rate than large market **old-school** GM teams (over 20-game sample)." G3: "Analytics GM teams have higher proportion of high-potential young players on 25-man vs old-school (sample 5 teams each)." G6: "Analytics GM teams show higher average prospect rating growth ... vs old-school."

**Impact:** These tests require BOTH cohorts to exist with adequate sample size in the same league. With 6 small markets and ~1/3 analytics probability, the expected number of small-market-analytics teams is ~2 and can be **0** for some seeds. "Sample 5 teams each" (G3) requires ≥5 analytics and ≥5 old-school teams — with 20 teams at ~1/3 each that is ~6.7 expected per archetype, but high variance means a seed can produce 3 analytics / 9 old-school and fail the sample requirement. **No test setup step seeds a controlled composition.** The Architect's R4 mitigation ("fall back to seeded random assignment") makes the distribution *deterministic* but not *balanced*. Either the worldgen must correlate archetype with market (per the spec's "Analytics (small/medium market bias)") OR the tests need a dedicated seeded fixture league with a known composition. Neither is specified.

### AB-07 (High) — `callNewsHeadlinesBatch` (10 events per call) has undefined behavior on partial returns and end-of-queue remainders

**Root cause:** The batch LLM function does not exist yet and the spec/eval do not define event↔headline mapping integrity.

**Evidence (spec/eval):** "News feed headlines... Batched 10 per call." Architect G17: "Batch up to 10... On response, write `headline_text`." Existing LLM functions (`llm.ts`) all return a SINGLE narrative; there is no per-item-keyed batch parser.

**Attack scenarios (none addressed):**
1. **LLM returns 9 headlines for 10 events** (or returns them out of order, or with a hallucinated 11th): which event gets no headline? Without a stable key in the response, the mapping is positional and a dropped/extra line silently shifts every headline onto the wrong event (e.g., a "manager fired" headline lands on an "injury" news item). `sanitizeNarrative` (`llm.ts:145`) operates per-string; it cannot detect misalignment.
2. **Queue has 7 pending when the season ends / phase transitions to playoffs/offseason:** is a sub-10 batch flushed, or do 7 items wait indefinitely with `is_headline_pending=1`? Architect G17's "after 30 game ticks pending, write procedural fallback" only fires during regular-season ticks; in offseason there are no game ticks, so a residual batch could hang until next season's tick loop, or never (if the league sits paused). Test G9 "All non-game events have non-empty LLM headline string" then intermittently fails for the trailing batch.
3. **Budget/circuit-breaker open mid-batch:** `breakerOpen()` (`llm.ts:55`) gates the whole call; the 30-tick fallback is the only recovery, but spring-cuts (G21: ~200 release news items at once) can blow the rolling-window threshold (250 calls/60s) if batched as 20 separate 10-item calls fired in a tight turbo burst, tripping the breaker and leaving the rest pending.

**Impact:** Headline-to-event misalignment is a correctness/integrity bug (wrong headline on wrong player is user-visible nonsense). Trailing-batch hang fails G9. **Resolve:** define a keyed batch response (`{eventId: headline}` map), define flush-on-phase-transition, and define behavior when the LLM returns fewer/extra/misordered items (recommend: per-event procedural fallback for any eventId missing from the response).

### AB-08 (High) — Spring training cuts run on `current_game_number == 0`, but the offseason already transitions phase to `regular_season` and finalize resets the counter — the cut gate can be skipped or re-run

**Root cause:** The spring-cuts gate timing collides with the existing offseason→regular-season transition and the missing-SP skip path.

**Evidence (code):**
- `offseason.ts:337-340` (finalize) sets `phase='regular_season'`, `current_game_number=0` inside a transaction, then `:355-356` runs `validatePostDraftRosters`. There is no spring-cuts hook here today.
- `engine.ts:293-294` `runOneTick` `case 'regular_season': await runGameTick(league)`. Architect D17 proposes inserting `if (springCutsNeeded(league)) { await runSpringCuts(league); return; }` before `runGameTick`.
- Architect G1/G27: gate on `current_game_number == 0 AND (spring_cuts_done_season IS NULL OR < season_number)`.

**Attack scenarios:**
1. **Game 1 also has `current_game_number == 0` at entry** (finalize set it to 0; the first game sets it to 1 only after `simulateGame` commits). So on the FIRST regular-season tick, `current_game_number == 0` is true → spring cuts run, set `spring_cuts_done_season`, `return` (no game simmed). Fine. But if `simulateGame` for game 1 **skips** (missing SP, `game.ts:256`), it sets `current_game_number = 1` and returns without committing a game — yet if the missing-SP condition is itself a *consequence of spring cuts having cut the only SP*, the next tick sees `current_game_number=1` (cuts already done) and proceeds with a broken roster. Spring cuts that over-trim pitching can brick the season exactly like the iter-3 zero-SP stall the team already fought.
2. **Restart mid-season-1-game-0:** if the server restarts after finalize (phase=regular_season, current_game_number=0) but before cuts run, `spring_cuts_done_season` is NULL → cuts run on restart. Good. But if it restarts AFTER cuts but the `spring_cuts_done_season` write was in a *different* transaction than the cuts themselves (Architect does not specify atomicity), cuts could double-run, cutting a second 15 players. Atomicity of (cuts + flag write) is unspecified.

**Impact:** Season can stall (mirroring AB3-01 from v0.1.0) or roster can be double-cut. **Resolve:** make (spring cuts + `spring_cuts_done_season` write) one transaction; have cuts respect position minimums (reuse `validatePostDraftRosters` AFTER cuts); and ensure cuts never reduce a team below the `C/SS/CF/SP>=2/CL>=1` floor that `game.ts` depends on.

### AB-09 (High) — Firing cadence + meddling-owner path can fire an interim manager, and dual GM/owner evaluation can double-fire the same person in one tick

**Root cause:** The "interims cannot be fired" floor and the "owner precedence" tie-break are stated but the evaluation ordering that enforces them is not.

**Evidence (spec):**
- "Interims cannot be fired mid-season (stability floor)." Edge case: "Both owner and GM want to fire manager simultaneously: owner's decision takes precedence. One event, not two."
- Manager firing evaluated every 5 games; GM firing every 10 games; meddling owner fires manager directly (bypasses GM); non-meddling owner fires manager only at 2.5× threshold.
- Architect C12: "firing eval function exits early if `interim_gm=1` or `interim_manager=1`." Architect D13/G13: per-team `last_firing_check_game` (5-game) and `last_gm_firing_check_game` (10-game) cadences.

**Attack scenarios:**
1. **Meddling owner + interim manager:** AB-01's cadence ambiguity means the meddling-owner-fires-manager path and the interim-protection check may live in different functions. If the meddling path (which "bypasses GM entirely") does not also check `interim_manager=1`, a meddling owner CAN fire the interim — violating the stability floor. The Architect's C12 ruling protects "the firing eval function" but the meddling-owner-direct path is a *separate* branch in the spec. Needs an explicit assertion that ALL four firing branches (GM-fires, meddling-owner-fires, non-meddling-owner-fires-at-2.5×, offseason-fires) short-circuit on interim.
2. **Same-tick double fire:** Manager eval (5-game) and GM eval (10-game) can both come due on the same tick (every 10th game both fire). If the GM is fired AND the manager is fired in the same tick, the spec's "owner's decision takes precedence, one event not two" applies only to the manager-firing collision; it does not address whether a just-fired GM's interim immediately re-evaluates the manager in the same maintenance pass. Architect D5 orders firing as a single step 5 but does not forbid cascading.
3. **The 2.5× owner-direct path lowers GM job_security by 2** (`-= 2`); if that pushes the GM below a fire threshold on the SAME tick's GM eval (if GM eval runs after manager eval), the owner could fire the manager AND trigger a GM firing from the embarrassment, producing two front-office events from one record state. Ordering unspecified.

**Impact:** Test G8 ("Interim roles cannot be fired mid-season (no second firing within same season)") can fail on the meddling path; double-events corrupt the news feed and `front_office_events`. **Resolve:** specify (a) all firing branches check interim first, (b) within a tick, evaluate owner-precedence BEFORE GM, and resolve manager firing exactly once, (c) a team that experiences any firing in a tick is exempt from further firing that tick.

### AB-10 (High) — Prospect development runs both in-season (every 10 games) AND in the offseason, double-aging and double-developing players

**Root cause:** The new in-season dev tick coexists with the existing offseason development step, and the bust mechanic keys on age which can change mid-season.

**Evidence (code):** `offseason.ts:93-134` `runDevelopmentStep` increments `age += 1` for EVERY player and applies rating changes every offseason. Architect D7 says the in-season tick runs every 10 games for AA/A/Rookie age 18-25, and the bust mechanic runs "at exactly the offseason transition."

**Attack scenarios:**
1. **Double development:** A 19-year-old A-ball prospect gets +0–1 every 10 in-season games (up to ~+5/season across 50 games) AND +0–3 in the offseason (`offseason.ts:108`). That is far faster growth than v0.1.0 intended; ratings will inflate, breaking the rating distribution that draft/FA value formulas depend on. The spec does not say the offseason dev step is removed or reduced.
2. **Age boundary race:** Offseason increments age. A player at age 25 in AA with potential C: the in-season tick (age 18-25) develops them all season; then the offseason bumps them to 26 and the bust mechanic (D7: "players turning 26... potential=D, overall capped at 65") downgrades them. But if the in-season tick uses a *stale* age (read once at season start) while the offseason already aged them, a player could be developed as "25" in-season AND busted as "26" in the same calendar transition, or skipped entirely if the order flips. The spec's "reaches age 26 still in AA" event is offseason-only per D7, but test G6 ("Player reaching age 26 still in AA with potential C/D: potential set to D, overall capped at 65") doesn't say WHEN within the sim it must be observable.
3. **AAA exclusion vs call-up:** Dev fires only for AA/A/Rookie, not AAA. A prospect promoted to AAA mid-season stops developing entirely, which contradicts "AAA players (any age): stable, preparing for MLB" only if "stable" means "no growth" — ambiguous but probably fine. However the hot-streak visibility boost (D7: `prospect_visible=1`) requires minor-league game stats which AB-02 shows don't exist.

**Impact:** Rating inflation destabilizes the economy (Architect R-class risk, unmitigated); test G6 timing is underspecified. **Resolve:** explicitly state whether offseason dev is retained, reduced, or removed; define a single age source-of-truth for the season; and gate the bust mechanic to run exactly once per player per season.

### AB-11 (High) — `/api/teams/:id/minors` query excludes 40-man minor leaguers, and the response shape change collides with the existing grouped-object contract

**Root cause:** Under C2, a minor leaguer can be `is_on_mlb_roster=1, is_on_25man=0` (on the 40-man, optioned to minors). The current minors query filters `is_on_mlb_roster=0`.

**Evidence (code):** `teams.ts:11-13` `buildMinorsObject` selects `WHERE team_id=? AND is_on_mlb_roster=0 AND is_drafted=1`. After C2, optioned 40-man players (`is_on_mlb_roster=1, is_on_25man=0, minor_level='AAA'`) will be **invisible** in the minors view, and also invisible in the roster view (`teams.ts:73` filters `is_on_mlb_roster=1` → they'd show as MLB roster despite being in AAA).

**Evidence (spec/test):** Test G6 wants minors to return live stats per player; D12 redefines the minors response to include `season_stats` per player. But the existing `/api/teams/:id` ALSO embeds `minors` via the same `buildMinorsObject` (`teams.ts:69,101`) — changing its shape changes two endpoints and may break existing UI consumers (Teams.tsx) and any v0.1.0 regression tests asserting the grouped shape.

**Impact:** Optioned players vanish from both roster and minors views (a player "disappears"); the dual-use helper makes the D12 shape change a wider blast radius than the eval acknowledges. **Resolve:** minors query must select `minor_level IS NOT NULL` (or `is_on_25man=0 AND minor_level IS NOT NULL`) regardless of 40-man flag; the roster query must select `is_on_25man=1`, not `is_on_mlb_roster=1`. The Architect's C2 ruling notes the column semantics but the route audit for these two queries is not in any phase gate.

### AB-12 (High) — Test G7 "at least 3 trades per season" with the procedural matcher has no guaranteed floor, and interim-GM exclusion can starve the market

**Root cause:** The trade matcher is best-effort ("after 3 failed match attempts per buyer, break") with no minimum-trade guarantee, and interim GMs are excluded as both buyers and sellers.

**Evidence (spec/eval):** Architect D9 matching: buyers sorted by games_back, find sellers with positional surplus, "after 3 failed match attempts per buyer, break." Skip teams with `interim_gm=1` (test G7, edge case). League cap 12, per-team cap 2. Test G7 floor: "at least 3 trades complete across the league per season."

**Attack scenarios:**
1. **Tight standings:** If by game 30 most teams are within 5–10 games (NEUTRAL: "small moves only"), there may be few BUYERs (games_back<=5) and few SELLERs (games_back>=10). With a 50-game season and reasonable parity, a cluster of NEUTRAL teams produces **zero** qualifying buyer/seller pairs → 0 trades → G7 fails. Architect C5 says "must produce 3-12 trades" but D9's algorithm has no forcing function (unlike non-tender G23 which explicitly force-creates one). No equivalent "force at least 3 trades" rule exists.
2. **Interim-GM starvation:** If the only seller with a needed position has an interim GM (excluded), the buyer passes (G22). In a league where several bad teams fired their GMs (the very teams most likely to be SELLERs), the seller pool shrinks → trade count drops below 3.
3. **Posture computed at game 30 (per-team or league? — AB-01):** if `trade_posture` is set at `current_game_number=30` (3-4 per-team games in), every team's `games_back` is near 0 → almost everyone is a BUYER, nobody is a SELLER → 0 trades.

**Impact:** G7's hard floor of 3 is not guaranteed by the design. **Resolve:** add an explicit "force minimum 3 trades" fallback (mirroring G23) and define posture timing per AB-01.

### AB-13 (High) — News ticker "on all tabs during active sim" and "scrolling" has no home in the current single-`<main>` layout; the duplicate-testid lesson is being re-violated

**Root cause:** `App.tsx` renders exactly one view at a time inside `<main>`; there is no persistent chrome region for a ticker, and the spec/eval add a per-tab ticker requirement plus an unfixed duplicate testid.

**Evidence (code):** `App.tsx:121-143` nav renders `data-testid="nav-${tab.id}"` buttons. `App.tsx:146-173` `<main>` conditionally renders ONE view. `App.tsx:111` and `:155` BOTH render `data-testid="new-dynasty-button"` — when `noLeague=true`, the header button (always rendered) AND the welcome-screen button (rendered when noLeague) are BOTH in the DOM → **duplicate testid present today**, violating Universal Lesson #4. Architect D13 acknowledges this and mandates a rename, but it is an existing live defect that v0.2.0 inherits.

**Evidence (spec):** "Scrolling bar displayed at top of all tabs during active sim." data-testid `news-ticker`, `news-ticker-item-{eventId}`. Architect C8/D13: one ticker instance in App.tsx header, CSS scroll, fed from `/api/news?limit=5`.

**Attack scenarios:**
1. The ticker must live OUTSIDE `<main>` (in header/nav) to appear "on all tabs," but the spec also defines a News **tab** with `news-feed`. If a developer renders the ticker inside each view (the natural place given the current structure), it duplicates `news-ticker` across tabs → Lesson #4 violation, exactly the failure mode the Architect's D13 warns about.
2. Architect D13 mandates `nav-news` AND `news-tab` synonyms on the same button — two testids on one element is fine, but the TabName union (`App.tsx:11`) and `tabs` array (`:77-84`) have no `news` entry; adding it touches the auto-navigate effect (`:50-54`) and the conditional render block.
3. The "active sim" gating ("during active sim") is not defined — does the ticker hide when paused? when in offseason/draft? The spec says "all tabs during active sim" but the News tab itself should presumably work when paused. Undefined visibility rule.

**Impact:** High risk of a re-introduced duplicate-testid failure (the single most-repeated lesson) and an undefined ticker visibility contract. **Resolve:** mandate one ticker in App chrome above `<main>`, define its visibility states, and fix the existing `new-dynasty-button` duplicate now.

## MEDIUM FINDINGS

### AB-14 (Medium) — Waiver claim order "reverse current standings" is undefined during spring training (all teams 0-0)

**Root cause:** Spring cuts release players (no waivers, per spec) but DFA can occur immediately at season start when standings are all 0-0.

**Evidence (spec):** "Claim order: reverse current standings order." Architect G5 tie-break: `runs_scored - runs_allowed ASC, team_id ASC`. At season start, finalize reset all teams to `wins=0, losses=0, runs_scored=0, runs_allowed=0` (`offseason.ts:343`). Spring cuts run at `current_game_number=0` (G1).

**Impact:** If a spring-cuts overflow forces a DFA (AB-04 scenario), the waiver claim order is fully degenerate — every team ties on W/L and run-diff, so order collapses to `team_id ASC`, deterministically favoring low-id teams for the entire opening claim window. Not random, not "reverse standings" in any meaningful sense. Spec says spring cuts go straight to FA (no waivers), but the Architect's G2 rule 3/4 explicitly DFAs spring overflow → waivers. Contradiction between spec ("no waivers for spring cuts") and Architect G2 ("DFA → waiver wire" for spring overflow). **Resolve the contradiction** and define claim order when standings are uniform.

### AB-15 (Medium) — Annual draft pool generates 200 prospects for 600 picks; pool exhaustion floods the league with "Replacement Player" and "Prospect DraftN" names

**Root cause:** Pre-existing v0.1.0 behavior carried into v0.2.0's now-procedural draft.

**Evidence (code):** `draft.ts:550-576` `generateDraftClass` inserts exactly 200 prospects named `Prospect Draft{i}`. The annual draft is 30 rounds × 20 teams = 600 picks (`draft.ts:471-472`). After ~200 picks the pool exhausts and `runDraftPickSync`/`handleExhaustedPool` generate "Replacement Player" rows (`draft.ts:166-168, 304-306`). The Architect's eval-5 backlog flags the placeholder names for "fix during Phase 8" but the 200-vs-600 shortfall (the *cause* of mass replacement-player generation) is not addressed.

**Impact:** Each annual draft creates ~400 "Replacement Player" / generic "Prospect DraftN" entries that then flow into the minor-league system the v0.2.0 spec is trying to make "live." News items, call-ups, and prospect dev will operate on hundreds of identically-named placeholder players. Test G6 minors and the News feed will look broken/spammy. **Resolve:** generate ≥600 named prospects per annual draft, or reduce annual draft rounds.

### AB-16 (Medium) — `/api/news?type=invalid` 400 contract vs the badge taxonomy; "roster" filter missing from spec testids

**Root cause:** The valid `type` set is defined in D12 but the spec's testid list omits `roster`, and the badge values contain a space ("FRONT OFFICE").

**Evidence (spec/eval):** Test G11: `GET /api/news?type=invalid` → 400 `{"error": "Invalid event type filter"}` (verbatim). Architect D12 valid types: `all, transactions, frontoffice, injuries, milestones, roster, game`. Spec testid list (lines 269-274) has filters for all/transactions/frontoffice/injuries/milestones — **no `roster` filter** and no `game` filter, yet ROSTER (call-up/send-down/DFA) and GAME are event badges. Architect D13 adds `news-filter-roster` defensively.

**Impact:** A user cannot filter to ROSTER events (call-ups/DFA/send-downs) — a core v0.2.0 feature — because no filter button is specced. The badge "FRONT OFFICE" (with space) vs query param "frontoffice" (no space) requires a normalization layer; if the Developer naively maps `?type` to `badge` equality, `frontoffice` will match zero rows (badge is "FRONT OFFICE"). **Resolve:** define the full filter set including roster, and specify the param→badge normalization map verbatim.

### AB-17 (Medium) — Owner-personality CHECK-constraint table swap (C14) risks data loss and breaks the one-active-league unique index

**Root cause:** SQLite cannot alter a CHECK constraint in place; the table-swap recreates `teams`, which has FKs from many tables and is referenced by a partial unique index.

**Evidence (code):** `001_init.sql:51` `owner_personality ... CHECK (... IN ('meddling','hands-off','moderate'))`. `:22` partial unique index `one_active_league ON leagues(archived)`. Teams are referenced by FKs in players, game_log, season_stats, transactions, front_office_events, draft_picks, season_narratives. Architect C14/D1 mandate a table-swap to add `win-now`/`patient`.

**Impact:** A teams-table swap (create new, INSERT SELECT, drop old, rename) must (a) recreate ALL indexes on teams, (b) preserve AUTOINCREMENT id continuity so existing FKs remain valid, (c) run with `PRAGMA foreign_keys` handled correctly (dropping a referenced table with FKs enabled fails or cascades). The migration runner wraps in a transaction (`db.ts:65-68`) but does not toggle `foreign_keys`. If FKs are ON, `DROP TABLE teams` fails; if OFF and not re-enabled, integrity checks are skipped. Test G0 ("v0.1.0 league data survives upgrade") is at direct risk. **Resolve:** the migration must explicitly manage `foreign_keys` pragma, recreate all teams indexes, and a `migration007.test.ts` must verify row counts and FK integrity on a real v0.1.0 snapshot before any feature work (Phase 1 gate — correctly placed, but the FK/pragma detail is unaddressed).

### AB-18 (Medium) — No per-tick transaction exists today; the Architect's atomicity guarantees (R3/R6/D17) assume infrastructure that must be built first

**Root cause:** `runGameTick` does not wrap game + maintenance in one transaction; `simulateGame` owns its own transaction and returns.

**Evidence (code):** `engine.ts:368-405` `runGameTick` calls `simulateGame` (opens/commits its own tx at `game.ts:409-458`) then optionally checks season completion. There is no outer transaction. Architect D5/D17 say `runRosterMaintenance` runs "inside the same logical transaction" as the game write — but to do that, `simulateGame`'s internal transaction must be refactored to accept an outer tx or the maintenance must open its own. Better-sqlite3 transactions do not nest into a single commit by default.

**Impact:** If maintenance (waiver/call-up/firing/news writes) runs in a SEPARATE transaction after `simulateGame` commits, a crash between them leaves the game committed but maintenance lost — and on restart the maintenance for that game never re-runs (current_game_number already advanced). This breaks the persistence tests (G10) for waiver windows and interim status if the crash lands in that gap. The Architect treats per-tick atomicity as a given; it is not. **Resolve:** specify whether maintenance is a sibling transaction (and accept the crash-gap) or refactor `simulateGame` to run inside an outer tick transaction.

### AB-19 (Medium) — Turbo "yield every 5 games" interacts badly with per-team 5-game / 10-game maintenance cadences

**Root cause:** The turbo yield boundary (`turboGameCount % 5`) and the maintenance cadence (`% 5`, `% 10`) are both keyed to game counts but on different counters, and turbo runs maintenance for many games with no HTTP interleaving.

**Evidence (code):** `engine.ts:268-272`: turbo yields to the event loop every 5 league games. Architect D5/D7: service time & prospect dev every 10 games (`current_game_number % 10`), call-up/firing every 5 (`current_game_number - last_check >= 5`).

**Impact:** During a turbo full-season burst (~500 games), prospect-dev runs ~50 times (each iterating ~400 minor leaguers), service-time batch-updates run ~50 times across all 25-man players, plus per-team call-up/send-down/firing evals. The Architect's R2 benchmark ("1 season in <12s") may be optimistic given AB-02's potential minor-league sim and the league-wide invariant check (R7) on EVERY tick. If turbo blocks for seconds between yields, the cooperative-pause and `/api/state` poll both stall (the iter-4 AB4-02 issue, accepted for v0.1.0, is now materially worse). Not a correctness bug per se, but a regression risk the eval underweights. **Resolve:** benchmark with the full maintenance load before committing to the <12s gate.

### AB-20 (Medium) — Service-time accrual "batch every 10 games" makes test G5 ("increments by 1 per game") only true on average, and a player called up and sent down within a 10-game window may accrue 0 or 10 incorrectly

**Root cause:** R5 batches accrual every 10 games for all `is_on_25man=1` players, but roster membership changes within the window.

**Evidence (eval):** Architect D8: `UPDATE players SET service_time_days = service_time_days + 10 WHERE team_id=? AND is_on_25man=1` every 10 games. Test G5: "Service time increments by 1 game for each game spent at MLB level."

**Impact:** A player called up at game 3 of a 10-game window and present at the batch tick gets +10 (overcount by 3). A player sent down at game 7 but off the 25-man at the batch tick gets +0 (undercount by 7). Service-time manipulation (G12) targets a `[157,172)` band that this coarse accrual can overshoot by up to 9 days in a single batch, jumping a player past the manipulation window before the delay logic sees it. Test G5 read literally ("increments by 1 for each game") fails for any player whose roster status changed mid-window. **Resolve:** either accrue per-game for is_on_25man players (R5's rejected option, but correctness-preserving) or define G5 as "monotonically increases while on the 25-man" and accept batch granularity explicitly.

## LOW FINDINGS

### AB-21 (Low) — WHIP leaders still use batter hits, not pitcher hits-allowed (carryover, now in scope)

`players.ts:83`: `(ss.walks_pitching + ss.hits) / ss.innings_pitched` — `ss.hits` is the batter hit accumulator; `season_stats` has no `hits_allowed` column. A pitcher with 0 batting hits but 80 IP gets WHIP = walks/IP only (understated); a two-way or mis-attributed row is nonsense. Architect schedules the fix in Phase 12 (new `pitcher_hits_allowed` accumulator). Flagged so it is not lost: the box-score generator (`game.ts:575`) computes `hitsAllowed` per pitcher line but `upsertPitcherStats` (`game.ts:870-893`) never persists it. The accumulator column AND the upsert write both need adding.

### AB-22 (Low) — `sanitizeNarrative` 280-char cap vs news headline "one-liner" — acceptable, but batch headlines are not length-validated per item before mapping

`llm.ts:162` caps at 280 chars. News headlines are "one-liners"; 280 is generous but fine. The risk is only if the batch parser maps a single 280-char blob across multiple events (see AB-07). No standalone defect; noted for completeness.

### AB-23 (Low) — `/api/news` cursor pagination (`?before=`) and `?limit` max 200 are Architect inventions (D24) not in the spec; testers only know the spec

Test spec (G0-G11) and the v0.2.0 spec only reference `?team`, `?type`, `?limit=50`. The Architect's `?before` cursor (D24) and `limit` max 200 are unspecced. UI Tester B tests only against the test spec and will not exercise `?before`; harmless, but the spec and eval diverge on the public contract. Document `?before` in the spec or drop it.

### AB-24 (Low) — Game-result news items synthesized at read-time (G19) via UNION with `game_log` will not have stable `eventId` for `news-item-{eventId}` / ticker keys

Architect G19: game results are NOT stored in `news_items`, synthesized from `game_log` at read time. But the ticker/feed testids are `news-ticker-item-{eventId}` and `news-item-{eventId}`. A synthesized game row's id namespace (`game_log.id`) overlaps with `news_items.id` (both AUTOINCREMENT from 1) → two different news items can share `eventId=5` (one a real news_item, one a synthesized game), producing duplicate `data-testid="news-item-5"` → Universal Lesson #4 violation in the feed. **Resolve:** namespace synthesized ids (e.g., `g{game_log.id}` vs `n{news_items.id}`) so testids are unique.

## Cross-Cutting Observations

1. **The unit-of-time ambiguity (AB-01) is the spine of the NOT READY verdict.** It must be resolved with a single explicit table mapping every spec phrase ("game 30," "every 5 games," "3 games of sim time," "172 games") to either `current_game_number` (league) or per-team `games_played`. Recommendation: per-team for all thresholds/cadences, since that matches MLB intuition and the existing `shouldFireTradeDeadline`.

2. **AB-02 (no minor-league sim) is a scope question the eval never confronts.** The spec's call-up/send-down/dev triggers and test G6 all assume minor-league statistical output that does not exist. This is either a large unscoped feature or a required descope.

3. **Test-determinism gap (AB-06, AB-12):** Several test groups (G3, G4, G6, G7) assert league-composition or market-behavior outcomes with no controlled fixture and no forcing function. These will be flaky across seeds. The build needs dedicated seeded fixture leagues with known archetype/market compositions, or forcing rules.

4. **Inherited live defects v0.2.0 must not ship with:** duplicate `new-dynasty-button` testid (AB-13), WHIP formula (AB-21). Both are acknowledged by the Architect but are real and currently shipping in v0.1.0.

## VERDICT: NOT READY

The spec cannot be built correctly as written. Four Critical defects (AB-01 time-unit ambiguity, AB-02 missing minor-league stats, AB-03 waiver-window race, AB-04 DFA/40-man deadlock) are structural: each breaks a core v0.2.0 subsystem and at least one mandatory test group. The Architect's eval-0 is thorough on schema and ordering but silently picks `current_game_number` as the time unit for cadences/windows where per-team `games_played` is the only sensible choice, and it never confronts that minor leaguers produce no statistics. Resolve all four Criticals (with an explicit time-unit mapping table, a minors-stats decision, a range-based waiver-claim model that survives skipped/turbo ticks, and a non-circular DFA-frees-40-man rule) and the High findings (especially AB-05 service-time math and AB-06/AB-12 test-determinism forcing functions) before issuing developer-instructions-1.
