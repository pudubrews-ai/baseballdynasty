## Executive Verdict

**NOT READY.** The migration, owner-death fix, spring cuts, waiver state machine, service-time model, in-season firings, trade deadline, and archetype table are largely well-built and resolve most pre-build Criticals. But the **entire News Feed subsystem is non-functional**: no event producer anywhere in the simulation writes a `news_items` row, and the LLM headline filler (`callNewsHeadlinesBatch`) has zero callers. The News tab, the live ticker, and every "generates a news item" promise in the spec produce nothing during gameplay. This is a Critical that fails Test Group 9 (News Feed UI) wholesale, plus four scattered spec line items, and it slipped through because `newsApi.test.ts` only calls `insertNewsItem` from inside the test, never asserting the sim produces news.

Two further roster-integrity defects compound it: optioned 40-man players are invisible in the minors API and bullpen selection still draws from the 40-man rather than the 25-man, and the spring-cuts "repair" path is dead code so "exactly 25" is not guaranteed. The duplicate `new-dynasty-button` testid the Architect explicitly mandated fixing (D13) is still present.

**Totals: 2 Critical, 5 High, 6 Medium, 4 Low. Plus a pre-build resolution matrix.**

## CRITICAL FINDINGS

### PB-01 (Critical) — The News Feed subsystem produces ZERO data during simulation; no event producer writes `news_items`, and the LLM headline filler is never called

**Root cause:** `server/sim/news.ts` defines a complete set of writer functions (`insertNewsItem`, `insertGameNewsItem`, `insertRosterNewsItem`, `insertTransactionNewsItem`, `insertFrontOfficeNewsItem`, `insertMilestoneNewsItem`) and `services/llm.ts` defines `callNewsHeadlinesBatch` — but **nothing in the simulation ever calls any of them.**

**Evidence:**
- Grep across `server/` (excluding `sim/news.ts` and `tests/`) for every news writer and the batch LLM filler returns **only the definitions** — zero production call sites:
  ```
  server/services/llm.ts:392  export async function callNewsHeadlinesBatch(   ← defined, 0 callers
  server/services/llm.ts:447  export async function callTransactionFlavorsBatch( ← defined, 0 callers
  (no insertNewsItem / insert*NewsItem / insertGameNewsItem callers in sim or routes)
  ```
- The event producers write to `transactions` / `front_office_events` only, never to `news_items`:
  - DFA: `waivers.ts:37-41` (transactions only).
  - Waiver claim / expiry: `waivers.ts:247-251, 274-278`.
  - Call-up: `callup.ts:96-100`. Send-down/DFA: `sendDown.ts:101-105, 110`.
  - Spring release/send-down: `springCuts.ts:151-162, 292-303`.
  - Firings (manager/GM/owner-breaks-glass): `firings.ts:102-106, 248-256` (transactions) + `front_office_events` (`firings.ts:86-90`).
  - Trades: `tradeDeadline.ts:125-138`. Non-tender: `offseason.ts:184-188`.
  - Owner sold/died: `offseason.ts:379-389, 398-408` (front_office_events only).
- Game results: `game.ts` has no `insertGameNewsItem` call anywhere (grep for `NewsItem`/`news_items` in `game.ts` is empty).
- The headline filler the Architect mandated (G17: "On each game tick … check `is_headline_pending=1` … batch up to 10 … `callNewsHeadlinesBatch`") is **not wired into `rosterMaintenance.ts`, `engine.ts`, or anywhere** (grep for `is_headline_pending` / `headline` in `sim/` and `routes/` outside `news.ts` is empty).

**Impact (fails the spec and Test Group 9 wholesale):**
- `GET /api/news` (`routes/news.ts`) always returns `[]` during real play — it reads `news_items`, which is never populated. Test spec line 175 ("empty array before any events") passes; lines 137–153 (News Feed UI: feed renders items, items show badge/headline/game-number, filters, click-to-expand) all **fail** because there are no items.
- The live ticker (`App.tsx:31` `if (items.length === 0) return null;`) renders **nothing**, ever. Test spec line 141 ("`news-ticker` visible on all tabs during active sim") and 142 ("shows last 5 events, updates as sim runs") fail.
- Test spec line 28/153 ("News feed headlines are non-empty LLM strings for all non-game events") is **unsatisfiable** — no non-game news items exist, and no code fills `headline_text`.
- Test spec line 69 ("Player with no options … send-down → DFA'd instead, **generates news item**") fails — `sendDown.ts:110` DFAs but writes no news item.
- Test spec line 152 ("Game result events in ticker show score only") fails — game results never enter the ticker.

**Why the green test suite missed it:** `newsApi.test.ts` calls `insertNewsItem` **directly from the test body** (lines 31-43, 200-206) and then asserts `getNewsFeed` returns those rows. It never runs `simulateGame`/`runRosterMaintenance` and asserts `news_items` grew. `e2eBenchmark.test.ts:3` *comments* "news created" but contains **no assertion on `news_items`** after the season. So the feature's only integration is the test pretending to be the sim.

**Resolution required:** Wire `insert*NewsItem` into every event producer (the same transaction as the source-of-truth row, per C9/D3), wire `insertGameNewsItem` into the game-result path (or the maintenance hook), and wire the `callNewsHeadlinesBatch` filler into the tick loop with the procedural fallback. Add an integration test that sims ≥1 season and asserts `news_items` is populated for each badge category and that pending headlines get filled (or fall back).

## HIGH FINDINGS

### PB-03 (High) — AB-11 only half-fixed: optioned 40-man players vanish from the minors API, MLB roster API shows them, and the bullpen still pitches AAA-optioned relievers

**Root cause:** Under C2, an optioned player is `is_on_mlb_roster=1, is_on_25man=0, minor_level='AAA'`. The lineup/SP selection was migrated to `is_on_25man` but the supporting queries were not.

**Evidence:**
- Minors API excludes optioned players: `teams.ts:18` and `:34` filter `is_on_mlb_roster = 0`. A player optioned to AAA (`is_on_mlb_roster=1`) is **not** returned by `/api/teams/:id/minors` nor the embedded `minors` object — a player who was sent down literally disappears from the minors view.
- Roster API includes optioned players: `teams.ts:123` (`/api/teams/:id`) and `:163` (`/api/teams/:id/roster`) filter `is_on_mlb_roster = 1`, so an AAA-optioned player shows on the **MLB roster** list.
- Bullpen still drawn from the 40-man: `game.ts:130,133` (relief rating sample) and `game.ts:264,267` (actual relievers used in the box score) filter `is_on_mlb_roster = 1`, **not** `is_on_25man = 1`. After spring cuts/send-downs move a reliever to AAA (`is_on_25man=0`, `is_on_mlb_roster=1`), that reliever can still appear out of the bullpen in MLB games.

**Impact:** Test G6 ("minors returns live stats for all minor leaguers") misses every optioned player. Roster integrity is wrong (a sent-down player both shows on the MLB roster and pitches relief). The lineup/SP fix at `game.ts:58,99` is internally inconsistent with the bullpen at `game.ts:264`.

**Resolution required:** Minors query → `WHERE minor_level IS NOT NULL` (regardless of 40-man flag). MLB roster query → `is_on_25man = 1`. Bullpen queries → `is_on_25man = 1`.

### PB-04 (High) — Spring cuts do not guarantee "exactly 25"; the repair-up path is dead code

**Root cause:** Spec ("trims … to 25-man active", Architect Phase 4 gate "each team has exactly 25 on `is_on_25man=1`") requires exactly 25. The implementation only ever cuts **down** and stops early on position minimums; it never trims teams that are over 25 once a minimum blocks, and it never promotes teams that are **under** 25.

**Evidence:**
- `springCuts.ts:119-171` iterates candidates and `continue`s past any cut that would violate a position minimum (`:126`). If the only remaining over-25 players are all min-position holders, the loop exits with the roster **above** 25.
- `repairPositionMinimums` (`springCuts.ts:195-270`) — the function that would promote players to backfill — is **never called** (grep: defined at `:195`, zero call sites). So a team that starts a season with `<25` on the 25-man (e.g., after retirement/FA depletion entering season N+1) is never brought up to 25.
- The team's own test relaxes the gate: `springCuts.test.ts:89` asserts `toBeLessThanOrEqual(30)`, not `=== 25`. The Architect's Phase 4 gate said **exactly 25**.

**Impact:** Roster sizes drift (24–30 range), contradicting the spec's "exactly 25 players on `is_on_25man`" and the Architect Phase 4 gate. Combined with PB-03's bullpen-from-40-man, a team under 25 could field a short lineup. The `checkRosterInvariant` in `rosterMaintenance.ts:30-44` only logs when `>25` and never repairs `<25`.

**Resolution required:** Call `repairPositionMinimums` after cuts; if still <25, promote best 40-man reserves up to 25; if >25 due to minimums, cut a non-min player. Restore the "exactly 25" assertion.

### PB-05 (High) — Owner-died / owner-sold news + the heir personality fix never reach the News feed, and there is still no isolated test for the personality randomization claim

**Root cause:** The owner-death personality bug fix itself is correct, but its news-item surfacing is absent (PB-01), and `ownerDeathPersonality.test.ts` (referenced by Architect Phase 3) does not actually validate fresh personality the way the spec demands.

**Evidence:**
- The fix is real and correct: `offseason.ts:407-408` draws the heir personality from all five `OWNER_PERSONALITIES` (`offseason.ts:14`) and writes both `owner_name` and `owner_personality`. Same for owner-sold at `:377,388`. This resolves the v0.1.0 bug.
- But: `owner_died` / `owner_sold_team` write only `front_office_events` (`offseason.ts:398-405, 379-386`). No `insertFrontOfficeNewsItem` call → these never appear in the News feed (spec table lines 242-243 require both). This is PB-01 manifesting on a specific event.
- The randomization is RNG-driven (`Math.floor(rng() * OWNER_PERSONALITIES.length)`); with 5 options there is a 20% chance the heir's personality equals the parent's. A test asserting "personality differs from prior" would be flaky. The spec says "completely fresh" / "not their parent" — the implementation does not *guarantee* a different value, only a freshly drawn one.

**Impact:** The bug-fix portion is delivered (good). But the news-surfacing fails the spec, and the "heir is not their parent" guarantee is probabilistic, not enforced — a dedicated test asserting inequality would be flaky.

**Resolution required:** Wire owner events into the News feed (part of PB-01). Decide whether "fresh" means "may coincidentally match" (then test only that it was re-rolled) or "must differ" (then resample on collision); document and test accordingly.

### PB-06 (High) — Trade matcher contains broken/dead comparison logic and an unused archetype import; analytics "demand 2 prospects" works only by accident

**Root cause:** `tradeDeadline.ts:79` compares the result of `getArchetype(...)` (which returns an `ARCHETYPES` object) against a *different* stub object `ARCHETYPES_DATA['analytics']` (`:98-100`), which can never be equal; the intended behavior is salvaged only by the `|| seller.gm_archetype === 'analytics'` clause.

**Evidence:**
```ts
// tradeDeadline.ts:75-79
const buyerArchetype = getArchetype(buyer.gm_archetype ?? 'balanced'); // assigned, never used
const sellerArchetype = getArchetype(seller.gm_archetype ?? 'balanced');
const prospectsRequired =
  (sellerArchetype === ARCHETYPES_DATA['analytics'] || seller.gm_archetype === 'analytics') ? 2 : 1;
// ARCHETYPES_DATA is a 1-field stub at :98-100, unrelated to ARCHETYPES
```
- `sellerArchetype` is an `ARCHETYPES.analytics` reference; it is **never** `===` the `ARCHETYPES_DATA.analytics` stub. The first operand is always false.
- `buyerArchetype` (`:75`) is computed and never read — dead.
- The `archetypes.ts` `ARCHETYPES` table the Architect mandated (D11: "all archetype logic reads from this table, never hardcoded") is bypassed here in favor of a string compare and a stub object.

**Impact:** Not a crash, but the matcher does not "read from the ARCHETYPES table" per D11; it's confused, half-dead code that happens to produce the right `prospectsRequired` via the fallback. Old-school's "demands proven players back" (D9 step 4) is not implemented at all — only the prospect count varies. Maintenance hazard and a spec-intent miss.

**Resolution required:** Replace the stub comparison with `getArchetype(seller.gm_archetype).draft_potential_weight`-style reads, remove `ARCHETYPES_DATA`, use `buyerArchetype`, and implement old-school's return-package preference.

### PB-07 (High) — "Recent" rolling windows never slide; OPS-over-last-15-AB and ERA-over-last-3-starts actually evaluate full-season cumulative stats

**Root cause:** The Architect's G9 plan was a sliding ~15-AB / last-N-starts window (deque). The implementation only **adds** to `recent_*` and never subtracts/decays, so `recent_ab == at_bats` and `recent_starts == games_started` for the whole season.

**Evidence:**
- `game.ts:872-875` (batters) and `:922-925` (pitchers) do `recent_ab = recent_ab + ?`, `recent_starts = recent_starts + 1`, etc. There is **no** decay, eviction, or window-cap anywhere (grep for any `recent_* = ... - ` or window-trim is empty).
- Call-up trigger reads `recent_ab/recent_hits` as a season figure: `callup.ts:174-183`, `:210-218`. Send-down likewise: `sendDown.ts:19-31` requires `recent_ab >= 20` and computes OPS over the entire season's `recent_ab`. `recent_starts >= 3/4` (`callup.ts:214`, `sendDown.ts:55`) is satisfied permanently after the 3rd/4th start.

**Impact:** The spec's "OPS < .580 over **last 15 AB**", "ERA > 7.00 over **last 3 starts**", "OPS < .560 over **20+ AB**", "ERA > 6.50 over **4+ starts**" all degrade to season-to-date averages. A pitcher who was bad early and good lately (or vice-versa) is judged on the full season; a recent slump never triggers a call-up because the cumulative number is diluted. The triggers still *fire* (so the feature isn't dead), but they do not implement the recency the spec requires. Determinism/tests don't catch it because no test asserts a recency window.

**Resolution required:** Implement a true sliding window (e.g., per-game deque in `recent_window_json` per G9, or reset `recent_*` every N games), so "last 15 AB" / "last 3 starts" reflect recency.

## MEDIUM FINDINGS

### PB-08 (Medium) — The ≥3-trades floor and the forced-trade path are not exercised end-to-end; the engine-only wiring is untested

**Root cause:** `forceMinimumTrades` is invoked only from `engine.ts:399-408` inside `runGameTick`, but the e2e benchmark sims games by calling `simulateGame`+`runRosterMaintenance` directly (`e2eBenchmark.test.ts:51-67`) and **never goes through `runGameTick`**, so the deadline marker + force path is never executed in any integration test. `tradeDeadline.test.ts` tests `evaluateTradeDeadline`/`setTradePosture` units but does not assert the league-wide ≥3 floor after a full season.

**Evidence:** `engine.ts:399` `if (shouldFireTradeDeadline(...))` → `fireTradeDeadline` + `forceMinimumTrades`. The e2e loop bypasses `runGameTick` entirely. No test asserts `COUNT(transaction_type='trade') >= 3` after a season.

**Impact:** Test G7's hard floor ("at least 3 trades complete across the league per season") is unverified. The forcing logic itself looks plausible (`tradeDeadline.ts:209-275`) but its only trigger lives in an untested code path, and `forceMinimumTrades` can still under-deliver if buyers lack AAA/AA prospects (`:250-257` `continue`s) — it logs "Only forced X/needed" (`:272`) rather than guaranteeing 3.

**Resolution required:** Add an integration test that runs a full season through `runGameTick` (or directly calls the deadline path) and asserts ≥3 trades. Make `forceMinimumTrades` fall back to a prospect-less package when buyers have no AAA/AA so the floor is truly guaranteed.

### PB-09 (Medium) — `gm_archetype` derivation is not market-correlated; G3/G4 "5 analytics + 5 old-school" sample requirements remain seed-dependent

**Root cause:** Worldgen still draws `gm_philosophy/risk/focus` independently of market size; the archetype is then derived from those flat fields. There is no guarantee of ≥5 analytics and ≥5 old-school teams in a league, nor that analytics skews small-market.

**Evidence:** Migration derivation `007:69-73` keys off `gm_risk_tolerance`/`gm_philosophy`/`gm_focus` with only a partial market clause (`philosophy='rebuild' AND market IN ('small','medium')`). Worldgen assigns those flat fields by uniform `randInt` (unchanged from v0.1.0). The Architect's R4 fallback (seeded random) makes it deterministic, not balanced. No fixture league with a controlled composition exists.

**Impact:** Test G3 ("sample 5 teams each") and G4 ("small-market analytics vs large-market old-school over 20-game sample") can fail on unlucky seeds — there may be <5 of an archetype, or zero small-market analytics teams. Flaky across seeds.

**Resolution required:** Either correlate archetype with market in worldgen (per the spec's "Analytics (small/medium market bias)") or provide a dedicated seeded fixture league for G3/G4 with a known composition.

### PB-10 (Medium) — Same-tick manager + GM firing can produce two front-office events for one record state

**Root cause:** `evaluateFirings` runs the manager check then the GM check within the same transaction; when both cadences are due (every 10th game both are), a single bad record can fire the manager AND the GM in one tick.

**Evidence:** `firings.ts:288-302` — manager check (`:289`) calls `evaluateManagerFiring` (which may promote an interim manager), then GM check (`:295`) calls `evaluateOwnerFiresGm` (which may install an interim GM) — both on the same `under500` figure, no "one firing per tick" guard. The spec's "one event, not two" edge case addresses only the owner-vs-GM **manager** collision, but two *distinct* roles firing simultaneously off one collapse is jarring and untested.

**Impact:** A team can lose both its manager and GM in the same game tick, generating two FRONT OFFICE events. Not a crash; cosmetically/narratively off and not covered by tests. Lower severity because both firings are independently "correct" by threshold.

**Resolution required:** Optionally gate to one front-office firing per team per tick (manager precedence, GM next tick), or explicitly document simultaneous firings as intended.

### PB-11 (Medium) — Minors live-stats response shape deviates from D12 (`stats`/`avg` vs `season_stats`/`battingAvg`)

**Root cause:** D12 specified the minors response gains `season_stats: { games_played, at_bats, hits, home_runs, battingAvg }`. The implementation nests a `stats` object with different keys.

**Evidence:** `teams.ts:44-57` sets `stats: { games, ab, avg, hr, rbi }` for batters and `{ games, ip, era, k }` for pitchers, attached as `stats` (`:68`), not `season_stats`. Keys are `avg`/`ab`/`games`, not `battingAvg`/`at_bats`/`games_played`.

**Impact:** Any test asserting the D12-documented `season_stats.battingAvg` shape fails. Test spec G6/D12 reference live stats `games, ab, avg or ip, era` — those substrings are present, so a loose UI test may pass, but the documented contract diverges. ## LOW FINDINGS

### PB-13 (Low) — `server/data/{cities,names,nicknames}.ts` are git-ignored; a clean checkout cannot run tests or boot

`.gitignore` line 3 (`data/`) matches `server/data/`, so the world-generation data files are untracked. This is a pre-existing hazard now amplified because v0.2.0's CI/test story depends on these. Recommend un-ignoring `server/data/*.ts` or narrowing the ignore to `server/data/*.json`.

### PB-14 (Low) — `manipulation_delay` and prospect-dev RNG use `Math.random()`, breaking determinism

`callup.ts:144,199` use `Math.floor(Math.random() * 11) + 10` for the manipulation delay, and `prospectDev.ts:57` uses the seeded `batchRng` correctly but the `Math.random()` calls in callup are non-deterministic. The Architect's D16 mandates `seedFor(...)` for all new procedural logic. Service-time-manipulation timing therefore varies run-to-run for the same seed.

### PB-15 (Low) — `repairPositionMinimums` dead code + `buyerArchetype` unused: lint passes but the build carries unreachable logic

`springCuts.ts:195` (`repairPositionMinimums`) and `tradeDeadline.ts:75` (`buyerArchetype`) are defined-but-unused. They pass lint (functions/locals aren't flagged here) but signal incomplete wiring (see PB-04, PB-06). Noted so they aren't mistaken for working features.

### PB-16 (Low) — Ticker visibility gate references a phase value that never exists

`App.tsx:199` gates the ticker on `state.phase !== 'no_league'`, but `mapPhase` (`engine.ts:77-88`) only ever returns `draft|regular_season|playoffs|offseason` and throws otherwise — `'no_league'` is never a phase. The guard is effectively `state && true`. Harmless today (ticker is empty anyway per PB-01) but the intended "hide when no league" / "during active sim" visibility contract (spec line 141) is not actually implemented.

## VERDICT: NOT READY

The release cannot ship: the News Feed — one of the four headline features of v0.2.0 and the entirety of Test Group 9 — writes no data during simulation (PB-01), and the test harness that "validated" it only inserts rows by hand. A UI tester will see an empty ticker and an empty News tab, and the "generates a news item" promises across the spec (firings, call-ups, DFA, trades, owner events, injuries, milestones) are all unmet at runtime. Compounding this, the empty-state still ships a duplicate `new-dynasty-button` testid (PB-02) that the Architect explicitly ordered fixed, blocking the first step of most UI test groups.

The simulation engine itself is in good shape — most pre-build Criticals (time-unit, waiver race, DFA deadlock, migration safety, service-time math) are genuinely resolved — but the roster-visibility regressions (PB-03), the un-guaranteed 25-man (PB-04), the season-cumulative "recent" triggers (PB-07), and the broken trade-matcher comparison (PB-06) keep the High count high.

**Required before READY:** Resolve both Criticals (wire every event producer + game results + the headline filler into `news_items`; rename the duplicate button), then PB-03 (roster/minors/bullpen flags), PB-04 (call the repair path; restore "exactly 25"), PB-07 (true sliding windows), PB-06 (fix the matcher to read the ARCHETYPES table), and add integration tests that **simulate a season and assert** news population, ≥3 trades, and roster invariants — not unit tests that pre-seed the data they then check.
