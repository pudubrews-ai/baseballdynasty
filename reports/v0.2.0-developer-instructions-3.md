## §1 — Critical Fixes (do these first; both §1.1 and §1.2 required)

### §1.1 — [FIX AB-01] Evaluate the recency window BEFORE you reset it — restore the live call-up/send-down/DFA/waiver chain

**The bug:** `server/sim/rosterMaintenance.ts:120-138`. Inside `if (callUpDue)` you reset `recent_*` to 0 (`:125-131`) and THEN call `evaluateSendDowns` (`:135`) / `evaluateCallUps` (`:136`). The send-down trigger needs `recent_ab >= 20` and `recent_starts >= 4`; the call-up SP trigger needs `recent_starts >= 3`. Right after a reset they are all `0`, so nothing ever fires. Across a full 500-game season this yields 0 in-season send-downs, call-ups, DFAs, and waiver claims.

**Required fix — reorder so evaluation reads a populated window, then reset for the NEXT window:**

In `rosterMaintenance.ts`, change the `if (callUpDue)` block so the reset runs **after** the evaluators, not before:

```ts
const callUpDue = team.games_played - team.last_call_up_check_game >= 5;
if (callUpDue) {
  // 1. Evaluate FIRST — reads the window accumulated over the last ~5 games
  evaluateSendDowns(team, leagueId, league.season_number, gameNumber);
  evaluateCallUps(team, leagueId, league.season_number, gameNumber);

  // 2. THEN reset the window for the next 5-game cycle
  try {
    prepared(
      `UPDATE season_stats
         SET recent_ab = 0, recent_hits = 0, recent_hr = 0, recent_walks = 0,
             recent_er = 0, recent_ip = 0, recent_starts = 0
       WHERE league_id = ? AND season_number = ?
         AND player_id IN (SELECT id FROM players WHERE team_id = ?)`
    ).run(leagueId, league.season_number, teamId);
  } catch (err) {
    console.warn(`[rosterMaintenance] recent_* reset error for team ${teamId}:`, err);
  }
}
```

This is the minimal, correct fix and matches your §2.7(b) "reset every N games" intent — the window now holds the last ~5 games of accumulation and is read *before* it is cleared. (You may instead implement a true per-game sliding deque per architect G9, but the reorder above is sufficient and least invasive — pick one and note it.)

**Caveat to verify:** Because `evaluateCallUps` updates `last_call_up_check_game` (per the comment at `rosterMaintenance.ts:137`), confirm the reset still runs even when `evaluateCallUps` advances the cadence marker — it does, since the reset is now sequentially after both evaluators inside the same `if` block. Confirm `last_call_up_check_game` is advanced exactly once per due tick (no double-advance).

**Downstream consequences (should now work without further change, but verify):**
- Send-downs fire → open a 25-man slot → `evaluateCallUps` restore-to-25 trigger refills it → call-up news + service-time accrual.
- A send-down on a player with `options_remaining = 0` routes to DFA → waiver wire → `processWaivers` claim/expiry → waiver-claim news.
- `/api/waivers` and `waiverCount` become non-empty during a season with churn.

**Why:** The live minor-league system is the second of the four headline v0.2.0 features and is currently 100% dormant at runtime. This single reorder is the highest-leverage fix in the iteration.

### §1.2 — [FIX AB-02] Wire INJURY + MILESTONE news producers and guarantee the ≥3-trades floor

**(a) INJURY & MILESTONE producers — write `news_items` off `notable_events_json`.**

The events are computed in `game.ts` (injuries at `:758-768`, milestones at `:778-827`) and serialized into `game_log.notable_events_json` (written at `game.ts:416`) — but nothing reads them back. `insertMilestoneNewsItem` (`news.ts:202`) has zero callers; there is no injury producer.

Wire them in the game-result hook in `engine.ts`, right where you already insert the game news item (`engine.ts:429-449`). You already re-read `game_log` there for the score (`:431-433`) — extend that SELECT to also pull `notable_events_json`, parse it, and emit a news item per injury/milestone event:

```ts
const gameRow = prepared(
  'SELECT home_score, away_score, notable_events_json FROM game_log WHERE league_id = ? AND game_number = ? AND season_number = ? ORDER BY id DESC LIMIT 1'
).get(league.id, nextGame.gameNumber, league.season_number) as
  { home_score: number; away_score: number; notable_events_json: string | null } | undefined;

if (gameRow) {
  insertGameNewsItem({ ... });   // existing

  // §1.2(a): surface injury + milestone NotableEvents into the feed
  if (gameRow.notable_events_json) {
    try {
      const events = JSON.parse(gameRow.notable_events_json) as Array<{ type: string; playerId?: number; description?: string }>;
      for (const ev of events) {
        if (ev.type === 'milestone' && ev.playerId) {
          insertMilestoneNewsItem({
            leagueId: league.id, seasonNumber: league.season_number, gameNumber: nextGame.gameNumber,
            playerId: ev.playerId, teamId: /* resolve from players.team_id */ ..., 
            sourceTable: 'game_log', sourceId: /* gameRow id */ ...,
          });
        } else if (ev.type === 'injury' && ev.playerId) {
          insertNewsItem({
            leagueId: league.id, seasonNumber: league.season_number, gameNumber: nextGame.gameNumber,
            eventType: 'injury', badge: 'INJURY',
            playerId: ev.playerId, teamId: /* resolve */ ...,
            sourceTable: 'game_log', sourceId: ...,
          });
        }
      }
    } catch (err) {
      console.warn('[engine] notable-events news insert error:', scrubError(err).message);
    }
  }
}
```

Requirements:
- Resolve `teamId` from `players.team_id` for the `playerId` (the NotableEvent carries `playerId` but not team). Use a parameterized `SELECT team_id FROM players WHERE id = ?`.
- Pass **structured `playerId`/`teamId` integers only** into the inserts (never raw `description` strings) — this preserves the CISO-verified injection-safe boundary; the headline filler resolves and scrubs names centrally.
- These rows are `is_headline_pending=1` so `fillPendingHeadlines` fills them (LLM or procedural). The procedural fallbacks already exist (`news.ts:277` "Player placed on injured list.", and the milestone fallback). Confirm `'injury'` and `'milestone'` are in the headline-filler's handled set and in `BADGE_MAP` (they are: `news.ts:34`, badge CHECK includes `INJURY`/`MILESTONE`).
- Add `INJURY` and `MILESTONE` to whatever the headline filler iterates so they get filled, not left pending.

After this, `news-filter-injuries` and `news-filter-milestones` return non-empty feeds in a season where injuries/milestones occur.

**(b) Guarantee the ≥3-trades floor regardless of seed.**

`forceMinimumTrades` (`tradeDeadline.ts:228-297`) `continue`s when the buyer has no AAA/AA prospect (`:276`) and merely logs "Only forced X/needed" (`:294-295`). On dry seeds (e.g. seed 11, where only 5 AAA/AA prospects exist league-wide), this yields 0 forced trades.

Required changes in `forceMinimumTrades`:
1. **Widen the prospect search progressively.** At `:269-274`, first try `minor_level IN ('AAA','AA')`; if none, fall back to `minor_level IN ('A','Rookie')`; if still none, fall back to ANY player on the buyer with `is_on_mlb_roster=1 AND is_on_25man=0` (reserve) or the buyer's lowest-rated 25-man bench player. The floor is a synthetic guarantee, not a realism requirement — any movable asset suffices.
2. **If truly no asset exists** (buyer roster minimal), move to the next buyer in the `buyers` list rather than `continue`-ing the seller loop, so you exhaust all buyer/seller pairings before giving up.
3. **Count distinct trades, not rows.** `countLeagueTrades` (`:37-40`) counts `transaction_type='trade'` rows, but a 2-prospect trade writes 3 rows (1 buyer + 2 seller). Change the floor check so the **target is 3 distinct trades**: either count `DISTINCT` on a trade-grouping key, or (simpler) increment a `forced` counter per `executeTrade` call and drive the loop off "distinct executeTrade invocations league-wide >= 3" rather than row count. Make `needed` reflect distinct trades already executed this season (you'll need a distinct-trade count, not the raw row count from `countLeagueTrades`).
4. Keep the `LEAGUE_CAP`/`PER_TEAM_CAP` guards.

**Why:** Test Group 7 mandates "at least 3 trades complete per season" and it must hold deterministically across seeds, not just seed 42.
