### §1.1 — [FIX AB-10 Part A] Persist in-game injuries as a temporary IL stint that vacates the 25-man slot

**Root cause (verified):** `game.ts:761` fires an `injury` NotableEvent, but the only consumers are `applyInjuryTruncation` (`game.ts:830-849`, truncates the box-score line) and `engine.ts:473-484` (inserts an injury news item). **Nothing writes `is_on_25man=0` or `is_injured=1`.** So the 25-man count (`callup.ts:132-134`) never drops, and call-up Trigger 1 (`callup.ts:136`, the upgrade-free restore-to-25 path that already searches AAA→AA→A→waivers→FA) never has a vacancy to fill.

**Required changes:**

1. **Migration 008** — add the recovery-timeline column (the ONLY schema change this iteration):
   ```sql
   -- server/migrations/008_injury_return.sql
   ALTER TABLE players ADD COLUMN injury_return_game INTEGER;
   ```
   (Nullable, no default, no CHECK — SQLite ALTER cannot add CHECK. `is_injured` already exists; do not re-add it.) Ensure the migration runner picks it up (it auto-runs numbered migrations; verify the build copies 8 migrations to `dist/server/migrations`).

2. **`shared/types.ts`** — add an optional field to `NotableEvent` so the engine can read a recovery timeline:
   ```ts
   recoveryGames?: number;
   ```

3. **`server/sim/game.ts:761-768`** — when the injury event fires, attach a recovery timeline (use the seeded `rng` for determinism). Keep the existing box-score truncation untouched:
   ```ts
   if (player && player.injury_prone >= 7 && rng() < 0.05) {
     events.push({
       type: 'injury',
       playerId: batter.playerId,
       playerName: batter.playerName,
       description: `${batter.playerName} left the game with an injury`,
       recoveryGames: 3 + Math.floor(rng() * 13), // 3-15 game IL stint

   ```

4. **`server/sim/engine.ts:473-484`** — in the EXISTING `ev.type === 'injury'` branch (which already re-resolves `team_id` from the player row, injection-safe), ADD the roster write alongside the existing news insert. Only injure players currently on the 25-man:

   This runs BEFORE `runRosterMaintenance` (called at `engine.ts:498`), so the vacancy is visible to the call-up evaluator on the same tick.

5. **`server/sim/rosterMaintenance.ts`** — add an injury-recovery step BEFORE `checkRosterInvariant` (so recovered players are eligible for reintegration). Place it right after the per-team loop (after `:186`, before Step 4 at `:188`):

   // Recovered players are is_injured=0 but is_on_25man=0; they sit on the 40-man
   // as minors-eligible reserves. The invariant/call-up/send-down passes reintegrate them.
   try {
     prepared(
       `UPDATE players
        SET is_injured = 0, injury_return_game = NULL,

6. **Do NOT add an `is_injured=0` clause to the count at `callup.ts:132-134`.** The `is_on_25man=0` write in step 4 already removes the injured player from that count. ### §1.2 — [FIX AB-10 Part B] Relax the stat-based send-down threshold so the DFA→waiver path is reachable

**Root cause (verified):** `sendDown.ts:44` requires the AAA replacement to be rated **strictly greater** than the slumping MLB player. With rank-based assignment, AAA (ranks 26-32) is always ≤ the 25-man (ranks 1-24), so the condition is never met and the stat-based send-down (the path that feeds option-less players into `dfaPlayer` → waivers → claim) never fires.

**Required change — `server/sim/sendDown.ts:44`** — change:
```ts
if (!aaaReplacement || aaaReplacement.overall_rating <= player.overall_rating) continue;
```
to (within-5 band — a struggling regular can be optioned for an equal/lesser AAA replacement, which is realistic):
```ts
if (!aaaReplacement || aaaReplacement.overall_rating < player.overall_rating - 5) continue;
```

That is the only required edit for Part B. It makes Trigger 1 reachable: when a regular's recent OPS < .560 over ≥20 AB and a same-position AAA player within 5 overall exists, `executeSendDown` fires — option-ful players go to AAA (real in-season `send_down` at `game_number > 0`); option-less players route to `dfaPlayer` (`sendDown.ts:126`) → `waiver_state='dfa'` → `processWaivers` resolves the 3-game window as a `waiver_claim` or silent FA. This reaches `send_down` + `dfa` + `waiver_claim` through the real state machine.

**Optional (apply ONLY if §2 coverage needs it):** for the rating-based *call-up* path, relax `callup.ts:209` and `:270` from `>= mlbPlayer.overall_rating + 5` / `>= mlbStarter.overall_rating + 5` to `>= mlbPlayer.overall_rating - 3`. Part A already guarantees `call_up` via the injury vacancy, so this is secondary — do it only if the §2 test shows the rating-based call-up path is needed for coverage, and document the choice. Do NOT touch the call-up evaluator structure otherwise.

**Why both parts:** Part A guarantees `call_up` (robust, injury-driven). Part B guarantees in-season `send_down` + `dfa` + `waiver_claim` (the real state machine). Together they drive all four transaction types ≥1 organically — the Definition of Done.
