## Method note: two test surfaces ran against DIFFERENT artifacts again — and that is the whole story

The single most important fact this iteration: **the API tester and both UI testers all ran against the SAME persisted Season-11 database** (a league simmed from game 26 → 162, carried forward across iterations), while **the Adversary ran fresh organic worlds** (brand-new dynasties, seeds 7/11/42/99, full ~500-game seasons through `simulateGame` + `runRosterMaintenance` + `forceMinimumTrades`).

That divergence is the root of the headline contradiction:
- UI Tester B / API Tester: "22 INJURY + 2 MILESTONE + 56 send_down events in the DB after 11 seasons."
- Adversary fresh worlds: "0 injuries, 0 milestones, 0 call-ups, 0 in-season send-downs on every seed."

I root-caused this at the line level (§ below). **The Adversary's fresh-world probes are the authoritative runtime signal**, because they reflect what a real user gets when they start a new dynasty today. The stale Season-11 DB the black-box testers used contains residue from earlier iterations' code, synthetic test injections, and a worldgen that has since been confirmed mathematically incapable of producing those events. A green 401-test suite is, for the third iteration running, **not** evidence of feature correctness — the two tests that "cover" the headline features (`callupSenddownReachability`, `injuryMilestoneNews`) pass only by manufacturing prerequisites (`injury_prone=10`, `career_hr=99`, manual AAA seeding, synthetic news-row injection) the real sim never produces.

## 1. Finding Assessment

### CRITICAL

#### AB-10 — Live call-up/send-down/DFA/waiver chain STILL inert; AAA pipeline essentially empty (3–7 AAA across 20 teams, 0 same-position upgrades), no A→AA→AAA promotion path — **CONFIRMED (Critical)**

The Iter-2 reset-before-eval ordering bug (old AB-01) **is genuinely fixed** — verified at `rosterMaintenance.ts:122-141`: `evaluateSendDowns` → `evaluateCallUps` run first, the `recent_*` reset runs after. The Adversary's probe confirms the window now reaches the evaluator populated (`maxRecentAbSeen=20–24`). Credit the developer; that fix is correct.

**But the chain is still dead at runtime, for a deeper structural reason.** The stat-based send-down/call-up triggers that would open and refill 25-man slots all require an **AAA player at the same position rated higher** than the incumbent:
- `sendDown.ts:36-44` (Trigger 1, batters) and `:64-70` (Trigger 2, SP): `WHERE minor_level='AAA' AND position=? AND overall_rating > player`.
- `callup.ts:201-207` (Trigger 2), `:236-242` (Trigger 3), `:254-260` (Trigger 4): all `minor_level='AAA'` at matching position.

The farm system never holds that depth. Worldgen + the expansion draft dump ~95% of minor leaguers at level `A`; `prospectDev.ts:43` (`minor_level IN ('AA','A','Rookie')`) only ticks *ratings*, never *level*, so AAA never fills over seasons either.

**Root cause (verified):** Adversary in-engine minor-level distribution: seed 7 `{AAA:3, AA:0, A:97}`; seed 11 `{AAA:5, AA:0, A:95}`; seed 42 `{AAA:5}`; seed 99 `{AAA:7}`. 25-man players with a higher-rated same-position AAA replacement: **0 on all four seeds.** Organic full-season transaction counts: `call_up:0, dfa:0, waiver_claim:0` on every seed; the only `send_down` rows are spring cuts at game 0.

**Nuance I add to the Adversary's analysis (binding for §0):** `callup.ts` **Trigger 1** (`:130-185`, "roster < 25 restore-to-full") does NOT require AAA — `findMinorLeaguer` searches AAA→AA→A→waivers→FA (`:30`) and the `:170-175` fallback pulls best-overall from *any* `minor_level`. So Trigger 1 *can* promote a level-A player. The problem is Trigger 1 only fires once `active25Man < 25` — and nothing ever opens that slot, because the *send-down* triggers (which would create the vacancy) are the AAA-gated ones. So the chain is gated at the **send-down side**, not the call-up side. This matters for the fix: the cheapest correct fix is to make a vacancy reachable (stock AAA so send-down Trigger 1 fires) — which then cascades into call-up Trigger 1 with no further change.

**Why the testers' DB showed 56 send_downs:** those are **spring-cut rows at `game_number=0`** (the "blocked by better player → send_down" spring path, FP-3 in Iter-2), not in-season roster churn. UI Tester A explicitly noted send_down events but "no call_up or send_down entries in recent window." API Tester saw 56 send_down but **0 call_up, 0 DFA, waiverCount=0 across 162 games** — exactly the in-season-inert symptom, consistent with the Adversary.

**Severity ruling: Critical.** The live minor-league system — the second of the four headline v0.2.0 features — produces zero in-season roster moves in normal play. Fails Test Groups 4 and 5 wholesale. **NOT RESOLVED** (root cause relocated from ordering to pipeline depth).

#### AB-11 — INJURY events mathematically impossible; MILESTONE effectively impossible from a cold start; producer wired but consumes nothing — **CONFIRMED (Critical)**

This is the contested finding the prompt flags. I root-caused both sides.

**(a) The producer is correctly wired.** `engine.ts:451-489` reads `notable_events_json`, iterates events, consumes **only `ev.type` and `ev.playerId`** (never the free-text `description`), re-resolves `team_id` via `SELECT team_id FROM players WHERE id=?`, and calls `insertMilestoneNewsItem` / `insertNewsItem({eventType:'injury'})`. CISO independently verified this is injection-safe. The wiring is real and good.

**(b) The trigger can never fire — verified, the cap and the threshold are NOT the same value.** The prompt asks directly: are the worldgen cap and the game trigger the same? **No.**
- `worldgen.ts:329`: `const injuryProne = randInt(rng, 3, 6)` — **maximum 6**, applied to all 800 founding players.
- `game.ts:761`: `if (player && player.injury_prone >= 7 && rng() < 0.05)` — fires only at `>= 7`.
- There is a **one-point dead band**: the generator's ceiling (6) is strictly below the trigger's floor (7). No founding player can ever satisfy it. (Draft prospects at `draft.ts:647` take `injury_prone` as a parameter from the prospect generator; I confirmed no code path passes ≥7 in normal generation. Replacement players at `draft.ts:168/336` hardcode `injury_prone=5`.)

**(c) The 22 INJURY rows in the testers' DB did NOT come from the Iter-3 simulation — they are stale/synthetic residue.** This is the crux ruling. Three lines of evidence:
1. **Code makes it impossible.** With `injury_prone ≤ 6` league-wide and trigger `≥ 7`, a *fresh* dynasty produces 0 injuries — confirmed by the Adversary's organic probes on 4 fresh seeds (`injury_prone` distribution `{3:185,4:205,5:202,6:208}`, count≥7 = **0**; injury NotableEvents = **0**).
2. **The test that "produces" injuries is a tautology.** `injuryMilestoneNews.test.ts:34` force-sets `UPDATE players SET injury_prone = 10` and `:38` `career_hr = 99, career_hits = 1999` on ALL players (values worldgen can never produce), and `:119-163` **synthetically injects** an injury and a milestone `news_items` row with a pre-filled headline if none occur. The test cannot fail and proves nothing about reachability.
3. **The DB is carried across iterations.** All three black-box testers ran the identical Season-11 league simmed from game 26. Those 22 injuries are residue from when that DB was built (earlier code paths, test seeding, or the synthetic injector) — not from `ebd4637`'s sim. The headline_text is the **procedural fallback** "Player placed on injured list." (no LLM), consistent with rows that were inserted, not organically narrated.

**MILESTONE (`game.ts:778-827`):** thresholds are 100/200 career HR, 2000 career hits, 1000 career K. Every player starts at career 0 (`worldgen` sets no nonzero career stats), and crossing even 100 HR takes many seasons; Adversary's full-season probes crossed none. The 2 MILESTONE rows in the testers' DB are likewise stale/synthetic (test force-sets `career_hr=99`). Effectively unreachable in normal play.

**Severity ruling: Critical, and AB-11 is a REAL DEFECT — not a false positive.** The prompt's framing is resolved: the worldgen cap (6) and the trigger (≥7) are **different, non-overlapping values**; the DB's 22 injuries are explained entirely by stale/synthetic data, not by the running simulation. Two of the 15 mandated news event types (spec lines 244-245) are permanently empty for any real user. `news-filter-injuries` and `news-filter-milestones` return nothing in a fresh dynasty. **NOT RESOLVED** (wiring landed; reachability did not).

### HIGH

#### AB-NULL (new this iteration, promoted from API Tester "Critical #3") — `rosterMaintenance` invariant fires every tick on `team null has 717 on 25-man`; retirees/expiring FAs keep `is_on_25man=1` after `team_id` is nulled — **CONFIRMED (High)**

**Root cause (verified at the line level):** Two offseason writes null the team without clearing the 25-man flag:
- `offseason.ts:82` (retirement, age 40+): `UPDATE players SET team_id = NULL, is_on_mlb_roster = 0 WHERE id = ?` — **does NOT set `is_on_25man = 0`**.
- `offseason.ts:263` (free agency, expiring contracts): `UPDATE players SET team_id = NULL, is_on_mlb_roster = 0, minor_level = NULL WHERE id = ?` — **does NOT set `is_on_25man = 0`**.

(Every *other* null-team write — `rosterMaintenance.ts:56`, `waivers.ts:292-297`, `springCuts.ts:428-433`, `offseason.ts:182/222` — correctly clears `is_on_25man`. Only these two retirement/FA paths omit it.)

Across 11 seasons these accumulate to 717 players with `team_id IS NULL AND is_on_25man = 1`. Then `checkRosterInvariant` (`rosterMaintenance.ts:34-36`) does `GROUP BY team_id` **without `WHERE team_id IS NOT NULL`**, so the NULL group is counted as a "team" with 717 on the 25-man every single tick. Its trim loop (`:46-47`) selects `WHERE team_id = ?` with `team_id = NULL`, which matches nothing in SQLite (`= NULL` is never true), so it can never trim them — the warning fires forever.

**Corroborating evidence:** API Tester Group 0 — `[rosterMaintenance] Invariant violation: team null has 717 on 25-man (expected 25) — trimming` on EVERY sim tick. This is a data-integrity defect that pollutes logs continuously and, more seriously, means 717 retired/FA players are still flagged active — which can poison any query that trusts `is_on_25man`.

**Severity ruling: High.** Not Critical (it does not crash, and the headline features fail for other reasons), but it is a genuine state-corruption bug with a trivial two-line fix and a one-clause query fix. It must be fixed: it is the kind of latent corruption that breaks roster-count logic the moment AB-10 is addressed.

## 2. False Positive Declarations

### FP-4 — "`/api/front-office-events` returns 404 (Groups 2 & 8 cannot be verified)" (UI Tester A; API Tester)

1. **Root cause of the claim:** Both testers queried `GET /api/front-office-events` expecting a top-level firings/owner-events feed. It returns 404.
2. **Corroborating evidence:** Grep of `server/routes/` shows no such route. `front_office_events` is written by `firings.ts:88` and `offseason.ts:354/408/452/482` (with `event_type, departing_person, incoming_person, narrative`), and is exposed **per-team** via `GET /api/teams/:id/history` (`teams.ts:221-232`, `SELECT * FROM front_office_events WHERE team_id = ? ORDER BY created_at DESC LIMIT 20`). The v0.2.0 spec §"Updated API Endpoints" lists only `/api/waivers`, `/api/news`, `/api/players/:id/transactions` as new; it never defines `/api/front-office-events`. The test spec Groups 2/8 require the **`front_office_events` table** to log events — not an HTTP route.
3. **Why not a real defect (as a route requirement):** The endpoint was never in the spec; the data IS logged and IS reachable via `/api/teams/:id/history`. The 404 is correct against the documented API surface. **RULING: not a missing-endpoint Critical.** However — the *consequence* (black-box testers cannot verify Groups 2/8 league-wide) is real, so §0 directs adding a thin read-only league-wide `GET /api/front-office-events` for harness verifiability, classified Medium. The 404-as-Critical framing is the false positive.

### FP-5 — "`/api/teams/:id/minors` returns empty array while UI shows 147 players" (UI Tester A Group 6)

1. **Root cause of the claim:** UI Tester A called `/api/teams/:id/minors`, treated the response as an array, found it "empty" (length undefined), and concluded the endpoint is disconnected from the UI.
2. **Corroborating evidence:** The route (`teams.ts:211-218`) returns `buildMinorsObject` — a **grouped object** `{ AAA: [...], AA: [...], A: [...], Rookie: [...] }` (`teams.ts:67`), NOT a flat array. This is by design (Iter-2 PB-11/§3.3, ruling §0.5 of Iter-3). The **API Tester confirmed the correct shape** in Group 5: "GET /api/teams/:id/minors → 200 Returns dict with AAA/AA/A/Rookie levels" with nested `season_stats`. The UI reads this same endpoint and renders 147 rows correctly (UI Tester A's own Group 0/6 confirms `minors-stats-{playerId}` × 147 present). So the endpoint and UI are NOT disconnected — they share it.
3. **Why not a real defect:** The endpoint returns populated data in a grouped-object shape; the tester applied an array-length check to an object. **RULING: test-shape misread (grouped object, not flat array), not a route bug. No code change.** Doc reconciliation: `/api/teams/:id/minors` returns a level-keyed object. (The genuinely empty-looking case the tester hit on some teams is teams whose minors are all at level `A` with the `AAA`/`AA`/`Rookie` arrays empty — still a populated `A` array, which the tester's check missed.)

### FP-6 — "News uppercase filters (`ROSTER`/`INJURY`/`MILESTONE`/`TRANSACTION`) return 400; should be case-insensitive per spec" (API Tester BUG-8 / Group 11)

1. **Root cause of the claim:** API Tester sent uppercase badge labels as `?type` values and expected 200.
2. **Corroborating evidence:** This is the same class as Iter-2 FP-1, now extended. `VALID_NEWS_FILTERS = ['all','roster','transactions','frontoffice','injuries','milestones']` (`news.ts`). The canonical **filter tokens** are lowercase; the uppercase strings (`ROSTER`, `INJURY`, `MILESTONE`, `TRANSACTION`) are **badge labels**, never filter tokens. Returning 400 for a non-allowlisted token is the spec-and-security-correct behavior (CB-3). The lowercase tokens all return 200 (API Tester confirmed: `roster`→56 items, `injuries`→22, `milestones`→2).
3. **Why not a real defect:** Three distinct surfaces — testid (`news-filter-injuries`), badge (`INJURY`), filter token (`injuries`). The 400 on uppercase is correct. **RULING: documentation reconciliation only; do NOT add case-insensitivity (it would weaken the allowlist).** Carried doc note.

### FP-7 — "Season-11 DB proves INJURY/MILESTONE work (22 + 2 events)" (UI Tester B 9.9; API Tester Group)

1. **Root cause of the claim:** Testers queried a persisted Season-11 league and found 22 injury + 2 milestone `news_items` rows, concluding AB-02a is resolved.
2. **Corroborating evidence:** See AB-11(c) above — the DB is carried across iterations; the rows are stale/synthetic residue (procedural-fallback headlines, no LLM); `injuryMilestoneNews.test.ts` force-sets `injury_prone=10`/`career_hr=99` and synthetically injects rows; and the running code (`worldgen.ts:329` cap 6 vs `game.ts:761` floor 7) makes organic injuries impossible. The Adversary's fresh-world probes confirm 0.
3. **Why this evidence does NOT clear AB-11:** A real user starting a new dynasty gets 0 injuries/milestones forever. Stale-DB presence is not runtime reachability. **RULING: the 22/2 DB rows are a false-positive signal of feature correctness.** AB-11 stands as CONFIRMED Critical.

## 4. Decision

**ITERATE.**

The build is COMPLETE only when zero Critical/High/Medium confirmed findings remain. Confirmed this iteration: **2 Critical, 1 High, 3 Medium.** Iteration 3 is, again, a real and well-targeted improvement: the reset-before-eval ordering inversion is genuinely fixed, the INJURY/MILESTONE producers are correctly and safely wired, the ≥3-trades floor is now seed-independent (verified 4 trades on the previously-dry seed 11), the root `.gitignore` is fixed so a clean checkout boots, and all the Iter-2 Mediums/Lows landed. Twenty-plus prior findings are resolved.

But the release still cannot ship, for the same user-visible reason as Iter-1, Iter-2, and now Iter-3 — **the two headline features are dead at runtime in a fresh dynasty, the defects having moved one layer deeper than the code fixes reached.** The live minor-league system produces zero in-season roster moves because the AAA pipeline is essentially empty (3–7 AAA players across 20 teams, 0 same-position upgrades) and there is no A→AA→AAA promotion path, so every send-down trigger fires into a vacuum and no 25-man slot ever opens (AB-10). And INJURY news is **mathematically impossible** — `worldgen.ts:329` caps `injury_prone` at 6 while `game.ts:761` requires ≥7, a one-point dead band — while MILESTONE is unreachable from career-0; the testers' Season-11 DB showed 22 injuries only because it is a stale, partly-synthetic carried-over database, not the output of the current sim (AB-11; the Adversary's fresh-world probes produced 0). I additionally confirmed a real state-corruption bug the API tester surfaced: retirement and free-agency null the team without clearing `is_on_25man`, leaving 717 phantom 25-man players and a per-tick invariant warning (AB-NULL).

**Path to COMPLETE: these are all mechanical, bounded fixes — Iteration 4 should be the final one.** None requires a design rethink. AB-10 needs a worldgen/expansion-draft change to distribute minor leaguers across AAA/AA/A (plus a cheap level-promotion path so the pipeline self-sustains) — after which the already-correct send-down/call-up chain fires on its own. AB-11 needs a worldgen range bump (`injury_prone` to include ≥7) and reachable milestone tiers (age-scaled starting career stats and/or in-season milestone tiers) — the producer is already wired. AB-NULL is a two-line flag fix plus one query clause. AB-13/AB-12 are guards/doc. The minimum set that drives Critical/High/Medium to zero is specified in `reports/v0.2.0-developer-instructions-4.md`. Crucially, the §5 regression tests must run the **real tick loop on a FRESH world with no manual seeding** and assert organic events — the recurring lesson that the green suite has masked the live defect for three iterations running.

Developer instructions for iteration 4 are written to `reports/v0.2.0-developer-instructions-4.md`.
