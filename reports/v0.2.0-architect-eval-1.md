## Method note on the two contradictory test surfaces

The reports disagree because they ran against **two different artifacts**:

- **API Tester** and **CISO/Adversary** inspected the **v0.2.0 branch source / a freshly-built dist** and correctly found the News pipeline inert and call-ups/DFA dormant.
- **UI Tester A and B** ran against a **stale Season-7 dist + existing v0.1.0 DB** on branch `feature/v0.1.0-initial-build` (UI Tester A header says so explicitly). That is why they saw `game-ticker` (a v0.1.0 League-tab element that still exists at `League.tsx:163`) and an empty `news_items` table — the running build predated several v0.2.0 client changes.

Where a UI finding is an artifact of the stale build, I say so and root-cause against current source. Where the defect is real in current source, I confirm it. Both sets converge on the same root cause: **the News subsystem writes nothing during simulation, and the live-roster triggers never fire.**

## 1. Finding Assessment

### CRITICAL candidates

#### PB-01 / NB-1 / API-Tester#2 — News feed event producers never called; `news_items` always empty — **CONFIRMED (Critical)**
**Root cause (with evidence):** Every writer in `server/sim/news.ts` (`insertNewsItem:49`, `insertGameNewsItem:93`, `insertRosterNewsItem:123`, `insertTransactionNewsItem:146`, `insertFrontOfficeNewsItem:171`, `insertMilestoneNewsItem:192`) is defined but **has zero production callers.** Grep of `server/` excluding `sim/news.ts` and `tests/` for all six writers returns empty. The LLM filler `callNewsHeadlinesBatch` (`llm.ts:392`), `callTransactionFlavorsBatch` (`llm.ts:447`), and `resetNewsCallsThisSeason` (`llm.ts:172`) likewise have no callers. No code anywhere flips `is_headline_pending` from 1→0.

The event producers write only to `transactions`/`front_office_events`:
- DFA: `waivers.ts:37-41`; waiver claim/expiry: `waivers.ts:247-251, 274-278`.
- Call-up: `callup.ts:96-100`; send-down/DFA: `sendDown.ts:101-105, 110`.
- Spring release/send-down: `springCuts.ts:151-162, 292-303`.
- Firings: `firings.ts:86-90` (front_office_events), `:102-106, 248-256` (transactions).
- Trades: `tradeDeadline.ts:125-138`. Owner sold/died: `offseason.ts` front_office_events only.
- Game results: no `insertGameNewsItem` call exists in `game.ts` or `engine.ts`.

`engine.ts:103-106` reads `MAX(id) FROM news_items` for `lastNewsId` but nothing ever inserts a row. Consequently `GET /api/news` returns `[]` for all filters during real play, the ticker (`App.tsx:31`, `if (items.length === 0) return null`) renders nothing, and every "generates a news item" promise in the spec is unmet at runtime.

**Severity ruling: Critical.** This is the single largest of the four headline v0.2.0 features and it produces zero data. Fails Test Group 9 wholesale plus spec lines for call-up/send-down/DFA/trade/firing/owner/injury/milestone news. The green test suite missed it because `newsApi.test.ts` calls `insertNewsItem` from the test body and `e2eBenchmark.test.ts` asserts nothing on `news_items`.

#### PB-02 — Duplicate `new-dynasty-button` testid — **CONFIRMED (Critical for the harness)**
**Root cause:** `client/src/App.tsx:164` (header button, always rendered) and `App.tsx:213` (welcome-screen button, rendered when `noLeague === true`) both carry `data-testid="new-dynasty-button"`. When `noLeague` is true — the exact first-load state — both are mounted, so Playwright `getByTestId('new-dynasty-button')` resolves to 2 elements and throws. The Architect's D13 rename (`-header` / `-welcome`) was not applied. Re-violates Universal Lesson #4.
**Severity ruling: Critical (harness-blocking).** Gates the first step of nearly every UI test group.

#### API-Tester — Call-ups / send-downs / DFA never trigger across 500+ games — **CONFIRMED (Critical)**
**Root cause:** The integration IS hooked into the tick loop — `engine.ts:431` calls `runRosterMaintenance` after `simulateGame`, and `rosterMaintenance.ts:86-90` calls `evaluateSendDowns`/`evaluateCallUps`. The problem is the **trigger conditions are unreachable in practice**:
- Call-up Trigger 1 (`callup.ts:122`) requires `active25Man < 23`. With a 60-player expansion draft and no injury system depleting the 25-man, the active count never drops below 23. Triggers 2-4 require a AAA player rated `>= MLB starter + 5` at the same position — after spring cuts push the worst players to AAA, AAA depth is rated *below* the 25-man, so this never fires either.
- Send-down (`sendDown.ts:19-31, 50-56`) requires `recent_ab >= 20` / `recent_starts >= 4` AND a higher-rated AAA replacement — same AAA-is-worse problem, compounded by PB-07 (the "recent" window is actually season-cumulative, diluting any slump).

So the entire DFA → waiver → claim chain is dead because its only mid-season entry point (send-down-with-no-options, call-up-needs-40-man-slot) never executes. API Tester saw zero `[callup]`/`[sendDown]`/`[waiver]`/`[dfa]` logs across 7 seasons.

**Severity ruling: Critical.** The live minor-league system — the second headline feature — is effectively inert. Fails Test Group 4 and 5 wholesale. The waiver state machine itself is well-built (CISO CB-07 RESOLVED) but never exercised.

#### API-Tester — Server requires migration-copy fix (tsc doesn't copy `.sql` to dist) — **CONFIRMED (Critical)**
**Root cause:** `package.json:13` build script is `tsc -p tsconfig.server.json && vite build && npm run security:bundle-grep`. `tsc` compiles only `.ts` → `.js`; it does **not** copy `server/migrations/*.sql` into `dist/`. A fresh build from artifacts has no migrations directory, so the runner finds nothing and the server crashes on first query with `no such column: waiver_state` / `no such table: leagues`. The API Tester had to copy the directory manually; the dist currently contains them only because of that manual copy (`dist/server/server/migrations/` has 001-007 present now, but a clean `npm run build` would not reproduce that).
**Severity ruling: Critical (deploy-blocking).** Every fresh deployment crashes on boot.

### HIGH candidates

#### Trades fire but not written to transactions table — **FALSE POSITIVE (see §2)**
The API Tester reported this, but trades ARE written to `transactions` with `transaction_type='trade'` (`tradeDeadline.ts:126-138`). UI Tester A independently confirmed **42 trade rows** in the transactions table. Root cause of the API Tester's miss: their `/api/transactions` query window was dominated by 51,068 `dev_tick` rows (UI Tester A count), pushing trades out of the last-50 view. Not a defect. See §2.

#### PB-03 — Optioned 40-man players: minors API hides them, MLB roster shows them, bullpen pitches them — **CONFIRMED (High)**
**Root cause:** Under the C2 model an optioned player is `is_on_mlb_roster=1, is_on_25man=0, minor_level='AAA'`. The lineup/SP selection was migrated to `is_on_25man` (`game.ts:60,101`) but three supporting queries were not:
- Minors API: `teams.ts:18` and `:34` filter `is_on_mlb_roster = 0` → an optioned AAA player (`is_on_mlb_roster=1`) is invisible in `/api/teams/:id/minors`.
- MLB roster API: `teams.ts:123, 163` filter `is_on_mlb_roster = 1` → optioned AAA players appear on the MLB roster list.
- Bullpen: `game.ts:130,133,264,267` filter `is_on_mlb_roster = 1` → an optioned reliever still pitches relief in MLB games, inconsistent with the lineup fix at `game.ts:60`.

**Severity ruling: High.** Roster integrity is internally inconsistent; fails Test G6 for optioned players. (Mostly latent today because PB-03's trigger — send-downs — rarely fires per the call-up Critical, but the fix must land for correctness and to support the unblocked roster churn.)

#### news-ticker not global / shows 20 not 5 / uses game-ticker — **CONFIRMED in part (High), with stale-build caveat**
**Root cause:** Current source DOES have a correct single `news-ticker` in `App.tsx:35` rendered once in chrome, showing `?limit=5`. The `game-ticker` the testers saw is a **separate** pre-existing v0.1.0 element at `League.tsx:163` (League tab only, 20 items) — it was never removed and is not the news ticker. The real defect: the `news-ticker` renders `null` whenever `items.length === 0` (`App.tsx:31`), which is **always**, because of PB-01 (empty `news_items`). So the live ticker is non-functional in practice.
**Severity ruling: High** — but the fix is PB-01 (populate news). Secondary cleanup: the visibility gate `state.phase !== 'no_league'` (`App.tsx:199`) checks a phase value `mapPhase` (`engine.ts:77-88`) never returns (it throws on unknown), so the gate is effectively always-true — see PB-16. The `game-ticker` is out of v0.2.0 spec scope; leave or optionally hide on non-League tabs.

#### news-tab nav testid is `nav-news` not `news-tab` — **CONFIRMED (High)**
**Root cause:** The nav button uses `data-testid="nav-${tab.id}"` (`App.tsx:178`) → `nav-news`. `news-tab` is placed on the **view root div** in `News.tsx:71`, not on the nav button. D13/AB-13 mandated the nav button carry BOTH `nav-news` and `news-tab` as synonyms. It does not.
**Severity ruling: High (harness contract).** UI Tester A 11.5 and UI Tester B 9.1 both fail on this.

#### waivers-list and waiver-player-{id} testids absent — **CONFIRMED (High)**
**Root cause:** Grep of `client/src/` for `waivers-list` / `waiver-player` returns nothing. There is no waivers UI component at all. The `/api/waivers` endpoint exists and works, but no view renders it.
**Severity ruling: High.** Test spec requires these testids; UI Tester B A.1/A.2 fail. (Spec §"Updated API Endpoints" lists `/api/waivers`; the data-testid reference block lists `waivers-list`/`waiver-player-{playerId}` — a UI surface is required.)

#### minors-stats-{playerId} testids absent — **CONFIRMED (High → see severity note)**
**Root cause:** The Teams minors tab renders player rows but attaches no `data-testid="minors-stats-{playerId}"` (UI Tester B A.3; grep confirms none in `client/src/`).
**Severity ruling: High** for harness compliance (spec data-testid reference). The underlying API data exists (`teams.ts:11-73`); this is a missing frontend attribute plus the PB-11 shape mismatch.

#### Spring cuts: 25-man invariant violation (team 15 has 27) — **CONFIRMED (High)**
**Root cause:** `springCuts.ts` only ever cuts **down** and `continue`s past any candidate whose removal would breach a position minimum (`:126`). If the only remaining over-25 players all hold minimum positions, the loop exits with the roster still above 25. The repair-up function `repairPositionMinimums` (`springCuts.ts:195`) is **dead code — zero callers** (grep confirms). Teams entering a season under 25 (post-retirement/FA attrition) are never brought up to 25 either. The invariant check `checkRosterInvariant` (`rosterMaintenance.ts:30-44`) only **logs** when `>25` and never repairs, and never checks `<25`. The team's own test relaxes the gate to `toBeLessThanOrEqual(30)` (`springCuts.test.ts:89`) instead of the Architect's Phase-4 "exactly 25" gate.
**Severity ruling: High.** Roster sizes drift 24-30; violates the spec's "exactly 25 on `is_on_25man`" and the Phase-4 gate. Compounds PB-03 (a short team plus 40-man bullpen draw).

#### PB-06 — Trade matcher uses a broken stub comparison; D11 table bypassed — **CONFIRMED (High)**
**Root cause:** `tradeDeadline.ts:79` computes `prospectsRequired` via `sellerArchetype === ARCHETYPES_DATA['analytics']`, where `sellerArchetype = getArchetype(...)` returns an `ARCHETYPES` reference (`:76`) and `ARCHETYPES_DATA` is an unrelated one-field stub object declared at `:98-100`. The two object references can never be `===`, so the first operand is **always false**; correct behavior survives only by the `|| seller.gm_archetype === 'analytics'` fallback. `buyerArchetype` (`:75`) is computed and never read. Old-school's "demands proven players back" (D9 step 4) is not implemented — only prospect count varies.
**Severity ruling: High.** Not a crash, but it violates D11 ("all archetype logic reads from the ARCHETYPES table, never hardcoded") and is a maintenance hazard plus a spec-intent miss. Demote to Medium would be defensible since trades still occur (42 observed); I keep it High because the dead/confused code masks whether archetype-driven trade fairness works at all and must be corrected to satisfy the spec's archetype behaviors.

#### PB-07 — "Recent" rolling windows never slide → OPS/ERA triggers are season-cumulative — **CONFIRMED (High)**
**Root cause:** `game.ts:872-875` (batters) and `:922-925` (pitchers) only **add** to `recent_ab/recent_hits/recent_starts/recent_er/recent_ip`; there is no decay, eviction, or window cap anywhere (grep for any `recent_* = ... -` is empty). So `recent_ab == at_bats` and `recent_starts == games_started` for the whole season. Call-up (`callup.ts:174-183, 210-218`) and send-down (`sendDown.ts:19-31, 50-56`) read these as season figures. The spec's "last 15 AB" / "last 3 starts" / "20+ AB" / "4+ starts" all degrade to season-to-date averages.
**Severity ruling: High.** Directly defeats the recency the spec requires and is a contributing cause of the dormant call-up/send-down Critical (a recent slump never stands out against a diluted season number).

#### PB-05 — Owner death/sale never reaches News; "fresh personality" only probabilistic — **CONFIRMED (High, but split)**
**Root cause:** The owner-death personality **bug fix is correct** — `offseason.ts` draws the heir/new-owner personality from all 5 `OWNER_PERSONALITIES` and writes both `owner_name` and `owner_personality` (Adversary verified `offseason.ts:377,388,407-408`). The two real defects: (a) owner events write only `front_office_events`, no `insertFrontOfficeNewsItem` → never in the News feed (a facet of PB-01); (b) randomization re-rolls but does not guarantee a value different from the parent (20% collision with 5 options), so a "must differ" test would be flaky.
**Severity ruling: High** for the news-surfacing facet (folds into PB-01). The personality fix itself is **CONFIRMED RESOLVED** (the v0.1.0 headline bug is fixed). The "fresh ≠ parent" semantics is a **Low** spec-clarity item (decide and document).

Same risk for the meddling early-return path. This makes manager evaluation far more frequent than spec.

## 2. False Positive Declarations

### FP-1 — "Trades fire but are not written to the transactions table" (API Tester #6)
1. **Root cause of the claim:** The API Tester queried `/api/transactions` and saw only `auto_balance`, `free_agent_signing`, `dev_tick`, `call_up` in the last-50 window.
2. **Corroborating evidence:** `tradeDeadline.ts:122-138` builds a `narrative` and runs two `INSERT INTO transactions ... 'trade'` statements (one for the buyer, one per prospect for the seller). `executeTrade` is invoked by both `evaluateTradeDeadline` (`:199`) and `forceMinimumTrades` (`:260`). **UI Tester A independently confirmed 42 `trade` rows in the transactions table.** The 51,068 `dev_tick` rows (UI Tester A) flood the recent-rows window.
3. **Why not a real defect:** Trades ARE persisted with `transaction_type='trade'` and a non-null narrative. The miss is a query-window artifact (no pagination/type filter on the tester's call), not missing writes. (The genuine adjacent issue — trades produce no `news_items` row — is PB-01, already Critical.)

### FP-2 — "Schema field-name mismatches: `level`, `is_on_40man`, `fa_eligible`, `type`" (UI Tester A)
1. **Root cause:** These are the **test spec's** assumed names, contradicted by deliberate Architect rulings.
2. **Corroborating evidence:** `level`→`minor_level` (+`is_on_mlb_roster`/`is_on_25man`) per C2/AB-11; `is_on_40man` collapsed into `is_on_mlb_roster` per C2 (migration 007 never adds `is_on_40man`); `fa_eligible`→`free_agent_eligible` per D8 (`007:88`); `type`→`transaction_type` (column name since `001_init.sql`). The implementation is internally consistent and matches the binding rulings.
3. **Why not a real defect:** The code is correct per spec; the test spec drifted. Residue is documentation only (§3), not a code change.

### FP-3 — "transactions uses `narrative` not `flavor`" (UI Tester A)
1. **Root cause:** Spec data-testid/field reference loosely called the LLM one-liner "flavor"; the column has always been `narrative`.
2. **Corroborating evidence:** `narrative` is the column in `001_init.sql` and every v0.2.0 INSERT (`tradeDeadline.ts:127`, `waivers.ts:39`, etc.). No ruling renamed it.
3. **Why not a real defect:** Correct column. The real issue (NULL narratives for procedural transactions because D14 batch flavor is unwired) is tracked separately as PB-12 (Medium).

### FP-4 — "news-ticker uses `game-ticker`" (UI Tester A/B, partial)
1. **Root cause of the claim:** Testers ran against a stale Season-7 dist on the v0.1.0 branch and found `game-ticker` (League tab) but no populated `news-ticker`.
2. **Corroborating evidence:** Current source HAS a correct `news-ticker` (`App.tsx:35`) and a SEPARATE pre-existing `game-ticker` (`League.tsx:163`). The two coexist; they are different elements.
3. **Why partly not a defect:** The `news-ticker` testid IS present and correctly placed (single instance in chrome, `?limit=5`). The **real** defect is that it renders empty because `news_items` is never populated (PB-01) and the visibility gate is a no-op (PB-16). So "uses game-ticker instead of news-ticker" is a misread of a stale build; the underlying "ticker shows nothing" is real and == PB-01.

### FP-5 — "Owner death personality bug not fixed / unverifiable" 
1. **Root cause of the claim:** No HTTP trigger and `owner_personality` not in the API made it unverifiable for black-box testers; events are probabilistic and didn't fire in their windows.
2. **Corroborating evidence:** `offseason.ts:407-408` (owner_died) and `:377,388` (owner_sold) draw from all 5 `OWNER_PERSONALITIES` and write both name and personality. Adversary verified the fix is real and correct.
3. **Why not a real defect:** The headline bug IS fixed. The verifiability gap is the separate Medium "owner_personality not in API," and the news-surfacing gap is PB-01. The fix itself is RESOLVED.
