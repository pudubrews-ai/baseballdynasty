## Changes Made

### Fix 1 — `baseball-dynasty/scripts/copy-migrations.mjs` line 13

**Before:**
```js
const dst = join(projectRoot, 'dist', 'server', 'migrations');
```

**After:**
```js
const dst = join(projectRoot, 'dist', 'server', 'server', 'migrations');
```

Root cause: `tsc -p tsconfig.server.json` (with `include: ["server","shared"]` and `outDir: "dist/server"`) preserves the common root, compiling `server/db.ts` to `dist/server/server/db.js`. At runtime `__dirname = dist/server/server/`, so the loader looks for `dist/server/server/migrations/`. The copy script was writing to `dist/server/migrations/` (one level up), causing `[db] No migrations directory found` and zero migrations applied in production.

### Fix 2 — `baseball-dynasty/package.json` scripts.start

**Before:**
```json
"start": "node dist/server/index.js",
```

**After:**
```json
"start": "node dist/server/server/index.js",
```

`dist/server/index.js` does not exist. The compiled entry point is `dist/server/server/index.js`.

### Fix 3 — `baseball-dynasty/server/tests/buildPackaging.test.ts`

- Changed `DIST_MIGRATIONS` constant to `dist/server/server/migrations` (runtime-read path).
- Added `DIST_SERVER_SERVER` and `DIST_DB_JS` constants pointing to `dist/server/server/db.js`.
- Added test: `compiled db.js exists at dist/server/server/db.js`.
- Added test: `migrations/ is a sibling of compiled db.js (runtime-read path)` — asserts `path.join(path.dirname(DIST_DB_JS), 'migrations')` exists, tying the test to the actual `__dirname` resolution rather than a hard-coded guess.
- Updated existing copy-check test to assert `dist/server/server/migrations` and `>= 8` files.
- Preserved the "dist not built → skip" guard.
