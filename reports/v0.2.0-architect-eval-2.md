## 1. Finding Assessment

### CRITICAL

#### AB-01 — Live call-up/send-down/DFA/waiver chain still inert; the §2.7 "rolling window" fix zeroes `recent_*` immediately before the evaluator reads them — **CONFIRMED (Critical)**

**Root cause (verified at the line level):** `server/sim/rosterMaintenance.ts:120-138`. Inside `runRosterMaintenance`, the per-team block does, in this exact sequence on the same tick:

```ts
const callUpDue = team.games_played - team.last_call_up_check_game >= 5;   // :120
if (callUpDue) {
  prepared(`UPDATE season_stats
    SET recent_ab = 0, recent_hits = 0, recent_hr = 0, recent_walks = 0,
        recent_er = 0, recent_ip = 0, recent_starts = 0
    WHERE league_id = ? AND season_number = ?
      AND player_id IN (SELECT id FROM players WHERE team_id = ?)`).run(...);   // :125-131  RESET TO 0
  evaluateSendDowns(team, leagueId, league.season_number, gameNumber);          // :135  reads recent_ab — now 0
  evaluateCallUps(team, leagueId, league.season_number, gameNumber);            // :136
}
```

The send-down trigger requires `recent_ab >= 20` (batters) and `recent_starts >= 4` (pitchers); the call-up SP-ERA trigger requires `recent_starts >= 3`. After the reset on line 125-131, every one of these reads `0`, so **none can ever fire.** The window is reset and then immediately evaluated within the same `if (callUpDue)` block, so the evaluator always sees a freshly-zeroed window. This is a logic-ordering inversion of the developer's own §2.7(b) intent ("reset every N games so the window holds the last ~5 games") — the reset must happen *after* (or independently of) the read, never before it.

**Corroborating evidence:** Adversary in-engine probe, full 500-game seasons through the real `runRosterMaintenance` loop, seeds 7 and 11: `inSeasonSendDowns: 0, totalCallUps: 0, totalDfa: 0, totalWaiverClaims: 0`. A replica of the reset+eval ordering observed `MAX(recent_ab)=0` across all 32 due-tick evaluations. The only `send_down` rows in the DB are spring-cut rows at `game_number=0`. The API Tester independently saw the same symptom black-box (Group 4 BUG-1 "waiverCount always 0 across 4 seasons"; Group 5 BUG-2 "zero call_up events, zero mid-season send_down"). UI Tester B saw a Season-11 feed of only `game_result, send_down, release, gm_fired, manager_fired, free_agent_signing` — no in-season roster churn TRANSACTION items.

**Why the green suite missed it:** `callupSenddownReachability.test.ts` (§5.2) calls `evaluateSendDowns(team, …)` **directly** after manually setting `recent_ab=25`, bypassing `runRosterMaintenance` and therefore the reset-before-eval ordering. `newsIntegration.test.ts` runs the real loop but its non-game-news assertion is `>= 0` (always true), so it cannot catch zero roster churn.

**Severity ruling: Critical.** This is identical to the Iter-1 Critical "call-ups/send-downs/DFA dormant" (eval-1 §1, C-3). The live minor-league system — the second of the four headline v0.2.0 features — produces zero in-season roster moves at runtime. The DFA→waiver→claim state machine, call-up logic, and service-time accrual are all well-built but never reached. Fails Test Group 4 and 5 wholesale. **NOT RESOLVED.**

#### AB-02 — INJURY & MILESTONE event types have NO news producer; the ≥3-trades floor is not guaranteed — **CONFIRMED (Critical)**

Two distinct runtime gaps, both defeating spec promises the test suite claims to cover.

**(a) Missing producers — INJURY & MILESTONE (verified).** Grep of `server/` (excluding `/tests/`) for `insertMilestoneNewsItem` returns **only its definition** at `news.ts:202` — zero callers. There is no `insert*NewsItem` call with `eventType:'injury'` anywhere (only the `'injury'` type literal at `news.ts:34` and the procedural-fallback string at `news.ts:277`). The events ARE computed: `game.ts:758-768` pushes `type:'injury'` NotableEvents, and `checkCareerMilestones` (`game.ts:778-827`) pushes `type:'milestone'` events. They are serialized into `game_log.notable_events_json` (written at `game.ts:416`) and **never read back** — `notable_events_json` has no consumer anywhere in `server/sim/` outside the write. So spec §"Event Types That Generate News Items" lines 244-245 ("Player injury | INJURY | Yes", "Player milestone | MILESTONE | Yes") are unmet, and `news-filter-injuries` / `news-filter-milestones` return an empty feed at all times.

**Corroborating evidence:** Adversary probe (seed 7) badge distribution `GAME:162, ROSTER:3, FRONT OFFICE:1, INJURY:0, MILESTONE:0`. UI Tester B 9.9/9.10 saw both filters return 0 items in Season-11 data; UI Tester A confirmed both filter buttons render but no items exist.

**(b) ≥3-trades floor not guaranteed (verified).** `forceMinimumTrades` (`tradeDeadline.ts:228-297`) requires the buyer to have an AAA/AA prospect (`:269-276`); if `prospects.length === 0` it `continue`s (`:276`), and at the end merely logs `"Only forced X/needed"` (`:294-295`). No fallback to A/Rookie prospects, no prospect-less/cash package. The §5.5 `tradeFloor.test.ts` passes because it uses friendly seed 42, sets artificial 35-10/10-35 standings, asserts `>= 3` **rows** (one 2-prospect trade = 3 rows), and bypasses `runGameTick`.

**Corroborating evidence:** Adversary in-engine probe through `shouldFireTradeDeadline`→`forceMinimumTrades`: **seed 11 → 0 trades** (only 5 AAA/AA prospects exist league-wide because expansion-draft minor leaguers mostly sit at level `A`); seed 42 → 4 rows (2 trades). API Tester saw only 2 trades/season in the news window for seasons 9-10 (consistent with a soft, seed-dependent floor). Spec §"Trade Deadline Logic" intent and Test Group 7's "at least 3 trades complete per season" are not guaranteed.

**Severity ruling: Critical.** I keep AB-02 at Critical rather than splitting. (a) leaves 2 of the 15 mandated news event types permanently empty — a headline-feature completeness failure on the News Feed, the first of the four v0.2.0 features. (b) is a hard floor in the test spec that is unmet on a substantial fraction of seeds, and is non-deterministic, which is itself disqualifying. Both are unambiguous spec violations with reproducible runtime evidence. (Defensible to call (b) High in isolation; the combined entry is Critical because (a) alone fails a headline feature's completeness.)

## 2. False Positive Declarations

### FP-1 — "GET /api/news?type=FRONT+OFFICE returns 400 (should be 200)" (API Tester BUG-3 / Group 11)

1. **Root cause of the claim:** The API Tester sent the **badge label** `FRONT OFFICE` (URL-encoded as `FRONT+OFFICE`) as the `?type` value, expecting it to be a valid filter token.
2. **Corroborating evidence:** `server/sim/news.ts:242-249` defines `VALID_NEWS_FILTERS = ['all','roster','transactions','frontoffice','injuries','milestones']`. The route (`news.ts:27-34`) validates `?type` against that allowlist and returns the exact `400 {"error":"Invalid event type filter"}` for anything not in it — by design (CISO CB-3, verified RESOLVED). `?type=frontoffice` returns 200 and filters correctly (API Tester confirmed in the same report: "GET /api/news?type=frontoffice PASS 200"). The spec's data-testid is `news-filter-frontoffice`, the badge label is `FRONT OFFICE`, and the **filter token** is `frontoffice` — three different surfaces. `FRONT OFFICE` is the badge string, never a filter token.
3. **Why not a real defect:** The valid filter string IS `frontoffice` (lowercase, no space). Returning 400 for `FRONT OFFICE` is correct, spec-aligned, security-validated behavior. **RULING: `frontoffice` is the canonical filter token.** The actionable residue is documentation only (see §0 ruling + DoD). Not a code change.

### FP-2 — "front_office_events details_json always null; departing/incoming names not logged" (API Tester BUG-5 / Groups 2, 8)

1. **Root cause of the claim:** The API Tester read the `details_json` column on `front_office_events` rows and found it NULL, concluding departing/incoming names are not logged.
2. **Corroborating evidence:** `front_office_events` has BOTH a `details_json` column (`001_init.sql:165`, carried into `007:146`) AND dedicated `departing_person` / `incoming_person` columns (`001_init.sql:175-176`). Every v0.2.0 writer populates `departing_person`/`incoming_person`: `firings.ts:91` (`logFrontOfficeEvent` runs the INSERT with both names) and `offseason.ts:354,395,439,469` (explicit `departing_person, incoming_person` columns). The data IS logged — under the dedicated columns, not the legacy `details_json` blob. The route `teams.ts:217` does `SELECT *`, so the response carries `departing_person`/`incoming_person`; the tester queried the wrong field.
3. **Why not a real defect:** Departing/incoming names ARE persisted and returned. `details_json` is a legacy/optional column the v0.2.0 design chose not to use. **RULING: this is test-spec field-name drift (`details_json` → `departing_person`/`incoming_person`), not missing data.** Documentation residue only (DoD §6.7 reconciliation). Not a code change. (If a stricter consumer truly needs a `details_json` blob, that is a v0.3.0 nicety, explicitly DEFER.)

### FP-3 — "Spring training cuts use `send_down` not `release` for released players" (API Tester BUG-7 / Group 3)

1. **Root cause of the claim:** The API Tester expected spring-cut **released** players to carry `transaction_type='release'` and observed `send_down` rows at `game_number=0`, concluding releases are misclassified.
2. **Corroborating evidence:** The spec (§"Spring Training Cuts") prescribes **two distinct** outcomes: "Overall < 45 with no upside → **release**" and "Blocked by better player → **send down**." Both are spec-correct outcomes; the cut type depends on the player. The Adversary verified spring cuts write both: `springCuts.ts` release path (→ FA, `transaction_type='release'`) and send-down path (→ AAA, `transaction_type='send_down'`). The API Tester observed only `send_down` rows in their window because, in the seasons they watched, the cut candidates were "blocked by better player" (send-down), not "<45 with no upside" (release). UI Tester B's Season-11 feed shows BOTH `send_down × 17` AND `release × 1` — confirming releases DO occur with `transaction_type='release'`.
3. **Why not a real defect:** `send_down` and `release` are both correct, spec-defined spring-cut outcomes; which one applies is player-dependent. The implementation emits both with the right types. The tester's expectation that *all* spring cuts are `release` is the misread. Not a code change. (The only genuine adjacent residue — `release`/`send_down` rows producing news — is already handled: PB-12 resolved; release/send-down news items are wired.)
