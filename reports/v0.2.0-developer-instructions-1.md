## §0 — Architect Rulings on Contested Points

### [AB-01 RULING] — The "game N" time-unit mapping (BINDING; this is the spine of the release)

There are two counters. **Every** v0.2.0 timing rule is mapped explicitly below. Do not infer; use this table verbatim.

**Principle:** All thresholds and cadences that the spec phrases as "game N" or "every N games" or "N games of sim time" are **PER-TEAM** (`teams.games_played`), because that matches MLB intuition and the existing `shouldFireTradeDeadline`. The ONE exception is the league-wide prospect-dev / service-time batch cadence, which keys off `current_game_number % 10` purely as an efficiency boundary (see notes).

| Spec phrase | Counter | Meaning | Where evaluated |
|---|---|---|---|
| "evaluate trade posture at game 30" | per-team `games_played` | a team's posture is set the first maintenance pass after **that team** reaches `games_played >= 30` | trade module, per team |
| "trades happen at game 33-37" / "window [30,37]" | per-team `games_played` | a team may execute a deadline trade while its `games_played` is in `[30, 37]` | trade module, per team |
| hard deadline marker (`trade_deadline` txn) | unchanged from v0.1.0 | fires once when >= 10 teams have `games_played >= 35` | `season.ts:shouldFireTradeDeadline` (keep as-is) |
| call-up "evaluated every 5 games" | per-team `games_played` | run call-up eval for a team when `games_played - last_call_up_check_game >= 5` | `rosterMaintenance`, per team |
| send-down evaluation | per-team `games_played` | same 5-game per-team cadence as call-up | `rosterMaintenance`, per team |
| manager firing "evaluated every 5 games" | per-team `games_played` | `games_played - last_firing_check_game >= 5` | `firings`, per team |
| GM firing "evaluated every 10 games" | per-team `games_played` | `games_played - last_gm_firing_check_game >= 10` | `firings`, per team |
| "firing when 8 games under .500" | derived from `wins`/`losses` | `games_under_500 = losses - wins >= threshold` AND `games_played >= 12` | `firings` |
| waiver "3 games of sim time" | per-team `games_played` of the **DFA'ing team** | window end = DFA team's `games_played + 3` at DFA moment | `waivers` |
| prospect dev "every 10 games" | league-wide `current_game_number % 10 == 0` | efficiency batch boundary; iterate eligible prospects league-wide | `prospectDev` |
| service time "172 games = 1 year" | per-player accumulator | +1 per game the player is on the 25-man (see [AB-05 RULING] for the constant) | `serviceTime` |
| service-time accrual batch | league-wide `current_game_number % 10 == 0` | efficiency batch boundary; accrue for is_on_25man players | `serviceTime` |

**Notes / rationale:**
- Per-team counters require knowing **which teams played in this game**. Every game has exactly 2 teams (home, away). The maintenance hook (§4) runs **only for the home and away teams of the game just simulated** — NOT for all 20 teams every tick. This makes "every 5 per-team games" exact: a team is evaluated only on ticks where it played, and its `games_played` advances by 1 each such tick.
- The two league-wide `% 10` boundaries (prospect dev, service time) are batch-efficiency choices (Risk R5). They are gated by `last_*_update_game` columns so a restart cannot double-apply (see [CB-08 RULING]).
- `teams.games_played` is reset to 0 in `finalizeOffseason` (`offseason.ts:343`), so all per-team thresholds restart cleanly each season. Confirmed.

### [AB-02 RULING] — Minor-league stats: rating-based triggers + synthesized display stats (option b + lightweight c)

We do NOT build a minor-league game simulator (out of scope, unbudgeted). Instead:

1. **Call-up / send-down DECISION triggers become rating-based, not stat-based.** Reinterpret the spec triggers as follows (these supersede the spec's OPS/ERA wording):
   - Call-up "starter has OPS < .580 over last 15 AB and AAA replacement rated higher" → **"an MLB position-starter with the lowest recent OPS on the 25-man is rated >= 5 points BELOW the best AAA player at the same position."** Recent MLB OPS is real (MLB games are simmed) — track it per [G9 below]. The AAA side is rating-only.
   - Call-up "SP ERA > 7.00 over last 3 starts and AAA arm available" → **"an MLB SP with recent ERA > 7.00 (real, from season_stats over the last 3 starts) and a AAA SP rated >= that SP."**
   - "Prospect in AAA rating >= MLB starter at same position + 5" → pure rating comparison (already rating-only).
   - Send-down "OPS < .560 over 20+ AB with better AAA option" → MLB OPS is real; AAA side is rating-only.
2. **Minor leaguers get synthesized display stats** so `/api/teams/:id/minors` can show "live stats" (test G6). Implement a per-10-game **minor-league stat synthesizer** (`prospectDev.ts`, runs in the same `% 10` batch): for each minor leaguer, add a small deterministic stat increment derived from `seedFor('minor_stat_<playerId>_<currentGameNumber>', seed)` and the player's ratings (e.g., batters: AB += randInt(8,18), hits scaled to a BA centered on `contact/300`, HR scaled to `power`; pitchers: IP += randInt(4,9), ER scaled to inverse of `pitching_control`). Write to the SAME `season_stats` row (so `/api/teams/:id/minors` reads it). This is display-only flavor — it MUST NOT drive call-up/send-down decisions (those are rating-based per #1). Mark synthesized rows so they are never confused with MLB box-score stats: synthesized minor stats accumulate in `season_stats` exactly like MLB stats; the distinction is purely that the player has `minor_level IS NOT NULL` and `is_on_25man = 0`.
3. **Hot streaks** (spec "5+ game hit streak in minors") are computed from the synthesized stat deltas: if a minor leaguer's synthesized batch produced hits in 5+ consecutive synthesized "games," set `prospect_visible = 1` for a +5 call-up scoring boost. This is the only place synthesized stats feed any decision, and only as a tiebreaker boost, never as a trigger.

Rationale: this satisfies G6 ("minors returns live stats") and keeps call-up/send-down deterministic and correct without a full minor-league sim. The OPS/ERA clauses in the spec are honored for the MLB side (real) and reduced to ratings on the AAA side (no real data exists).

### [AB-03 RULING] — Waiver expiry is range-based and fires on a hook that survives skipped/turbo ticks

1. **Window unit:** per-team `games_played` of the DFA'ing team, per [AB-01 RULING]. At DFA: `claim_game_window_end = <dfa team>.games_played + 3`.
2. **Range check, NOT equality.** On every maintenance pass, for the teams that just played, AND additionally on a **league-wide waiver sweep** that runs every tick: expire/claim every entry where `<dfa team>.games_played >= claim_game_window_end`. Using `>=` (not `==`) guarantees an entry is never "missed" if its exact expiry game was skipped.
3. **Idempotent claim** (see [CB-07 RULING]): claim resolution uses `UPDATE players SET ... WHERE id=? AND waiver_state='waivers'`; if `changes()===0` the entry was already resolved — no-op. State advances to `none` (claimed) or to FA (`team_id=NULL, is_on_mlb_roster=0, waiver_state='none'`) immediately, so a re-sweep next tick does nothing.
4. **The maintenance hook MUST run even on skipped games.** See [AB-18 RULING] — the hook is moved OUT of `simulateGame` and called by the engine after every `runGameTick`, including the missing-SP skip path. Concretely: `runGameTick` always calls `runRosterMaintenance(...)` after the `simulateGame(...)` call returns, regardless of whether a game_log row was written.
5. "DFA appears within same sim tick" (test G4): when a DFA is triggered (inside `runRosterMaintenance`), the player's `waiver_state` is set to `'dfa'` in the SAME pass, so `/api/waivers` shows it immediately. Confirmed satisfiable because DFA and the waiver listing read the same `waiver_state` column.

### [AB-04 RULING] — DFA immediately vacates the 40-man slot (breaks the circular dependency)

This OVERRIDES architect-eval D4's "DFA keeps is_on_mlb_roster=1."

- **On DFA:** set `is_on_25man = 0`, `is_on_mlb_roster = 0`, `waiver_state = 'dfa'`, `dfa_team_id = <team>`, `claim_game_window_end = <team>.games_played + 3`. The player is NO LONGER counted against the 40-man. `team_id` is RETAINED during the waiver window so `/api/waivers` can show "DFA'd by X" and a claim/expiry can reference the origin team.
- **40-man membership = `is_on_mlb_roster = 1`.** A DFA'd player is in waiver-limbo: `team_id` set but `is_on_mlb_roster = 0` and `waiver_state IN ('dfa','waivers')`. This is the limbo state the Adversary asked for.
- **What triggers a DFA** (DFA is always a consequence, never a standalone decision):
  1. Send-down triggered on a player with `options_remaining = 0` → DFA instead (spec).
  2. Spring-cuts overflow: a player must be removed from the 40-man and is neither releasable (rule a) nor optionable (no options) → DFA. **Correction to spring-cuts contradiction, see [AB-14 RULING].**
  3. Call-up needs a 40-man slot and the team is at 40: pick a DFA candidate by archetype (analytics → lowest `overall_rating` proxy for WAR; old-school → youngest `is_on_mlb_roster=1` minor-optioned player with options), DFA it (slot frees immediately per above), THEN call up.
- **Because DFA frees the slot immediately, the call-up that triggered the DFA succeeds in the same pass.** No 3-game wait. The spec's "DFA to create space" now works.
- **40-man cap enforcement:** a new helper `count40Man(teamId) = SELECT COUNT(*) FROM players WHERE team_id=? AND is_on_mlb_roster=1`. Before any call-up promotes a player to `is_on_mlb_roster=1`, assert `count40Man < 40`; if at 40, DFA a candidate first.

### [AB-05 RULING] — Service-time constant rescaled for the 50-game world; contract expiry remains the primary FA driver

The spec's "172 games = 1 year, 6 years = FA" is unreachable (1,032 MLB games ≈ 20.6 seasons). Rescale:

- **[RULING]** `GAMES_PER_SERVICE_YEAR = 30` (a service "year" is 30 MLB games in this 50-game world). `FREE_AGENT_SERVICE_GAMES = 6 * 30 = 180`. A player on the 25-man for the bulk of ~4 seasons reaches FA eligibility — reachable, rare, flavorful.
- **Test G5 reconciliation:** Test G5 says "After 172 games service time: free agency eligibility flag set." We honor BOTH numbers without contradiction: keep a `service_time_days` accumulator (raw games on 25-man). Set `free_agent_eligible = 1` when `service_time_days >= 180`. Additionally, because the test asserts the 172 figure, expose the threshold as a named constant `SERVICE_YEAR_GAMES` and the unit test (`serviceTime.test.ts`) asserts the FLAG-SETTING behavior at the configured threshold, reading the constant — NOT a hardcoded 172. The test must read `FREE_AGENT_SERVICE_GAMES` from the module, set a player's `service_time_days` to that value, run the accrual, and assert the flag flips. (UI Tester B's G5 checklist item "after 172 games" is interpreted as "after the service-year threshold accrues to the FA boundary" — document this in the release notes so the tester reads the flag, not a literal 172.)
- **Manipulation band** (spec G12): analytics GM delays call-up when target has potential A/B and `service_time_days` within `[FREE_AGENT_SERVICE_GAMES - 23, FREE_AGENT_SERVICE_GAMES)` i.e. `[157, 180)` — a ~23-game band wide enough that the 10-game batch accrual (which can jump 10) still lands inside it at least once. Delay by `uniform(10,20)` games via `manipulation_delay_until_game`.
- **Contract expiry stays the real FA driver** (v0.1.0 `offseason.ts:142` unchanged). Service-time FA is the rare bonus path. This is intentional and correct.

### [AB-09 RULING] — Firing evaluation order is fixed; interim is checked in ALL branches; one firing per team per tick

Within a single team's maintenance pass, firing evaluation runs in this exact order and STOPS after the first firing for that team this tick:
1. **Interim guard (ALL branches):** if `interim_gm = 1` OR `interim_manager = 1`, skip ALL firing evaluation for this team. (Covers GM-fires, meddling-owner-fires, non-meddling-owner-fires, offseason. No branch may fire an interim.)
2. **Owner-precedence:** evaluate owner actions BEFORE GM actions.
   - If owner is `meddling`: evaluate meddling-owner-fires-manager (lower threshold). If it fires, set a `firedThisTick` flag and STOP.
   - If owner is non-meddling and record hits the `2.5×` catastrophic threshold: owner fires manager directly, GM `job_security -= 2`, set `firedThisTick`, STOP.
3. **GM-fires-manager** (only if `firedThisTick` is false AND owner is NOT meddling): if manager threshold met, GM fires manager. STOP.
4. **Owner-fires-GM** (10-game cadence): evaluated separately; if the team already had a manager firing this tick, the GM firing still MAY occur (different role) BUT the just-installed interim manager is exempt from any re-eval this tick (rule 1 already handles future ticks). To avoid the "embarrassment cascade" the Adversary flagged: the `job_security -= 2` from rule 2 does NOT trigger a same-tick GM firing — GM firing reads `games_under_500`, not job_security, so there is no cascade. Document this.
- Result: at most one manager firing and at most one GM firing per team per tick; interims never fired mid-season; owner always wins manager-firing ties.

### [AB-10 RULING] — In-season prospect dev replaces the offseason dev for minor leaguers; single age source; bust once per season

### [AB-18 RULING] — Maintenance runs as a SIBLING transaction after simulateGame; hook runs on every tick including skips

We do NOT refactor `simulateGame` to accept an outer transaction (too invasive). Instead:
- `runRosterMaintenance(leagueId, homeTeamId, awayTeamId, gameNumber)` opens its OWN `db.transaction()` and runs all maintenance writes atomically.
- It is called by `runGameTick` AFTER `simulateGame(...)` returns, **unconditionally** (including when the game was skipped for a missing SP). Pass the home/away team ids from `nextGame`.
- **Crash-gap acceptance:** if the process crashes between the game commit and the maintenance commit, the game is recorded but that tick's maintenance is lost. This is acceptable because: (a) all maintenance is cadence/range-based and idempotent — the NEXT tick's maintenance re-evaluates the same conditions (waiver range-check via `>=`, firing via `games_under_500`, call-up via roster count); (b) persistence test G10 tests waiver-window/interim/service-time which are all stored in DB columns and survive restart; the only loss is one tick of re-evaluation which self-heals. Document this in code comments.
- The skipped-game path (`game.ts:253-258`) is unchanged; maintenance still runs because the hook is in `runGameTick`, not `simulateGame`.

### [AB-19 RULING] — Maintenance cost is bounded; benchmark gate relaxed to <15s/season

- Per-tick maintenance touches only the 2 teams that played (call-up/send-down/firing) plus a league-wide waiver sweep (cheap, indexed) plus the `% 10` batches. This is far cheaper than evaluating all 20 teams per tick.
- The end-of-tick roster invariant check (Risk R7) runs as ONE indexed query per tick.
- **Benchmark gate:** turbo must sim one full season (500 games + maintenance + news inserts) in **< 15 seconds** (relaxed from eval's 12s to account for the new load). Measured in Phase 14.

### [AB-20 RULING] — Service-time accrual is the 10-game batch; test G5 reads the flag, not literal per-game

Use the ACTUAL games elapsed (`current_game_number - last_service_time_update_game`) capped at a sane max, not a hardcoded +10, so partial windows are correct.
- Test G5 "increments by 1 per game" is interpreted as "monotonically increases while on the 25-man" and verified by the FA-flag-flip test ([AB-05 RULING]), not a literal per-game assertion. Roster-status mid-window changes are acceptable approximation per Risk R5.

## §3 — Waiver Wire System

**File:** new `server/sim/waivers.ts`; route `server/routes/waivers.ts`; integrated via `rosterMaintenance` (§4) and a league-wide sweep.

### State machine (player.waiver_state)
```
none → dfa     : on DFA. Set is_on_25man=0, is_on_mlb_roster=0, waiver_state='dfa',
                 dfa_team_id=<team>, claim_game_window_end=<team>.games_played + 3,
                 team_id RETAINED. (AB-04: slot freed immediately.)
dfa  → waivers : conceptually the same listed state; we keep 'dfa' as the on-waivers value.
                 (Treat 'dfa' and 'waivers' as both "on the wire". /api/waivers shows
                 waiver_state IN ('dfa','waivers').)
on-wire → none + claimed : at expiry, a claiming team wins → team_id=claimer,
                 is_on_mlb_roster=1, is_on_25man=0 (joins 40-man, optioned), waiver_state='none'.
on-wire → FA   : at expiry, unclaimed → team_id=NULL, is_on_mlb_roster=0, is_on_25man=0,
                 waiver_state='none', minor_level=NULL. (Not 'expired' as a resting state — we
                 collapse straight to 'none' off-roster to keep the domain simple; 'expired' is
                 a transient label only.)
```

### DFA triggers (always a consequence — never standalone)
1. Send-down on a player with `options_remaining = 0` → DFA. (spec, §4)
2. Call-up needs a 40-man slot and `count40Man(team) >= 40` → choose DFA candidate by archetype, DFA, then promote. (§4, [AB-04 RULING])
3. NOT spring cuts (spring releases go straight to FA per [AB-14 RULING]).

### Window tracking
- Column: `players.claim_game_window_end` (per DFA team's `games_played + 3`). Survives restart (G10/G28).
- `/api/waivers` computes `claim_window_games_remaining = MAX(0, claim_game_window_end - <dfa_team>.games_played)`.

### Expiry / claim resolution (range-based, idempotent — CB-07, AB-03)
A function `processWaivers(leagueId)` runs every tick (league-wide sweep, cheap, indexed):
```
for each player WHERE league_id=? AND waiver_state IN ('dfa','waivers'):
  dfaTeamGamesPlayed = SELECT games_played FROM teams WHERE id = dfa_team_id
  if dfaTeamGamesPlayed >= claim_game_window_end:   // >= not == (survives skips/turbo)
     evaluate claims (claim order = reverse standings, G5 tie-break)
     in ONE db.transaction():
        UPDATE players SET ... WHERE id=? AND waiver_state IN ('dfa','waivers')  // idempotent guard
        if changes()===0: skip (already resolved)
        if claimed: transfer to claimer (40-man), log transaction 'waiver_claim', enqueue news (badge TRANSACTION)
        else: to FA pool, log transaction 'waiver_expired' (news only if overall>=65 per spec edge case)
```

### Claim resolution (reverse standings)
- Eligible claimers: all teams EXCEPT the DFA team, ordered by `wins ASC, (runs_scored - runs_allowed) ASC, team_id ASC` (worst team first — matches standings sort inverted). Tie-break per eval G5.
- For each team in order, compute `claim_score` from the archetype table (§6, ARCHETYPES). First team with `claim_score >= 0.6` wins.
- Claimed player joins claimer's 40-man (`is_on_mlb_roster=1, is_on_25man=0, minor_level='AAA'`), retains contract.

### Edge cases
- DFA appears same tick (test G4): the DFA write sets `waiver_state='dfa'` in the maintenance pass, immediately visible on `/api/waivers`.
- Unclaimed after window: to FA silently unless `overall_rating >= 65` (then a TRANSACTION news item). (spec edge case)
- Restart mid-window (G10/G28): no in-memory state; the range-check resumes correctly from DB columns.
- Roster invariant: after any claim, the end-of-tick invariant (§7/R7) confirms each team still has exactly 25 on `is_on_25man=1` (a claim adds to 40-man not 25-man, so 25-man is unaffected — confirm).

   b. **Call-up eval** (same 5-game gate; reuse the same `last_call_up_check_game`, set it after both).
   c. **Firing eval** ([AB-09 RULING] ordered) if `games_played - last_firing_check_game >= 5` (manager) / `last_gm_firing_check_game >= 10` (GM).
   d. **Trade posture / deadline** (§6) when `games_played` in window.
4. **Prospect dev tick** (league-wide, only when `gameNumber % 10 === 0` — §7).
5. **Roster invariant check** (R7) — log + autoBalance if any team != 25 on 25-man.

### Counter to evaluate against
Per [AB-01 RULING]: call-up/send-down/firing use PER-TEAM `games_played`. Cadence gates compare `games_played` to the per-team `last_*_check_game` columns and update them after evaluating.

### Call-up logic (callup.ts)
Triggers (rating-based per [AB-02 RULING]):
- MLB 25-man `is_on_25man=1` count < 23 (injuries) → call up best available at the most-needed position.
- An MLB position-starter's recent OPS (from `season_stats.recent_*`) is >= 5 points of rating below the best AAA player at that position → call up.
- An MLB SP recent ERA (from `recent_er/recent_ip` over `recent_starts<=3`) > 7.00 AND a AAA SP rated >= it → call up.
- Best AAA prospect at a position rated >= MLB starter at that position + 5 → call up.

Source order: AAA first (`minor_level='AAA'`), then AA, then A, then waiver wire, then FA. If none: `console.warn('[callup] No eligible minor leaguers for team N, position P')`, leave roster short (do not crash — test G11).
- Promotion: set `is_on_25man=1`, `is_on_mlb_roster=1` (ensure `count40Man < 40` first, else DFA a candidate — §3 trigger 2), `minor_level=NULL`. Set `first_mlb_call_up_game = current_game_number` if NULL (service clock start). Log `transaction_type='call_up'`, enqueue ROSTER news.
- Service-time manipulation ([AB-05 RULING]): if calling-up team's `gm_archetype='analytics'` AND target `potential IN ('A','B')` AND `service_time_days IN [157,180)` → set `manipulation_delay_until_game = current_game_number + uniform(10,20)`, skip this call-up. Old-school/balanced never manipulate.

### Send-down logic (sendDown.ts)
Triggers (MLB side real, AAA side rating-only):
- Player recent OPS < .560 over recent_ab >= 20 AND a AAA replacement rated higher.
- SP recent ERA > 6.50 over recent_starts >= 4 AND AAA replacement available.
Procedure: if `options_remaining > 0` → send down (`is_on_25man=0`, `is_on_mlb_roster=1`, `minor_level='AAA'`, `options_remaining -= 1`), log `transaction_type='send_down'`, ROSTER news. If `options_remaining = 0` → DFA instead (§3), generate ROSTER news (test G5/G4).

### Service time (within serviceTime portion, batched)
- When `gameNumber % 10 === 0` and `current_game_number > last_service_time_update_game`:
  `service_time_days += (current_game_number - last_service_time_update_game)` for `is_on_25man=1` players on each team (additive-only, CB-08), then set `last_service_time_update_game = current_game_number`.
- `UPDATE players SET free_agent_eligible=1 WHERE league_id=? AND service_time_days >= 180 AND free_agent_eligible=0`.

### Edge cases (all from spec / test G11)
- No AAA available → AA → A → waivers → FA → leave short (warn, no crash).
- No options on send-down → DFA.
- 40-man full on call-up → DFA candidate first.

## §5 — Firing Logic

**File:** new `server/sim/firings.ts`; integrated in `rosterMaintenance` step 3c. Offseason firing path in `offseason.ts:runFrontOfficeStep` is RETAINED unchanged (handles end-of-season cleanup).

### Integration / cadence
Per [AB-01 RULING]: manager eval every 5 per-team games (`games_played - last_firing_check_game >= 5`); GM eval every 10 per-team games (`>= last_gm_firing_check_game + 10`). Precondition: `games_played >= 12` ([C13/G14 RULING]). Update the `last_*` columns after evaluating.

### Threshold formula (C13)
```typescript
const baseThreshold = 8;
const patienceMod = { meddling: 0.6, 'win-now': 0.8, moderate: 1.0, patient: 1.5, 'hands-off': 2.0 }[owner_personality] ?? 1.0;
const aggMod = { aggressive: 0.75, moderate: 1.0, conservative: 1.2 }[gm_risk_tolerance] ?? 1.0;
const managerThreshold = Math.round(baseThreshold * patienceMod * aggMod);
const gmThreshold       = Math.round(baseThreshold * patienceMod);
const ownerDirectMgrThreshold = Math.round(baseThreshold * patienceMod * 2.5);
const gamesUnder500 = team.losses - team.wins;  // fire when >= threshold
```

### Firing chain (ordered per [AB-09 RULING] — STOP after first manager firing for the team this tick)
1. **Interim guard:** if `interim_gm=1 || interim_manager=1` → return (no firing).
2. **Owner precedence:**
   - meddling owner: if `gamesUnder500 >= managerThreshold_meddling` (use meddling patienceMod) → fire manager directly (actor='owner', event_type='manager_fired_by_owner'), install interim manager, STOP manager-firing.
   - non-meddling owner: if `gamesUnder500 >= ownerDirectMgrThreshold` → owner fires manager directly (actor='owner', event_type='manager_fired_by_owner', "owner breaks glass" news), GM `job_security -= 2`, install interim manager, STOP.
3. **GM fires manager** (non-meddling owner only, if no manager firing yet): if `gamesUnder500 >= managerThreshold` → fire (actor='gm', event_type='manager_fired'), install interim manager.
4. **Owner fires GM** (10-game cadence, independent): if `gamesUnder500 >= gmThreshold` → fire GM (actor='owner', event_type='gm_fired'), install interim GM. (No cascade from step 2's job_security delta — GM firing reads gamesUnder500, not job_security.)

### Interim replacements
- **Interim manager** (bench coach promoted): `interim_manager=1`, `manager_style` UNCHANGED, ratings each = `MAX(25, original - 10)` (manager_tactics/motivation/communication), `job_security=5`. New `manager_name` from name pools.
- **Interim GM** (senior scout): `interim_gm=1`, `gm_philosophy` set to `'balanced'`/`gm_risk_tolerance='conservative'`/derive `gm_archetype='balanced'`, tenure semantics = fresh. New `gm_name` from pools.
- Interims persist to end of season; permanent hire happens in offseason `runFrontOfficeStep` (which must CLEAR `interim_gm`/`interim_manager` when it hires permanents). Interims cannot be fired mid-season (rule 1).

### News + events
Every firing writes a `front_office_events` row (with new `actor` column) AND a `news_items` row (badge='FRONT OFFICE', `is_headline_pending=1`) in the same maintenance transaction.

### Edge cases (test G8)
- Meddling owner fires before patient owner: enforced by patienceMod (0.6 vs 1.5).
- Aggressive GM shorter leash: enforced by aggMod (0.75 vs 1.2).
- Owner+GM both want manager: owner wins, one event (rule 2 STOP).
- Interim cannot be fired: rule 1.
- Firing during turbo: maintenance runs every tick including turbo bursts (the hook is in runGameTick), so firings fire in turbo.

### Waiver claim score (uses ARCHETYPES + G7 formula)
```
base_interest(archetype, overall):
  analytics:  overall in [50,65] → 0.7 else 0.2
  old-school: overall >= 60 AND age >= 28 → 0.5 else 0.2
  balanced:   overall >= 55 → 0.4 else 0.2
position_need_bonus: +0.3 if claimer has 0-1 players at that position (is_on_25man)
market_size_modifier: analytics+(small|medium) +0.3; old-school+(large|mega) +0.1; else 0
contender_modifier: +0.2 if claimer games_back <= 5
random_jitter: seedFor('waiver_claim_'+leagueId+'_'+currentGameNumber, seed) uniform(-0.05,0.05)
claim if total >= 0.6; multiply by waiver_claim_probability_modifier from ARCHETYPES.
```

### Trade posture / deadline (per-team games_played per [AB-01 RULING])
- When a team reaches `games_played >= 30` (first maintenance pass), set `trade_posture`:
  `games_back <= 5 → BUYER; games_back >= 10 → SELLER; else NEUTRAL`. (games_back computed from division/league leader wins.)
- Trade execution: in the maintenance pass, for a team whose `games_played` is in `[30, 37]` (offset earlier for analytics: a team's effective start is `30 - veteran_trade_offset_games_before_deadline`), attempt trades. Skip `interim_gm=1` teams entirely (test G7).
- Matching (D9): list non-interim BUYERs (games_back ASC); for each, find non-interim SELLERs with positional surplus; package = 1 veteran (age>=28, contract_years<=2) for 1-2 AAA/AA prospects; archetype evaluates fairness (analytics demands more prospects, old-school more proven). Execute via `transaction_type='trade'`, enqueue TRANSACTION news. Per-team cap 2, league cap 12. After 3 failed match attempts per buyer, break.
- **Forced minimum 3 trades** ([AB-12 RULING]): after the league-wide deadline marker fires (`shouldFireTradeDeadline` true), if `< 3` trades executed this season, force trades (worst non-interim seller × best non-interim buyer) until 3 exist or no pairs remain.

### Small/large market behaviors
- Small/medium: veteran trade bait if age>=30, contract_years<=2, and 8+ games back by `games_played>=30`. Claim waivers rated 50-65. Non-tender if arb salary projection > $8M. Budget hard cap on FA bids.
- Large/mega: buy at deadline (acquire veterans, trade prospects). Pursue top FAs. Claim waivers for depth only.

### Spring training cuts (springCuts.ts) — see [AB-08/AB-14 RULING]
Implemented per §0; releases go straight to FA, `transaction_type='release'`, TRANSACTION news (batch-filled to avoid flood — eval G21). Runs as first regular-season event, atomic with `spring_cuts_done_season`.

### Non-tender (offseason, new step before free_agency in `offseason.ts`)
- After retirement, before contract-expiry free agency: for analytics/small-market GMs, non-tender players with arb-year >= 3 (approximate via `service_time_days >= 3*30`) and projected salary > archetype's `nontender_salary_threshold`. Non-tendered → FA pool (NOT waivers — test G7). `transaction_type='non_tender'`, TRANSACTION news.
- **Force at least one** (eval G23): if zero natural candidates, force the highest-salary player with `service_time_days >= 4*30 AND age >= 30` on a small/medium team.
