## §1 — Critical Fixes (do these first; both §1.1 and §1.2 required)

### §1.1 — [FIX AB-10] Stock the farm with real AAA/AA depth, and add a level-promotion path, so the live roster chain has something to act on

**Root cause (verified):** Two compounding gaps.

**(a) `assignRosterLevels` assigns minor levels by absolute RATING, so AAA stays empty.** `draft.ts:386-401`: players ranked 26-40 on each team (the leftovers after the top 25 go to MLB) are gated into AAA only at `overall_rating >= 60`, AA at `>= 50`. But by construction these are a team's 26th-40th best players — almost all rated below 60 — so they cascade to `A`. Result (Adversary probes): 3–7 AAA players league-wide across 20 teams, AA=0, ~95% at `A`. The send-down/call-up triggers (`sendDown.ts:36-44`, `callup.ts:201-260`) all require a same-position AAA upgrade, so they never fire and no 25-man slot ever opens.

**(b) No A→AA→AAA promotion path.** `prospectDev.ts:18,43` develops *ratings* for `minor_level IN ('AA','A','Rookie')` but never changes `minor_level`. AAA never fills over seasons either.

**Required fix (a) — assign minor levels by ROSTER RANK, not absolute rating.** In `assignRosterLevels` (`draft.ts:381-403`), the players are already `ORDER BY overall_rating DESC` (`:378`). Replace the rating-threshold branches at `:386-401` with rank-based assignment so every team gets a populated AAA tier:

```ts
for (let i = 0; i < drafted.length; i++) {
  const player = drafted[i]!;
  if (i < 25) {
    // MLB 25-man active
    prepared('UPDATE players SET is_on_mlb_roster = 1, is_on_25man = 1, minor_level = NULL WHERE id = ?').run(player.id);
  } else if (i < 40) {
    // 40-man, optioned to AAA/AA by RANK (guarantees ~7 AAA, ~8 AA per team)
    const level = i < 32 ? 'AAA' : 'AA';
    prepared('UPDATE players SET is_on_mlb_roster = 1, is_on_25man = 0, minor_level = ? WHERE id = ?').run(level, player.id);
  } else {
    // Pure minor leaguers (not on 40-man): AA/A/Rookie by rank-from-40
    const depth = i - 40;
    const level = depth < 10 ? 'AA' : depth < 30 ? 'A' : 'Rookie';
    prepared('UPDATE players SET is_on_mlb_roster = 0, is_on_25man = 0, minor_level = ? WHERE id = ?').run(level, player.id);
  }
}
```

This guarantees each team has ~7 AAA players (ranks 26-32) and a populated AA tier, regardless of absolute ratings. The AAA players are a team's 26th-32nd best — by construction some will out-rate a slumping 25-man player at the same position, so the AAA-upgrade send-down/call-up triggers become reachable. (Tune the 32/`depth<10`/`depth<30` cutoffs if positional coverage needs it, but the key change is **rank-based, not rating-threshold**.)

**Required fix (b) — add a level-promotion path so the farm self-sustains across seasons.** In the offseason development step (`offseason.ts` runDevelopmentStep, ~`:98`) OR in `prospectDev.ts`, after rating ticks, promote prospects up a level when they outgrow their current one. Minimal rule:

- A prospect at `A` with `overall_rating >= 50` and age ≤ 25 → promote to `AA`.
- A prospect at `AA` with `overall_rating >= 58` and age ≤ 27 → promote to `AAA`.
- Do this once per offseason (not per 10-game tick) to avoid churn. Cap so a team's AAA does not exceed its 40-man space (excess stays AA).

This keeps AAA stocked over multi-season play even as the original expansion-draft AAA cohort graduates to MLB or ages out.

**Downstream (should now work without further change — verify):** With AAA populated, `sendDown.ts` Trigger 1/2 fires when a 25-man player slumps and a higher-rated same-position AAA player exists → opens a slot → `callup.ts` Trigger 1 (roster < 25, `:130-185`, which already searches AAA→AA→A→FA) refills it → call-up news + service-time accrual. A send-down on an option-less player routes to DFA → waivers → claim/expiry. `/api/waivers` and `waiverCount` become non-empty during churn.

**Why:** This is the single highest-leverage fix in the iteration. The minor-league system is the second of four headline features; it is 100% dormant in fresh play today purely because the farm has no AAA depth.

### §1.2 — [FIX AB-11] Make INJURY and MILESTONE events actually occur in normal play

The producers are correctly wired (`engine.ts:451-489`). The events simply can never fire. Fix the generation/threshold mismatch.

**(a) INJURY — close the dead band between the generator cap and the trigger floor.** `worldgen.ts:329` generates `injury_prone = randInt(rng, 3, 6)` (max 6); `game.ts:761` fires only at `injury_prone >= 7`. No founding player can ever qualify. Fix the generator so the tail reaches the trigger:

In `worldgen.ts:329`, widen the range to include ≥7:
```ts
// injury_prone: most players 3-6, a tail of injury-prone players 7-9
const injuryProne = randInt(rng, 3, 9);
```
(Alternatively lower the `game.ts:761` threshold to `>= 6`, but widening the generator is cleaner and keeps injuries appropriately rare — `randInt(3,9)` puts ~43% of players at ≥7, then the `rng() < 0.05` per-game gate keeps actual injuries sparse.) Pick the generator-widen approach unless you have a balance reason not to; note your choice.

Confirm draft prospects already vary (`draft.ts` prospect generation uses `randInt(rng, 1, 8)` for injury_prone — fine, leave it). The replacement-player hardcode `injury_prone=5` (`draft.ts:168,336`) is fine — those are emergency fills.

**(b) MILESTONE — make milestones reachable within the normal play horizon.** `game.ts:778-827` fires on 100/200 career HR, 2000 career hits, 1000 career K — all start at 0 and take many seasons to cross. Two acceptable approaches (do at least one; (i) is cheaper and sufficient):

  **(i) Seed age-scaled starting career stats at worldgen.** In `worldgen.ts` player generation, for players older than ~24, set nonzero career stats scaled by age and rating so veterans sit near thresholds and cross them in normal play. E.g., approximate `career_hr ≈ (age - 22) × (power/12)`, `career_hits ≈ (age - 22) × (contact × 4)`, `career_k ≈ (age - 22) × (pitching_control × 6)` for pitchers — clamp so none start *above* a threshold (you want them to *cross* it, not start past it). This makes the existing 100-HR/2000-hit/1000-K checks reachable.

  **(ii) OR add lower, in-season milestone tiers** in `checkCareerMilestones` (`game.ts:778-827`) that a player can reach in a single season — e.g., a 30-HR season, a 200-hit season, a 250-K season (tracked against `season_stats`, with the same `prev < threshold && new >= threshold` crossing guard already used at `:794`). These fire organically every season.

After (a)+(b): a fresh multi-season sim produces ≥1 INJURY and ≥1 MILESTONE news row, and `news-filter-injuries` / `news-filter-milestones` return non-empty feeds.

**Why:** Two of the 15 mandated news event types (spec lines 244-245) are permanently empty for any real user. The producer is built; only reachability is missing.

## §2 — High Fix

### §2.1 — [FIX AB-NULL] Stop creating phantom 25-man players; make retirement and free-agency clear `is_on_25man`; harden the invariant query

**Root cause (verified):** Two offseason writes null the team without clearing the 25-man flag, leaving players with `team_id IS NULL AND is_on_25man = 1` that accumulate across seasons (717 observed after 11 seasons):
- `offseason.ts:82` (retirement, age 40+): `UPDATE players SET team_id = NULL, is_on_mlb_roster = 0 WHERE id = ?`
- `offseason.ts:263` (free agency, expiring contracts): `UPDATE players SET team_id = NULL, is_on_mlb_roster = 0, minor_level = NULL WHERE id = ?`

Then `checkRosterInvariant` (`rosterMaintenance.ts:34-36`) does `GROUP BY team_id` with no `WHERE team_id IS NOT NULL`, so the NULL group is reported every tick; its trim loop (`:46-47`, `WHERE team_id = ?` with `team_id = NULL`) matches nothing (`= NULL` is never true in SQL), so the warning fires forever.

**Required fixes (all three):**

1. `offseason.ts:82` — add `is_on_25man = 0` (and `minor_level = NULL` for completeness):
   ```ts
   db.prepare('UPDATE players SET team_id = NULL, is_on_mlb_roster = 0, is_on_25man = 0, minor_level = NULL WHERE id = ?').run(player.id);
   ```
2. `offseason.ts:263` — add `is_on_25man = 0`:
   ```ts
   prepared('UPDATE players SET team_id = NULL, is_on_mlb_roster = 0, is_on_25man = 0, minor_level = NULL WHERE id = ?').run(fa.id);
   ```
3. `rosterMaintenance.ts:34-36` — scope the invariant query to real teams so a stray NULL group can never be reported or loop:
   ```ts
   const counts = prepared(
     `SELECT team_id, COUNT(*) as cnt FROM players
      WHERE league_id = ? AND is_on_25man = 1 AND team_id IS NOT NULL
      GROUP BY team_id`
   ).all(leagueId) as Array<{ team_id: number; cnt: number }>;
   ```
   Optionally add a one-time cleanup at league-load/migration that clears `is_on_25man` for any existing `team_id IS NULL` rows, so already-corrupted DBs self-heal. (Not required for fresh worlds, but cheap and helps the carried test DB.)

**Why:** Genuine state corruption — 717 retired/FA players flagged active, per-tick log spam, and any query trusting `is_on_25man` is poisoned. Trivial fix; must land before AB-10's roster churn relies on accurate 25-man counts.

## §3 — Medium Fixes

### §3.1 — [FIX AB-13] Don't ship active 25-man regulars to satisfy the trade floor

`findBuyerProspect` (`tradeDeadline.ts:265-294`) Tier 3 (40-man reserve) and Tier 4 (lowest-rated 25-man bench) can move active roster players as the "prospect package" when the farm is empty. Once §1.1 stocks AAA, the natural AAA/AA package path (Tier 1) will carry most trades and this rarely triggers — but guard it anyway:

- Reorder so Tier 1 (AAA) and Tier 2 (AA/A/Rookie) are exhausted across ALL buyer/seller pairings before any active-roster tier is considered.
- Restrict Tier 4 to bench players clearly surplus at their position (position count > `POSITION_MINIMUMS[pos]`), never a sole regular.
- Acceptable alternative: a prospect-less/cash-equivalent package (a synthetic "future considerations" trade row) rather than moving a 25-man regular, since the floor is a synthetic guarantee, not a realism requirement.

**Why:** User-visible realism defect in the News feed (contenders trading everyday regulars for depth). Largely mitigated by §1.1.

### §3.2 — [FIX/DEFER AB-12] `manager_resigned` — either add a trivial trigger OR document as deferred

`manager_resigned` is in `BADGE_MAP`/`proceduralHeadline` and the spec news-type table (line 241) but has no producer. The spec firing chain describes firings, not resignations. **You may choose either:**

- **(acceptable)** Document `manager_resigned` as deferred to v0.3.0 in the release note and reconcile the 15-type table (note it is an optional event with no v0.2.0 mechanic).

Either closes this. Do not leave it silently unaddressed.

### §3.3 — [FIX FP-4 consequence] Add a thin read-only `GET /api/front-office-events` so Groups 2 & 8 are verifiable

Add a league-wide read endpoint (mirrors `/api/teams/:id/history` but unscoped by team) so black-box testers can verify firings and owner events:

```ts
// e.g. in routes/news.ts or a small frontOffice router, mounted at /api/front-office-events
router.get('/', (req, res, next) => {
  try {
    const league = getActiveLeague();
    if (!league) { res.json([]); return; }
    const limit = Math.min(Number(req.query.limit) || 50, 200); // CB-4 clamp
    const rows = prepared(
      `SELECT id, season_number, team_id, event_type, departing_person, incoming_person, narrative, created_at
       FROM front_office_events WHERE league_id = ? ORDER BY created_at DESC LIMIT ?`
    ).all(league.id, limit);
    res.json(rows);
  } catch (err) { next(err); }
});
```

Use **explicit columns** (not `SELECT *`), clamp `limit ≤ 200`, scope by `league_id` (CB-4/CB-10). Wire it into the route registration alongside the other routers.

**Why:** Unblocks Test Group 2 (owner_died/owner_sold_team departing/incoming names) and Group 8 (firings with event_type/departing_person/incoming_person) for black-box verification. The data already exists; only the read surface is missing.
