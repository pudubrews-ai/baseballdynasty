## 3. The Binding Ruling on AB-10 (the prompt's explicit ask)

The prompt asks me to choose between Option A (add `is_injured` flag + IL/recovery), Option B (relax send-down threshold to "within 5"), or Option C (both). Below is the binding ruling, informed by the source facts above.

### RULING: **Option C — BOTH**, scoped to the minimum that drives `call_up`, in-season `send_down`, `dfa`, and `waiver_claim` to ≥1 organically.

**Rationale.** Neither A nor B alone reaches all four required transaction types organically:

- **Option A alone** (persist injuries as an IL stint that vacates the 25-man slot) reliably reaches **`call_up`**: the vacancy fires call-up Trigger 1 (`callup.ts:136`, which is NOT upgrade-gated and already searches AAA→AA→A→waivers→FA). But the *recovery* of the injured player (over-25 condition) is resolved by `checkRosterInvariant` auto-trim (`rosterMaintenance.ts:51-72`) — which sends the lowest-rated player to AAA or, only if option-less, nulls `team_id` directly. The invariant path does **not** route through `dfaPlayer` → the waiver state machine, so **`waiver_claim` would still be 0** and `dfa` would be the bare invariant null-out, not a true DFA transaction. Option A gets call_up + an injury-recovery send, but leaves the DFA→waiver→claim machine dormant.

- **Option B alone** (relax `sendDown.ts:44` from strictly-greater to "within 5 overall") makes the *stat-based* send-down path reachable: a struggling regular (recent OPS < .560 over ≥20 AB) can be optioned for an equal/lesser AAA replacement → `executeSendDown` → for option-less players, `dfaPlayer` → waivers → **`waiver_claim`/expiry**, and for option-ful players a real in-season **`send_down`** that opens a slot → call-up Trigger 1 fires **`call_up`**. This reaches send_down + dfa + waiver_claim + call_up. BUT it depends entirely on a regular slumping below .560 over a 20-AB window that the 5-game `recent_*` reset keeps narrow — reachability is plausible but not *robust* across all seeds, and it does not model the injury→IL→call-up loop the spec's Call-Up trigger #1 ("MLB roster drops below 23 active (injuries)") explicitly requires.

- **Option C** combines them so the chain is reachable by **two independent paths**: injuries open slots (robust, ~157-201 injuries/season already firing → frequent vacancies → frequent call_up), and the relaxed stat-based send-down routes option-less slumping regulars through DFA → waivers (reaching dfa + waiver_claim through the real state machine). This is the minimum that *guarantees* all four transaction types organically and is faithful to the spec's call-up trigger #1. It is also the option the Adversary recommended ("persist injuries as a temporary IL stint … and/or drop the higher-rated requirement").

This is NOT scope creep: both halves are small, bounded data/threshold changes against already-correct downstream code (the call-up/send-down evaluators, the waiver state machine, the news producers, and the engine wiring are all confirmed correct and must NOT be rewritten). The `is_injured` column already exists; no migration is needed.

### Exact file:line changes required (binding — detailed in developer-instructions-5 §1)

**Part A — Persist in-game injuries as an IL stint, and recover them.**

1. **`server/sim/game.ts:761-768`** — when the injury NotableEvent fires, attach a recovery timeline to the event payload (e.g., `recoveryGames`) so the engine can set a return game. ## 6. Decision

**ITERATE.**

The build is COMPLETE only when zero Critical/High/Medium confirmed findings remain. Confirmed this iteration: **1 Critical, 0 High, 2 Medium.** Iteration 4 is, again, real and well-targeted work: AB-11 is genuinely fixed (injuries are no longer mathematically impossible — `injury_prone` now spans 3-9 against the `≥7` trigger — and milestones are reachable from age-seeded career stats; the Adversary's organic probes confirm 157-201 INJURY and 63-90 MILESTONE events/season, and both news filters populate in fresh play). AB-NULL is fixed at both offseason write sites plus a scoped invariant query and a per-tick self-heal (organic probe: zero phantom rows). AAA *population* is now correct (20/20 teams, ~5 AAA each, a working A→AA→AAA offseason promotion path). The `injuryMilestoneNews` test is honestly rewritten, the `/api/front-office-events` harness route landed clean, and CISO finds zero new exploitable findings.

But the release still cannot ship for the same user-visible reason as Iter-1/2/3: **the live minor-league call-up/DFA/waiver chain — the second of the four headline v0.2.0 features — produces zero in-season roster moves in a fresh dynasty.** I root-caused the two compounding gaps at the line level. First, in-game injuries fire as news (and truncate a box score) but **never vacate a 25-man slot** — `game.ts:761`'s injury event flows only through `applyInjuryTruncation` and the `engine.ts:473` news insert; nothing writes `is_on_25man=0` or `is_injured=1`, so the unconditional restore-to-25 call-up (`callup.ts:136`) never has a hole to fill. Second, every stat-based send-down/call-up trigger requires a same-position AAA player rated *strictly higher* (or +5) than the incumbent (`sendDown.ts:44`, `callup.ts:209/244/270`), but rank-based assignment places AAA at ranks 26-32 — by construction below the 25-man (ranks 1-24) — so the upgrade condition is unsatisfiable on every seed and in every one of the 5 simulated seasons. The chain is gated at the vacancy-creation step exactly as in Iter-3; the developer fixed AAA population (the symptom) rather than reachability (the disease), as their own Deviation #1 concedes. The two tests that "cover" this chain pass only by manufacturing the AAA upgrade and low-OPS window the real sim never produces, or by asserting population instead of chain execution (AB-16) — the mechanism that has masked AB-10 for four iterations.

**Path to COMPLETE — this should be the final iteration.** The binding ruling is **Option C (both)**, scoped to the minimum: (A) persist in-game injuries as a temporary IL stint (`is_on_25man=0, is_injured=1` + a recovery timeline) in the engine's already-correct injury branch, which fires the upgrade-free call-up Trigger 1; and (B) relax the stat-based send-down threshold (`sendDown.ts:44`) from strictly-greater to within-5, so a slumping regular can be optioned for an equal/lesser AAA replacement, routing option-less players through the real DFA→waiver→claim state machine. The `is_injured` column already exists (`001_init.sql:89`); the only schema change permitted is one nullable `injury_return_game` column. Both halves are small data/threshold changes against downstream code that is confirmed correct and must NOT be rewritten (the evaluators, the waiver state machine, the news producers, the engine wiring). AB-13 self-resolves once the farm holds reachable assets; AB-16's organic integration test is the verification gate that must fail against `532942c` and pass after; AB-17 closes with a one-line in-season `job_security` decrement or a documented deferral.
