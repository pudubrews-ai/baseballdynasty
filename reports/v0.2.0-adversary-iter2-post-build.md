## CRITICAL FINDINGS

### AB-01 (Critical) — The live call-up/send-down/DFA/waiver chain is still inert; the "rolling window" fix zeroes `recent_*` immediately before the evaluator reads it, so stat-based send-down triggers can NEVER fire

**Root cause:** The developer's recency-window fix is a per-team reset of the `recent_*` columns **inside the same `callUpDue` block, immediately before** `evaluateSendDowns`/`evaluateCallUps` are called. Because the reset and the evaluation execute sequentially on the same tick, the evaluator always sees freshly-zeroed windows.

**File:line:** `server/sim/rosterMaintenance.ts:120-138`
```ts
const callUpDue = team.games_played - team.last_call_up_check_game >= 5;
if (callUpDue) {
  prepared(`UPDATE season_stats SET recent_ab=0, recent_hits=0, ...           // §2.7 RESET to 0
            WHERE league_id=? AND season_number=? AND player_id IN (...team...)`).run(...);
  evaluateSendDowns(team, leagueId, league.season_number, gameNumber);          // reads recent_ab — now 0
  evaluateCallUps(team, leagueId, league.season_number, gameNumber);
}
```
The send-down trigger requires `ss.recent_ab >= 20` (`sendDown.ts:27`) and `ss.recent_starts >= 4` (`sendDown.ts:57`); the call-up SP-ERA trigger requires `ss.recent_starts >= 3` (`callup.ts:228`). After the reset, all of these read **0**, so none can ever fire.

**Evidence (Adversary in-engine probes, full 500-game season through the real `runRosterMaintenance` loop):**
- A replica of the exact reset+eval ordering shows the evaluator sees `MAX(recent_ab)=0` across **all 32 due-tick evaluations** (`evalSeesMax_afterReset:0`).
- Full-season transaction counts (seed 7): `inSeasonSendDowns: 0, totalCallUps: 0, totalDfa: 0, totalWaiverClaims: 0`. The only 3 `send_down` rows in the DB are all from **spring cuts** (game 0), not in-season.
- The DFA → waiver → claim chain therefore never executes mid-season, so `/api/waivers` is empty in real play and the entire live minor-league system (the second headline v0.2.0 feature) is dormant — identical to the Architect's Iter-1 Critical (eval-1 lines 46-53), which is **not resolved**.

**Why the green suite missed it:** `callupSenddownReachability.test.ts` (§5.2) calls `evaluateSendDowns(team, …)` **directly** (lines 109, 223) after manually setting `recent_ab=25` in `season_stats` (lines 86-93). It never runs `runRosterMaintenance`, so the reset-before-eval ordering bug is never exercised. The test asserts "send-down fires when recent_ab≥20" — true in isolation, false in production because the loop zeroes recent_ab first. `newsIntegration.test.ts` runs the real loop but only asserts news rows exist; its non-game-news assertion is `>= 0` (always true, line 148), so it cannot catch zero roster churn.

**Impact:** Fails Test Group 4 (call-ups) and Group 5 (send-downs/DFA/waivers) wholesale at runtime. The waiver state machine, DFA logic, and call-up logic are all well-built but never reached. TRANSACTION-badge news (waiver claims), ROSTER-badge news (in-season call-ups/send-downs/DFA), and `/api/waivers` content are all empty in normal play.

**Resolution required:** Do not reset the window in the same tick you evaluate it. Either (a) maintain a true per-game sliding deque (architect G9), or (b) evaluate **first**, then reset for the next window, or (c) keep a separate "last-N-games" accumulator that is read before any reset. Add an integration test that sims a full season through `runRosterMaintenance` and asserts in-season `send_down`/`call_up`/`dfa`/`waiver_claim` counts are each ≥1.

### AB-02 (Critical, downgradeable to High) — Of the 15 spec event types, INJURY and MILESTONE have NO news producer; the ≥3-trades floor is not guaranteed (seed-dependent)

Two distinct runtime gaps share this entry because both defeat spec promises that the test suite "covers."

**(a) Missing event-type producers — INJURY & MILESTONE.**
- `insertMilestoneNewsItem` (`news.ts:202`) has **zero callers** anywhere in `server/` (verified by grep). Milestones are generated as `NotableEvent`s in `game.ts:772-826` and stored in `game_log.notable_events_json`, but never written to `news_items`.
- There is **no** `insertNewsItem`/`insert*NewsItem` call with `eventType:'injury'` anywhere. In-game injuries are generated in `game.ts:758-766` and stored in `notable_events_json`, never surfaced to the feed.
- `manager_resigned` is in `BADGE_MAP`/`NewsEventType` but has no producer either (no game path resigns a manager).
- **Adversary probe (full season, seed 7):** `news_items` badge distribution = `GAME:162, ROSTER:3, FRONT OFFICE:1`. **`INJURY:0, MILESTONE:0`.** So `news-filter-injuries` and `news-filter-milestones` (UI buttons present in `News.tsx`) always return an empty feed. Spec §"Event Types That Generate News Items" lines 244-245 ("Player injury | INJURY | Yes", "Player milestone | MILESTONE | Yes") are unmet at runtime.
- **Resolution:** In the game-result hook (`engine.ts` runGameTick, where `insertGameNewsItem` already runs), iterate the just-stored `notable_events_json` and call `insertMilestoneNewsItem` / `insertNewsItem({eventType:'injury'})` for each milestone/injury event.

**(b) ≥3-trades floor not guaranteed.**
- `forceMinimumTrades` (`tradeDeadline.ts:228-297`) requires the **buyer** to have an AAA/AA prospect (`:269-276`); if none, it `continue`s, and merely logs "Only forced X/needed" (`:295`).
- **Adversary probe through the real engine deadline path (`shouldFireTradeDeadline`→`fireTradeDeadline`→`forceMinimumTrades`, full 500-game season):**
  - **seed 11 → 0 trade rows, 0 trade news items.** At deadline-fire time only **5 AAA/AA prospects exist league-wide** (most expansion-draft minor leaguers sit at level `A`, and `findTradePackage`/`forceMinimumTrades` only consider AAA/AA), so buyers have nothing to trade.
  - seed 42 → 4 trade rows (= 2 actual trades). So the floor **passes on some seeds, fails on others.**
- The §5.5 `tradeFloor.test.ts` passes because it (i) uses seed 42, (ii) artificially sets divergent 35-10 / 10-35 standings, and (iii) asserts `COUNT(transaction_type='trade') >= 3` **rows**, where one trade with two prospects = 3 rows — so a single trade can satisfy it. It also bypasses `runGameTick`. Its comment "forceMinimumTrades guarantees the floor even when no natural trades occur" (line 90-92) is **false** per the seed-11 probe. Spec/Test "at least 3 trades complete across the league per season" is therefore unverified and not guaranteed.
- **Resolution:** Make `forceMinimumTrades` fall back to an `A`/Rookie or even prospect-less / cash-equivalent package when buyers lack AAA/AA prospects, so the floor is truly guaranteed; count distinct trades, not rows; and add an integration test that runs a full season through `runGameTick` on multiple seeds and asserts ≥3 distinct trades.

## HIGH FINDINGS

### AB-03 (High) — News producers are wired, but two of the 15 badges (INJURY/MILESTONE) and the TRANSACTION-badge roster events (waiver claims, in-season trades on dry seeds) remain empty at runtime

This is the runtime consequence of AB-01 + AB-02 combined, called out separately because it is the user-visible symptom a UI tester hits: the News tab and ticker show only GAME results plus a trickle of spring-cut ROSTER items and an occasional offseason FRONT OFFICE item. ### AB-04 (High) — Spring-cuts "exactly 25" has a documented escape hatch, and `checkRosterInvariant` is the only backstop

`runSpringCutsForTeam` (`springCuts.ts:100-240`) is now a correct 4-step pipeline (cut → repair minimums → trim back to 25 → fill to 25) and `repairPositionMinimums` is called (`:185`). This resolves PB-04 in the common case. **Residual:** Step 3's trim loop `break`s and *accepts an overcount* when every over-25 player is minimum-protected (`:205-208`). The only thing that then enforces 25 is `checkRosterInvariant` in `rosterMaintenance.ts:30-79`, which now both trims `>25` and promotes `<25` — but that runs during the *regular season*, not at the spring-cut gate, so the Architect's Phase-4 "exactly 25 immediately after cuts" gate can be transiently violated. Adversary probes never observed a stuck non-25 roster, so this is High-residual / latent, not blocking on its own. **Resolution:** in Step 3, if all candidates are minimum-protected, relax one non-critical minimum or cut the lowest-rated minimum holder with a backfill, so the invariant holds at the gate too.

## VERDICT: NOT READY

Iteration 2 fixes nearly every Iteration-1 finding — the News pipeline is real, the testids are fixed, the trade matcher reads the ARCHETYPES table, spring cuts hold at 25, roster flags are consistent, and owner-death personality + news surface correctly. But the release still cannot ship because **the live minor-league system — the second of the four headline v0.2.0 features — produces zero in-season roster moves at runtime** (AB-01). The developer's recency-window "fix" resets `recent_*` to zero in the same tick it evaluates them, so the OPS/ERA-based send-down and call-up triggers are *structurally* unreachable; an Adversary full-season probe recorded 0 send-downs, 0 call-ups, 0 DFAs, and 0 waiver claims, and the §5.2 reachability test passes only by calling the evaluator directly with pre-seeded stats, bypassing the buggy ordering. Compounding it, **INJURY and MILESTONE — 2 of the 15 mandated news event types — have no producer at all** (their events are computed and stored in `notable_events_json` but never written to `news_items`), and the **≥3-trades floor is not guaranteed** (0 trades for seed 11, because `forceMinimumTrades` requires AAA/AA prospects that mostly don't exist; the §5.5 test passes only on a friendly seed and counts rows, not trades). One low item the report claimed fixed (the `server/data/*.ts` gitignore) edited the wrong file and is still broken, so a clean checkout still can't boot.

**Required before READY:** (1) AB-01 — evaluate the recency window *before* resetting it (or implement a true sliding window), and add a full-season integration test through `runRosterMaintenance` that asserts in-season call-up/send-down/DFA/waiver counts ≥1 each; (2) AB-02 — wire `insertMilestoneNewsItem` and an injury news item off `notable_events_json` in the game-result hook, and make `forceMinimumTrades` fall back so ≥3 distinct trades are guaranteed regardless of seed, with a multi-seed engine-path test; (3) AB-05 — fix the root `.gitignore` and track `server/data/*.ts`. AB-04/AB-06/AB-07 should land for correctness/harness compliance; AB-08/AB-09 are low.
