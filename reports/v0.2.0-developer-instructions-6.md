## §1 — AB-19 (Critical): Production build cannot apply migrations; `start` script is broken

### Root cause (confirmed at the source level)

1. **Migration loader reads the wrong dist path.** `server/db.ts:43` resolves migrations via `join(__dirname, 'migrations')`. After `tsc -p tsconfig.server.json` (which has `include: ["server","shared"]` and `outDir: "dist/server"`), the compiled `db.js` lands at **`dist/server/server/db.js`** (tsc preserves the common-root, nesting under `server/`). So at runtime `__dirname = dist/server/server/` and the loader looks for **`dist/server/server/migrations/`** — which does not exist. `scripts/copy-migrations.mjs` copies the SQL files to **`dist/server/migrations/`** (one level up). Result: `initDb()` hits its `catch`, logs `[db] No migrations directory found`, and applies **zero** migrations. On any DB missing a migration (a fresh world, or a carried DB lacking migration 008) the server has no schema or crashes every tick with `no such column: injury_return_game`.

2. **`start` script targets a nonexistent file.** `package.json` has `"start": "node dist/server/index.js"`, but the compiled entry is at **`dist/server/server/index.js`**. `dist/server/index.js` does not exist.

### Required fix

**Pick ONE of the two equivalent approaches for the migration path (1A preferred — it's the smallest change and aligns the copy target with the loader). Then do 1B.**

#### 1A (preferred) — Make `copy-migrations.mjs` write to the path the runtime loader actually reads

Change `scripts/copy-migrations.mjs` so the destination matches `dist/server/server/migrations/` (next to the compiled `db.js`):

```js
// scripts/copy-migrations.mjs, line 13
const dst = join(projectRoot, 'dist', 'server', 'server', 'migrations');
```

This is a one-line change. The loader at `db.ts:43` (`join(__dirname, 'migrations')`) will now resolve correctly from `dist/server/server/db.js`. Do **not** change `db.ts`.

> Why not "fix `db.ts` to look one level up"? Because the dev path (`tsx server/index.ts`, where `__dirname` of `server/db.ts` is `server/` and `server/migrations/` is correct) must keep working. Adding a `../migrations` fallback in `db.ts` would break or complicate the dev path and the in-memory test path. Aligning the *copy target* with the existing loader keeps both dev (`server/migrations`) and prod (`dist/server/server/migrations`) correct with no source change.

#### 1B — Fix the `start` script

In `package.json`:

```json
"start": "node dist/server/server/index.js",
```

(Optional, recommended) — also fix the `copy:migrations` standalone script reference if it duplicates the destination string.

### Acceptance for §1

After `npm run build`:
- `dist/server/server/migrations/` exists and contains all **8** `.sql` files (001–008).
- Launch the built server on a **fresh** DB (delete or point `DB_PATH` at a new file): `npm start` (or `node dist/server/server/index.js`) must boot WITHOUT `[db] No migrations directory found`, must log `[db] Applying migration 00X...` for each of the 8 migrations, and `/api/state` must return 200.
- Launch the built server on a DB that has migrations 1–7 but not 8 (simulate the carried Iter-4 case): it must apply migration 008 (`injury_return_game`) on boot and sim must NOT crash with `no such column: injury_return_game`.
- `npm start` resolves to a file that exists.

## §2 — MB-1 (Medium): `buildPackaging.test.ts` validates the copy destination, not the runtime-read path

### Root cause

`server/tests/buildPackaging.test.ts:9-10` asserts SQL files exist at `dist/server/migrations/` — the *old* copy destination, NOT the path `db.ts`'s loader reads. The test was green while production migration loading was broken. This is the exact gap that let AB-19 ship undetected.

### Required fix

Update `buildPackaging.test.ts` so it asserts the property that actually matters — **the built server's loader path resolves to the migrations**:

1. Change `DIST_MIGRATIONS` to the runtime-read path:
   ```ts
   const DIST_MIGRATIONS = path.join(REPO_ROOT, 'baseball-dynasty', 'dist', 'server', 'server', 'migrations');
   ```
   (This must match the destination you chose in §1A. After §1A, the 8 SQL files live here.)

2. Add an assertion that the compiled `db.js` exists at `dist/server/server/db.js` and that a `migrations` directory exists as its sibling (i.e., `path.join(path.dirname(<db.js path>), 'migrations')` exists and has 8 `.sql` files). This ties the test to the *actual* runtime resolution rather than a hard-coded guess, so a future tsc-layout change can't silently re-break it.

3. Keep the existing "dist not built → skip" guard so the test stays CI-friendly when `dist/` is absent.

### Acceptance for §2

- `buildPackaging.test.ts` FAILS against the current built artifact (pre-§1 fix: SQL files are at `dist/server/migrations/`, not the loader path) and PASSES after §1A. Verify this by building once before your §1 fix, running the updated test (should fail), applying §1A, rebuilding, and re-running (should pass). Note this in your completion report.
- All other tests still pass (`npm test` → all green; you changed no sim logic).
