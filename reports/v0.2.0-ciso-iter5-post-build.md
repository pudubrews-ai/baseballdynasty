## Iteration-5 Verification Items — Audited

### Item 1 — Injury vacate in `writeGame` (+ engine defense-in-depth): parameterized, correctly scoped — ✔ CLEAN

**`game.ts:431-439`** — inside the `writeGame` D9 transaction, after the `game_log` insert:
```
for (const ev of notableEvents) {
  if (ev.type === 'injury' && ev.playerId) {
    db.prepare(`UPDATE players SET is_injured = 1, is_on_25man = 0, injury_return_game = ?
                WHERE id = ? AND is_on_25man = 1`).run(gameNumber + (ev.recoveryGames ?? 7), ev.playerId);
  }
}
```
- **Parameterized:** both written values (`injury_return_game` = `gameNumber + recoveryGames`, an integer; the `WHERE id`) are bound as `?` parameters. `is_injured=1`/`is_on_25man=0` are SQL literals, not interpolated values. **No string concatenation, no injection vector.** `ev.playerId` originates from `batter.playerId` → `player.id`, an integer PK read from the DB (`game.ts:773` `SELECT * FROM players WHERE id = ?`); `recoveryGames` is `3 + Math.floor(rng() * 13)` (`game.ts:780`) — a bounded seeded-PRNG integer in [3,15]. `gameNumber` is an internal integer counter. No external/HTTP input reaches this statement.
- **Scoped correctly:** the `WHERE id = ? AND is_on_25man = 1` guard ensures (a) exactly one player row is touched (PK match), and (b) the update is idempotent and only affects a currently-active 25-man player — it can never double-injure, never touch a minor-leaguer, and never touch a player on a different team (the player's own `team_id` is unchanged, so league/team scoping is preserved transitively). **Crucially `team_id` is NOT modified** — the injured player stays attached to their team with `is_on_25man=0`, so this can never recreate the AB-NULL null-group phantom condition. The roster drops to 24 transiently and call-up Trigger 1 (`callup.ts:136` `active25Man < 25`) restores it.
- **Defense-in-depth twin (`engine.ts:474-481`):** the `runGameTick` post-game news loop issues the identical parameterized `UPDATE` with the same `WHERE id = ? AND is_on_25man = 1` guard, bound `nextGame.gameNumber + (ev.recoveryGames ?? 7)` and `ev.playerId`. Redundant with the `game.ts` write (which already fires in the tick path via the same `notableEvents`), idempotent by the `is_on_25man=1` guard, no security delta.

### Item 2 — Injury recovery update (`rosterMaintenance.ts:192-198`): parameterized, league-scoped, `injury_return_game` not manipulable — ✔ CLEAN

```
UPDATE players
SET is_injured = 0, injury_return_game = NULL,
    minor_level = CASE WHEN is_on_mlb_roster = 1 THEN 'AAA' ELSE minor_level END
WHERE league_id = ? AND is_injured = 1 AND injury_return_game IS NOT NULL AND injury_return_game <= ?
```
- **Parameterized & league-scoped:** `league_id` and the comparison bound `gameNumber` are the only bound values, both integers, both bound as `?`. The `CASE` uses only the SQL literal `'AAA'` and a self-column reference (`minor_level`) — no interpolation. `is_injured`/`injury_return_game` set values are literals. **No injection vector.** The `WHERE league_id = ?` clause satisfies CB-10 — recovery never crosses league boundaries.
- **Could `injury_return_game` be manipulated?** No. It is set **only** by the two injury-vacate writes (Item 1), each to `gameNumber + recoveryGames` where `recoveryGames` is a seeded-PRNG integer in [3,15] and `gameNumber` is an internal counter — there is no HTTP route, no user input, and no other code path that writes `injury_return_game` (confirmed by `grep`: the only writers are `game.ts:435`, `engine.ts:479`, and the recovery NULL-clear here). It is a bounded server-derived integer compared against another server-derived integer; an attacker has no surface to set it to a malicious value or string. The recovery is also idempotent (`is_injured=1 AND injury_return_game IS NOT NULL` guard) and leaves `is_on_25man=0` so the invariant/call-up logic—not this statement—re-adds the player to the active roster, preserving the AB-NULL invariant.
