## Summary

Critical: 0 | High: 0 | Medium: 0 | Low: 2 (1 carried NB-2/CB-2 part-2, unchanged this iter, non-exploitable; 1 **new informational/defense-in-depth** — trade path does not clear `is_on_25man` at write-time, but is self-healed every tick). **0 new exploitable findings introduced by Iteration 4.**

Iteration 4 was a correctness iteration (AAA pipeline, injury/milestone reachability, phantom-25man cleanup, a new read-only route, offseason event coverage) that I audited from a security standpoint and find clean. The single new HTTP attack surface — `GET /api/front-office-events` — was built the security-correct way: it Zod-coerces+clamps `limit` to `[1,200]` with `.catch(50)` (no throw on garbage input), scopes the query by the active `league_id` re-derived server-side from `leagues WHERE archived = 0` (never a client-supplied tenant key), enumerates explicit columns (no `SELECT *`), binds both `league_id` and `limit` as parameters, and routes all errors through `next(err)` to the central `scrubError` handler that returns a fixed `{error:'internal_error'}` body. The rank-based roster-level assignment introduces no injection surface — level strings are JS-side literals bound as `?` parameters. The `injury_prone` range widen is a pure gameplay-balance constant change with no security dimension. All Iter-2/Iter-3 resolved items remain resolved.

The one item worth flagging for the Architect (informational, not a security vuln): the developer correctly added `is_on_25man = 0` clearing to **retirement** (`offseason.ts:84`) and **free-agency release** (`offseason.ts:321`), and the new **non-tender** writes (`offseason.ts:239,279`) also clear it — but the **trade** path (`tradeDeadline.ts:133,138`) issues a bare `UPDATE players SET team_id = ? WHERE id = ?` that does **not** adjust `is_on_25man` on either the moved veteran or the moved prospects. This is a *roster-count* drift (a traded 25-man veteran arrives on the buyer still flagged `is_on_25man=1`, pushing the buyer to 26; the seller drops to 24), **not** a phantom-row vuln like AB-NULL: `team_id` is never NULL on a traded player, so it cannot recreate the null-group infinite-loop risk that AB-NULL fixed. It is fully self-healed within the same `runGameTick` by `checkRosterInvariant` (`rosterMaintenance.ts:38-91`, runs every tick), which auto-trims the over-25 buyer and auto-promotes the under-25 seller back to exactly 25, and the new one-time `cleanupPhantom25man` (`rosterMaintenance.ts:30-34`) scrubs any NULL-team stale flags belt-and-suspenders. No persistent malformed state, no security impact — recommend completing flag hygiene in `executeTrade` as a cheap correctness follow-up, but it need not block ship.

## Iteration-4 Verification Items — Audited

### Item 1 — `GET /api/front-office-events` route: Zod-validated, league-scoped, error-scrubbed — ✔ CLEAN

`server/routes/frontOffice.ts` (registered `server/index.ts:200-201`):
- **Zod-validated.** `limit` parsed via `z.coerce.number().int().min(1).max(200).catch(50)` then `.parse(req.query['limit'] ?? 50)` (`frontOffice.ts:26-27`). `.catch(50)` means non-numeric / out-of-range input degrades to the default rather than throwing — no DoS via malformed query, and the clamp enforces CB-4's 200-row cap. There are no path/body params on this route, so `limit` is the only client-controlled input and it is fully validated.
- **League-scoped.** The active league is re-derived server-side: `SELECT id FROM leagues WHERE archived = 0 ORDER BY id DESC LIMIT 1` (`:16-18`); the data query is `… FROM front_office_events WHERE league_id = ? … LIMIT ?` with `(league.id, limit)` bound (`:30-36`). No client-supplied tenant identifier; satisfies CB-10. No active league → `res.json([])` (`:20-23`), no leak.
- **Explicit columns (CB-9).** `id, season_number, team_id, event_type, departing_person, incoming_person, narrative, created_at` — no `SELECT *`. (Note: `departing_person`/`incoming_person`/`narrative` are fixed-pool worldgen names, same fictional-data class already scanned secret-free in Iter-3.)
- **Error-scrubbed.** Handler is wrapped in `try/catch (err) { next(err); }` (`:39-41`), forwarding to the central error middleware (`index.ts:239-242`) which logs `scrubError(err)` and returns a fixed `{error:'internal_error'}` with status 500 — no stack trace, SQL, or secret reaches the client. Verified `scrubError` live-redaction in this run's test logs.

(Side note, not introduced this iter: the pre-existing `teams.ts:221-232` `GET /:id/history` route reads `front_office_events` with `SELECT *` scoped by `team_id` only. It is server-internal fictional data, carries no secrets, and is out of scope for CB-9's route-leak surface — same disposition as the Iter-3 internal-sim `SELECT *` carve-out. Flagging for awareness; not a regression.)

### Item 2 — Rank-based level assignment: no injection surface, correctly parameterized — ✔ CLEAN

`draft.ts assignRosterLevels` (`:372-401`): players are ordered by `overall_rating DESC` (parameterized `WHERE p.team_id = ?`), then the **level string is computed entirely in JS** — `'AAA'`/`'AA'`/`'A'`/`'Rookie'` selected by integer rank index (`i < 25`, `i < 40`, `depth < 10`, etc.), never derived from any external/DB string. Every write is `UPDATE players SET … minor_level = ? WHERE id = ?` with the level and integer `id` bound as parameters (`:388,392,397`). No string concatenation into SQL, no client input anywhere in this path (it runs server-internally after a draft). The companion `runProspectPromotionStep` (`offseason.ts:163-206`) is identical in discipline: literal `'A'`/`'AA'`/`'AAA'` comparisons in the WHERE, integer `id`/`aaaCapacity` bound as `?`. `selectTopN`'s SQL CASE (`draft.ts:108-118`) uses only literal position strings and the bound `?` overall bounds. **No injection vector.**
