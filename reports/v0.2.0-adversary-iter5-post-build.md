## AB-10 Verification — Point-by-Point (the prompt's mandate)

1. **Injury slot vacating — YES.** `game.ts:431-439` (inside the `writeGame` db.transaction): for each `notableEvents` entry with `ev.type === 'injury' && ev.playerId`, runs `UPDATE players SET is_injured = 1, is_on_25man = 0, injury_return_game = ? WHERE id = ? AND is_on_25man = 1` with `gameNumber + (ev.recoveryGames ?? 7)`. **Exact lines: `game.ts:433-437`.** Duplicated as defense-in-depth on the engine path at `engine.ts:477-481`.

2. **Recovery mechanism — YES.** `rosterMaintenance.ts:191-198`: `UPDATE players SET is_injured = 0, injury_return_game = NULL, minor_level = CASE WHEN is_on_mlb_roster = 1 THEN 'AAA' ELSE minor_level END WHERE league_id = ? AND is_injured = 1 AND injury_return_game IS NOT NULL AND injury_return_game <= ?` with `gameNumber`. Runs every tick, **after** the call-up/send-down eval, **before** `checkRosterInvariant`. Recovered players become AAA reserves (`is_on_25man` stays 0) and are reintegrated by the next call-up/invariant pass — exactly as the Architect specified.

3. **evaluateCallUps active-roster count — does NOT filter `is_injured`, and that is CORRECT per Architect ruling.** `callup.ts:132-134` counts `WHERE team_id = ? AND is_on_25man = 1` with no `is_injured` clause. The Architect's Eval-4 §3.1.6 explicitly ruled "Do NOT add an `is_injured=0` clause to that count — the `is_on_25man=0` write already removes the injured player from it." Verified: injured players have `is_on_25man=0`, so they drop out of the count and Trigger 1 fires. The developer correctly followed this (Deviation #3). **Not a defect.**

4. **send-down threshold — YES, within-5.** `sendDown.ts:46`: `if (!aaaReplacement || aaaReplacement.overall_rating < player.overall_rating - 5) continue;` (was `<= player.overall_rating` pre-fix; I diffed `532942c` and confirmed the old strictly-greater gate).

## Also-Probed Questions (#9-#11)

**9. New regression from injury vacating? Could an injured player be called up again before recovering? — NO.** Injured players have `is_on_25man=0`. `findMinorLeaguer`/Trigger-2/3/4 select from `minor_level IN ('AAA','AA','A')`, but a freshly-injured player still has `minor_level=NULL` (it is only set to 'AAA' on *recovery*, `rosterMaintenance.ts:195`), so a still-injured player is **not** an eligible call-up source until they recover. The `findMinorLeaguer` and Trigger queries filter `minor_level = ?`/`IS NOT NULL`, excluding NULL-level injured players. My regression counter (`is_injured=1 AND is_on_25man=1`, summed every game-tick) was **0** across all three full seasons — no injured player is ever simultaneously active.

**10. Recovery ever stuck (permanent `is_injured=1`)? — NO.** Recovery (`rosterMaintenance.ts:191-198`) fires unconditionally on `is_injured=1 AND injury_return_game <= gameNumber`, filtered by `league_id` only (not team/roster state), so it clears even players who were traded, DFA'd, claimed, or sent to FA mid-stint. My end-of-season counter (`is_injured=1 AND injury_return_game <= currentGame`) was **0** on all three seeds. The only residual `is_injured=1` rows at any snapshot are players whose return game is still in the future (correct).

**11. Within-5 send-down → same-tick send-down + call-up of the SAME player? — YES, but mild, bounded, non-looping (AB-18, Low).** I detected 3 of 19 distinct send-down players on seed 7 who received a `send_down` AND a `call_up` **at the same game_number** (players 502@285, 728@290, 777@99): `evaluateSendDowns` optioned a slumping regular to AAA (burning one option), then `evaluateCallUps` Trigger 1 — seeing the roster now at 24 — re-promoted that same player as the best available AAA at that position. The player ends where they started, having burned an option and generated a redundant send-down+call-up news pair. **It does NOT loop** (options decrement and floor at 0; the cadence gate is 5 games), and roster integrity holds (`over25Teams=0`). It is wasteful churn and slightly noisy in the News feed. Low.

## LOW FINDINGS

### AB-18 (Low) — Within-5 send-down can be undone by Trigger-1 call-up in the same maintenance tick, churning the same player (burns an option, emits redundant send-down+call-up news)
`rosterMaintenance.ts:144-145` runs `evaluateSendDowns` then `evaluateCallUps` in the same tick. When a slumping regular is optioned (`sendDown.ts:96-104`, `options_remaining-1`, `minor_level='AAA'`), the roster drops to 24, and call-up Trigger 1 (`callup.ts:136`, unconditional restore-to-25) re-promotes the best available AAA at the needed position — which can be the player just demoted (still `waiver_state='none'`, now `minor_level='AAA'`). Observed 3/19 send-down players on seed 7 with a same-game `send_down`+`call_up` pair. Bounded (options finite, 5-game cadence), no roster-integrity violation, self-terminating. **Resolution (cheap, optional):** in `evaluateCallUps` Trigger 1, exclude players whose most-recent transaction this tick was a `send_down` (or skip a player optioned within the last N games), or set a one-tick "just-optioned" guard. Non-blocking realism polish.

### AB-17 (Low) — The new integration test's in-season `send_down` assertion is satisfiable by spring cuts alone (no `game_number` on `transactions`); only the `call_up`/`dfa`/`waiver_claim` assertions truly gate AB-10
`iter5LiveOrgChain.test.ts:87-89` filters `sendDownsInSeason` with `r.game_number === undefined || r.game_number === null || r.game_number > 0`. The `transactions` table has no `game_number` column (`001_init.sql:157-167`), so `r.game_number` is always `undefined` and the filter counts every `send_down` row — including the spring-cut send-downs produced at `springCuts.ts:150/235/255`. Against `532942c` this assertion passed purely on spring cuts. The test still correctly gates AB-10 via the `call_up` and `dfa/waiver_claim` assertions (both 0 pre-fix), so it is fit for purpose, but the `send_down` line gives false reassurance. **Resolution:** join `transactions` to a game-number source (e.g., correlate `created_at`/news `game_number`, or add `game_number` to in-season `send_down` transaction inserts and assert `>0`), so the assertion measures *in-season* send-downs specifically. Documentation/test-quality only; does not affect runtime behavior.
