## 0. Headline: the AB-10 mechanism is fixed, but the production build is broken

This pass produced a sharp split that I root-caused at the source level. **The v0.2.0 live-org feature logic (AB-10) is genuinely and completely fixed** — confirmed by the Adversary's organic fresh-world probes and by a new integration test that I verified fails against the pre-fix commit. The developer earned the close on AB-10, AB-13, AB-16, AB-17, AB-14. CISO is clean.

**But the API Tester surfaced a previously-undetected, genuine Critical defect in the build/packaging layer:** the compiled production server cannot apply database migrations, and its documented launch command points at a file that does not exist. This is *not* AB-10 resurfacing — the sim logic is correct — but it means **the shippable artifact is unrunnable on any database that needs a migration**, including a fresh world and the restored Iter-4 DB. The entire test fleet's runtime evidence (Adversary, unit suite) ran the **TypeScript source in-process**, which masks this defect entirely, because in-process the migration directory resolves correctly. Only the two black-box testers who launched the **built server** could see it — and both did.

I cannot issue COMPLETE on a build that cannot apply its own migrations in production. **ITERATE**, scoped narrowly to two small build-config fixes.

## 1. Resolution of the Adversary-vs-API-Tester discrepancy (the prompt's central question)

The prompt asks me to rule whether the API Tester's 0-DFA / 7-call-up finding is (A) a false positive, or (B) a genuine production defect. **The answer is: BOTH — and they must be cleanly separated.**

### 1a. The live-org *mechanism* is genuinely fixed — the API Tester's low counts ARE a false positive (for the reasons the prompt anticipated)

The API Tester's 0-DFA / 7-call-up result is a false positive **as evidence that the chain is inert**, for two compounding, source-confirmed reasons:

1. **Restored Iter-4 DB blocks the DFA path by construction.** Migration `007_v0_2_0_schema.sql:87` declares `options_remaining INTEGER NOT NULL DEFAULT 3`. The Iter-5 worldgen fix that seeds `options_remaining` by service time (`worldgen.ts:359`, `serviceTime>=3 → 0`) **only runs at world creation** — it cannot retroactively touch players in a DB built under Iter-4. So every player in the restored Season-11 DB has `options_remaining=3`. No veteran has exhausted options → the option-less → `dfaPlayer` path (`sendDown.ts:128`) is unreachable on that DB. **This is exactly the prompt's root-cause point #2, confirmed at the schema level.** 0 DFAs on that DB is expected and not a defect.

2. **Post-deadline / mid-season test window.** The API Tester resumed at Season 11, Game 281 and simmed forward — past each season's call-up-pressure and deadline windows in the early-season cohorts, same artifact class as Eval-4 §4 Q3. Low call-up counts reflect the window, not the mechanism.

**The mechanism is independently confirmed fixed** by two sources I cross-checked:
- **Adversary organic fresh-world probes** (NO manual seeding): seed 7 → 137 call_up / 21 send_down / 8 dfa / 4 waiver_claim; seed 11 → 134/21/4/3; seed 42 → 148/29/8/5. The DFA→waiver→claim state machine fires (3-5 waiver_claim/season — *more* than the developer's own report claimed).
- **`iter5LiveOrgChain.test.ts`** — I confirmed (via the Adversary's reproduced run, and by reading the test) it asserts `call_up≥1 / send_down≥1 / (dfa∨waiver_claim)≥1` on seeds 7 AND 11 with zero manual `UPDATE players SET` seeding, and **FAILS 5/11 against pre-fix `532942c`** (the assertion the suite lacked for four iterations — AB-16 closed).

So the call_up=7 / dfa=0 black-box numbers are an artifact of the test substrate, not the feature. **Ruling (A) on the chain-inertness claim: false positive, root-caused, corroborated.**

### 1b. But the API Tester ALSO surfaced a genuine Critical build defect — ruling (B) on the migration failure

The prompt's context point #1 said the API Tester "may have started the wrong/stub server binary" via `node dist/server/index.js`. I investigated and **the truth is worse and is a genuine defect, not tester error:**

- `package.json` declares **`"start": "node dist/server/index.js"`** — but **that file does not exist.** The compiled entry is at `dist/server/server/index.js` (nested, because `tsconfig.server.json` has `include: ["server","shared"]`, so tsc preserves the common-root and nests server output under `dist/server/server/`). **The documented production launch command is broken.** The API Tester did not pick a "stub" — they ran the path the `start` script and prior iterations documented; it simply doesn't exist, so they fell back and hit the second defect.

- **Migrations are unreachable at runtime in the built server.** `server/db.ts:43` resolves migrations via `join(__dirname, 'migrations')`. The compiled `db.js` lives at `dist/server/server/db.js`, so `__dirname = dist/server/server/` → it looks for `dist/server/server/migrations/`, **which does not exist**. `scripts/copy-migrations.mjs` copies the SQL files to `dist/server/migrations/` (one level *up*). The loader hits its `catch` (`db.ts:47-49`), logs `[db] No migrations directory found`, and **returns without applying a single migration.** I verified on disk: `dist/server/server/migrations` → "No such file or directory"; the 8 SQL files sit at `dist/server/migrations/`.

**Consequence:** the production server applies **zero** migrations. It only works if the DB was already migrated by a *dev* run (`tsx server/index.ts`, where `__dirname` of `server/db.ts` is `server/` and `server/migrations/` exists). On a fresh DB, the built server would create no schema beyond `schema_versions`; on the restored Iter-4 DB (missing migration 008's `injury_return_game` column), the built server cannot add the column, so **every sim tick crashes with `no such column: injury_return_game`** — exactly what the API Tester observed at game 296, requiring a manual `ALTER TABLE` to unblock. This is a P0-class regression in the shippable artifact.

- **Why the whole fleet missed it:** `iter5LiveOrgChain.test.ts:19,122-123` sets `DB_PATH=:memory:` and calls `initDb()` **from the TypeScript source under Vitest** — `__dirname` of `server/db.ts` is `server/`, so `server/migrations/` resolves and all 8 migrations apply in-memory. The Adversary's probes likewise run in-process against TS source. None of them touch the built artifact. UI Testers A/B launched the *correct* nested path (`dist/server/server/index.js`) but on an **already-migrated** DB, so the silent no-op was invisible to them (UI Tester A even noted `[db] No migrations directory found` and dismissed it as "informational" — it is in fact the symptom of this defect).

- **`buildPackaging.test.ts` masks it:** the test asserts SQL files exist at `dist/server/migrations/` (the copy *destination*) rather than the path the runtime loader actually reads (`dist/server/server/migrations/`). It is green while production is broken — a test-quality gap.

**Ruling (B): genuine production defect, Critical.** A build whose server cannot apply its own migrations, and whose `start` script targets a non-existent file, is not shippable. This is a NEW finding this iteration (the dist-path defect existed earlier but only became *fatal* in Iter-5 because Iter-5 introduced the first migration — 008 — that a carried/restored DB would be missing; prior iterations happened to be tested on DBs that already had migrations 1-7 from a dev run).

## 2. Finding Assessment

### CRITICAL

#### AB-19 (NEW) — Production build cannot apply migrations; `start` script targets a non-existent file — **CONFIRMED (Critical)**

Two compounding build-config defects in the shippable artifact:

1. **Migration directory unreachable at runtime.** `db.ts:43` `join(__dirname, 'migrations')` from `dist/server/server/db.js` resolves to `dist/server/server/migrations/` (nonexistent). `copy-migrations.mjs:13` copies to `dist/server/migrations/`. Result: the production server applies **zero** migrations (`[db] No migrations directory found`, silent return). On any DB missing a migration — a fresh world, or a carried DB lacking migration 008 — the server either has no schema or crashes every tick (`no such column: injury_return_game`).
2. **Broken `start` script.** `package.json` `"start": "node dist/server/index.js"` — file does not exist; the real entry is `dist/server/server/index.js`.

**Evidence:** on-disk `dist/` inspection (db.js at `dist/server/server/`, migrations at `dist/server/migrations/`, `dist/server/index.js` absent); API Tester crash at game 296 + manual `ALTER TABLE` workaround; `db.ts` source; `copy-migrations.mjs` source; `tsconfig.server.json` include set; UI Tester A's `[db] No migrations directory found` log line on the correct nested entry path. Both black-box testers that ran the built server hit the symptom; both in-process test substrates (unit suite, Adversary) cannot see it.

**Severity ruling: Critical.** The shippable production artifact is unrunnable on a fresh or migration-pending database. This is independent of AB-10 (the sim logic is correct); it is a packaging defect. It blocks COMPLETE.

### HIGH

**None.**

### MEDIUM

#### MB-1 (NEW) — `buildPackaging.test.ts` validates the copy destination, not the runtime-read path — **CONFIRMED (Medium, test-quality, must land with AB-19)**

`buildPackaging.test.ts:9-10` checks `dist/server/migrations/` (where `copy-migrations.mjs` writes) but the runtime loader reads `dist/server/server/migrations/`. The test is green while the production migration path is broken — the exact mechanism that let AB-19 ship undetected. This is the build-layer analogue of the AB-16 lesson: a green packaging test that doesn't assert the property that actually matters (can the *built server* find and run migrations). The fix for AB-19 is not "done" until a test asserts the built server's loader path resolves and a fresh-DB build applies all 8 migrations (or `buildPackaging.test.ts` is corrected to assert SQL files exist at the path `db.js` will actually read, post-fix).

**Severity ruling: Medium (test-quality), must be fixed in lockstep with AB-19.**

The `call_up`/`dfa`/`waiver_claim` assertions are the real, sound gates; the `send_down` line is decorative. Test-quality only. Defer.
- **NB-2/CB-2 part-2 (CISO carried Low) — write-time raw-name scrub still deferred.** Non-exploitable under fixed-name-pool single-user-local threat model. May carry.
- **Iter-4 trade-path `is_on_25man` write-time hygiene (CISO carried Low).** Self-healed every tick by `checkRosterInvariant`. May carry.
