# Architect Evaluation — v0.2.0 Pre-Build (Eval 0)
**Date:** 2026-05-24
**Reviewer:** Architect
**Inputs:** `app-spec.md` (v0.1.0 baseline), `v0.2.0-app-spec-section.md`, `v0.2.0-test-spec.md`, `reports/architect-eval-5.md`, `build-rules.md`, direct source inspection of `baseball-dynasty/` v0.1.0 implementation.

## Scope summary

v0.2.0 is a substantial release. It changes seven things simultaneously:

1. LLM restructure (all decision calls removed, narrative-only calls retained).
2. Bug fix: owner death personality randomization.
3. Live minor league system (spring cuts, DFA, waiver wire, call-ups, send-downs, service time, prospect development).
4. Market dynamics & GM archetypes (analytics/old-school/balanced with concrete behavioral parameters).
5. In-season firing logic (formula-driven, evaluated mid-season).
6. News feed (live ticker + News tab with LLM headlines).
7. New API endpoints (`/api/waivers`, `/api/news`, `/api/players/:id/transactions`).

The spec leaves many cross-system interactions unspecified. This eval enumerates 14 contradictions, 28 spec gaps, 9 architecture risks, and issues 21 binding decisions. CISO, Adversary, and Developer must treat these as authoritative.

## 1. Cross-Document Contradictions

### C1 — GM personality model: v0.1.0 has THREE flat fields, v0.2.0 spec assumes ONE archetype field

**Conflict:** v0.1.0 schema (`migrations/001_init.sql:45-47`) defines three independent GM columns with strict CHECK constraints:
- `gm_philosophy ∈ {'win-now', 'rebuild', 'balanced'}`
- `gm_risk_tolerance ∈ {'conservative', 'moderate', 'aggressive'}`
- `gm_focus ∈ {'hitting', 'pitching', 'defense'}`

The v0.2.0 spec §"GM Archetypes" introduces three new archetypes — `analytics`, `old-school`, `balanced` — and uses them as a single behavioral axis driving waiver claims, trades, drafting, etc. **None of these archetype values exist anywhere in the v0.1.0 schema or `worldgen.ts`.** Test spec G3 ("Analytics GM teams have higher proportion...") and G4 ("Small market analytics GM teams") assume `analytics` is a queryable concept.

**Binding ruling:** Add a NEW column `gm_archetype TEXT NOT NULL DEFAULT 'balanced' CHECK (gm_archetype IN ('analytics','old-school','balanced'))` in migration 007. Derive its value during worldgen from existing `gm_philosophy` and `gm_risk_tolerance`:
- `analytics` if `risk_tolerance='aggressive' OR (philosophy='rebuild' AND market_size IN ('small','medium'))`
- `old-school` if `risk_tolerance='conservative' AND focus IN ('hitting','defense')`
- `balanced` otherwise

Backfill existing v0.1.0 teams using the same derivation in the migration. All v0.2.0 procedural decision logic reads `gm_archetype`. Original three flat columns are retained for backward compatibility (existing tests pass).

### C2 — Roster size: app-spec says "25-man active" but v0.2.0 introduces "40-man" and "DFA for 40-man space"

**Conflict:** App-spec `players` table lists `is_on_25man` and `is_on_40man`. v0.1.0 schema (`migrations/001_init.sql:76`) collapsed these to a single boolean `is_on_mlb_roster`. v0.2.0 spec talks about "DFA when a team needs 40-man space mid-season" — this requires distinguishing 40-man from 25-man.

**Binding ruling:** Migration 007 adds `is_on_25man INTEGER NOT NULL DEFAULT 0` while keeping `is_on_mlb_roster` as the 40-man flag (rename of intent only — schema column name stays `is_on_mlb_roster` to preserve v0.1.0 query compatibility). Going forward:
- `is_on_mlb_roster = 1` → on the 40-man (active or DFA-able)
- `is_on_25man = 1` → on the 25-man active roster (subset of 40-man)
- Both columns simultaneously 1 = "active MLB player"
- `is_on_mlb_roster=1, is_on_25man=0` = on 40-man but not active (DFA candidate, or in minors with an option)
- Both 0 with `minor_level != NULL` = minor leaguer not on 40-man

Spring cuts (D5 below) enforce `is_on_25man=1` for exactly 25 players per team. The validator `validatePostDraftRosters` is extended to assert this invariant.

### C3 — Free agency pool source: v0.1.0 has automatic offseason release; v0.2.0 spec adds non-tender, release, DFA, retirement — overlap unclear

**Conflict:** v0.1.0 `offseason.ts:runFreeAgencyStep` releases all players with `contract_years_remaining <= 0`. v0.2.0 spec introduces additional pathways: spring cuts (release), DFA → waivers → FA, non-tender → FA. The spec is silent on whether contract-expiration releases still happen in the offseason in addition to non-tender.

**Binding ruling:** All four pathways coexist and are ordered:
1. **Retirement** (offseason, age ≥ 40) — runs first, unchanged from v0.1.0
2. **Non-tender** (offseason, after retirement, before free agency) — analytics/small-market GMs cut players with projected_salary > $8M AND in arbitration year 3+
3. **Contract expiration** (offseason free_agency step) — `contract_years_remaining <= 0` → FA pool (existing v0.1.0 logic, unchanged)
4. **Spring training cuts** (start of regular season, immediately after offseason) — trim from 40-man to 25-man active; cut players go directly to FA pool (no waivers)
5. **DFA mid-season** (any time during regular season) — only path through waivers

### C4 — "Trade deadline at game 35" (app-spec) vs "Trade Deadline Logic (game 30-40 window)" (v0.2.0 §"Trade Deadline Logic")

**Conflict:** App-spec line 39 and v0.1.0 `season.ts:shouldFireTradeDeadline` fix the deadline at game 35. v0.2.0 §"Trade Deadline Logic" says "evaluate trade posture at game 30" and "Deadline trades happen at game 33-37 window."

**Binding ruling:**
- Posture evaluation (BUYER/SELLER/NEUTRAL) is computed at game 30 and persisted to a new column `trade_posture` on `teams`.
- Trade execution window is games 30-37. Trades may fire on any game tick in that window subject to the cooldown rules in D9.
- The hard "deadline passed" `transaction_type='trade_deadline'` marker still fires once when median team reaches game 35 (preserves v0.1.0 behavior). After game 35, no new deadline trades; only DFA/waiver/call-up activity continues.
- Test spec G7 "By game 30" wording is correct under this ruling.

### C5 — Test spec G7 "At trade deadline: at least 3 trades complete per season" vs spec narrative implying many trades

**Conflict:** Spec §"Trade Deadline Logic" suggests every contending team is a buyer trying to acquire veterans, and every rebuilder is a seller. With 20 teams that's potentially 10-15 trades. Test spec sets a floor of 3.

**Binding ruling:** The test floor of 3 is the minimum. Architecture must produce **3–12 trades per season** (a reasonable upper bound to keep the league economy stable). Implement a per-team max of 2 deadline trades and a league-wide cap of 12 to prevent runaway trade cycles. Document this as D9.

### C6 — Service time: app-spec says "172 games = 1 full year" / "6 years = FA eligibility"; spec §"Service Time" repeats this. But v0.1.0 already had a `service_time` INTEGER column ambiguously meaning "years"

**Conflict:** v0.1.0 `migrations/001_init.sql:80` defines `service_time INTEGER NOT NULL DEFAULT 0`. Worldgen (`worldgen.ts:290`) assigns `serviceTime = randInt(rng, 0, age - 22)` clearly in **years**. v0.2.0 spec requires **games** granularity for the 172-games-per-year accumulator + the 15-extra-game manipulation tactic.

**Binding ruling:** Migration 007 adds `service_time_days INTEGER NOT NULL DEFAULT 0` (1 game = 1 day for v0.2.0; per app-spec convention of one game-day = 10 games, this is the simplest model). The existing `service_time` column is preserved and treated as `service_time_years` (kept for v0.1.0 compatibility); a NEW derived column `service_time_years` is computed as `FLOOR(service_time_days / 172)`. To avoid name collision rename the legacy column via the migration to `service_time_legacy_years` (kept for any read paths still using it) and introduce both new columns. All v0.2.0 logic reads `service_time_days`. UI shows `service_time_days / 172` formatted as "X.YYY years."

Backfill in migration: `service_time_days = service_time_legacy_years * 172` for all existing players.

### C7 — Test spec G6 "After 10 games simmed: at least one minor leaguer has a rating change logged" vs spec §"Prospect Development" "ratings tick up 0-1 points per 10-game tick"

**Conflict:** Test asserts at least one ticks up. With only 0-1 rating change probability and a 10-game cadence, a single 10-game window could realistically yield zero rating changes if no eligible AA/A/Rookie player meets the coachability × work_ethic threshold.

**Binding ruling:** The test is correct as a floor — but implementation must guarantee it. Set the probability such that the expected number of rating ticks per 10-game window across all eligible prospects league-wide is ≥ 5. With ~400 minor leaguers in AA/A/Rookie aged 18-25, a per-prospect tick probability of `0.05 + coachability/100 × work_ethic/100 × coach_quality/10` should yield ≥ 5 ticks per window in expectation, satisfying the test floor with extremely high probability. Add a `rating_changes` event log (or use the existing `transactions` table with `transaction_type='dev_tick'`) to make the test verifiable.

### C8 — Spec §"Live Ticker" "Shows last 5 events, scrolls as new ones arrive" — test spec G9 just says "shows last 5 events, updates as sim runs." Conflict with poll-based architecture.

**Conflict:** v0.1.0 frontend polls `/api/state` every 2 seconds. The spec describes a "scrolling bar" implying real-time push. There is no WebSocket infrastructure.

**Binding ruling:** No new infrastructure. News ticker is rendered from the most recent 5 rows of `/api/news?limit=5`, fetched alongside the `/api/state` poll. The `/api/state` response gains a `lastNewsId` field; the client refetches news only when `lastNewsId` changes. "Scrolling" is a pure CSS animation in the ticker component, decoupled from data fetching.

### C9 — News feed event types vs existing `transactions` and `front_office_events` tables

**Conflict:** v0.1.0 already has two event-log tables: `transactions` (trades, signings, retirements, etc.) and `front_office_events` (manager/GM/owner changes). Spec §"News Feed" lists 15 event types — most overlap with existing tables. Spec is silent on whether news rows are derived from these existing tables or written to a new `news` table.

**Binding ruling:** Create a new `news_items` table (see D3). The existing tables remain the canonical record. News rows are written **at the same time** as the source-of-truth row (same transaction). The `news_items` row stores: `event_id, event_source_table, event_source_id, event_type, badge, headline_text (LLM or null until filled), is_headline_pending (bool), game_number, created_at`. This denormalization is justified because:
- Filtering by event-type-category in the UI is O(1) on indexed news_items
- LLM-fill batching needs a flat pending queue
- Game-result events ("GAME" badge) are explicitly EXCLUDED from news_items per spec ("game result … no LLM"); ticker reads game scores separately from `game_log`

### C13 — Spec §"Firing Threshold Formula" `base_games_under_500: 8 games` — but games are tracked as wins/losses, not "games under .500"

### C14 — Spec mentions "patient" owner personality, but v0.1.0 schema has only `meddling | hands-off | moderate`

**Conflict:** App-spec line 92 says `owner_personality (patient/win-now/meddling/hands-off)`. v0.1.0 `migrations/001_init.sql:51` CHECK constraint enforces `IN ('meddling','hands-off','moderate')`. The original app-spec wording uses "win-now" as an owner personality (likely meant for GMs) and "patient" which doesn't exist. v0.2.0 spec §"Firing Threshold Formula" references all four: `meddling, win-now, patient, hands-off`.

**Binding ruling:** Migration 007 expands the owner_personality CHECK constraint to include `win-now` and `patient`. Backfill: for existing rows, leave existing values intact (`moderate` rows stay `moderate`). The firing formula multiplier map adds:
- `meddling: 0.6x`
- `win-now: 0.8x`
- `patient: 1.5x`
- `hands-off: 2.0x`
- `moderate: 1.0x` (NEW — added because v0.1.0 already produced `moderate` owners)

## 2. Spec Gaps

### G1 — Spring training cuts: timing
Spec says "start of each season" but doesn't say exactly when. **Ruling:** Cuts run as the first event of `regular_season` phase, after the offseason completes and before game 1 of the new season is simulated. Cuts are gated on `current_game_number == 0` for the active season. After cuts complete, a one-time flag is set on the league row (`spring_cuts_done_season INTEGER`) to prevent re-running on server restart.

### G2 — Spring training cuts: order of release vs send-down
Spec says "release" OR "send-down" but doesn't specify priority. **Ruling:** Iterate roster from lowest overall rating up. For each player on 40-man not yet processed:
1. If overall < 45 AND potential = D AND age ≥ 24: release to FA
2. Elif blocked by ≥ 2 better players at same position with higher rating AND has options_remaining > 0: send down to AAA/AA
3. Elif blocked AND no options remaining: DFA → waiver wire
4. Else: keep on 25-man
Continue until 25-man = 25 exactly. If still > 25 after rules, force-DFA the next-lowest player.

### G3 — Spring cuts: what if a team has fewer than 25 players on the 40-man?
**Ruling:** Promote best available minor leaguers to the 40-man until 40 players exist. If still < 25 active possible, the validator (`validatePostDraftRosters` extended) injects replacement-level free agents to bring count to 25.

### G4 — DFA: who decides to DFA a player?
Spec says "when a team needs 40-man space" but doesn't define the trigger. **Ruling:** DFA fires only as a side effect of:
- Send-down trigger on a player with `options_remaining = 0` (per spec)
- Spring cuts overflow (G2 rule)
- Call-up needs 40-man slot and team is already at 40 (rare — call-up logic must pick a DFA candidate using GM archetype rules: analytics picks lowest projected WAR; old-school picks youngest minor leaguer with options)

There is no standalone "DFA decision" — DFA is always a consequence.

### G5 — Waiver wire: claim order tie-break
Spec says "reverse current standings order" but ties exist. **Ruling:** Tie-break by `runs_scored - runs_allowed ASC` then `team_id ASC` (matching the standings sort in `routes/standings.ts:12-20`).

### G6 — Waiver wire: how do we sim "3 games of sim time"?
**Ruling:** Each waiver entry has `claim_game_window_end INTEGER` = `current_game_number + 3` at the moment of DFA. On every game tick, the waiver-processing function checks: (a) for each entry with `claim_game_window_end == current_game_number`, evaluate claims; (b) for entries with `claim_game_window_end < current_game_number`, expire to FA. Claims are evaluated EXACTLY ONCE at expiration, not continuously.

### G7 — Waiver claim probability formula
Spec says "Small market analytics GMs actively monitor waivers — higher claim rate." Floats no formula. **Ruling:**
```
claim_score(team, player) =
  base_interest(archetype, player.overall) +
  position_need_bonus(team, player.position) +
  market_size_modifier(team.market_size, archetype) +
  contender_modifier(team.games_back) +
  random_jitter(seed)

base_interest:
  analytics:  player.overall ∈ [50, 65] → 0.7  else 0.2
  old-school: player.overall ≥ 60 AND age ≥ 28 → 0.5  else 0.2
  balanced:   player.overall ≥ 55 → 0.4  else 0.2

position_need_bonus: +0.3 if team has 0-1 players at that position
market_size_modifier:
  analytics + (small|medium): +0.3
  old-school + (large|mega): +0.1
  else: 0
contender_modifier: +0.2 if games_back <= 5 (acquiring depth)
random_jitter: uniform(-0.05, 0.05)
```
A team claims if `claim_score >= 0.6`. First eligible team in waiver order with score ≥ 0.6 wins. If multiple ties at the threshold, lowest-standing team wins (already the waiver order).

### G8 — Call-up evaluation cadence
Spec: "evaluated every 5 games." **Ruling:** On each game tick, check if `current_game_number - last_call_up_check_game >= 5` for the team. If yes, run call-up evaluator for that team and set `last_call_up_check_game = current_game_number`. Per-team check, not league-wide.

### G9 — Call-up "OPS < .580 over last 15 AB" — how do we track "last 15 AB"?
Spec assumes we have per-game batter lines. v0.1.0 only has season aggregate (`season_stats`). **Ruling:** Use a rolling window proxy. Track on `season_stats` table additional columns:
- `recent_ab INTEGER NOT NULL DEFAULT 0` (running 15-AB window)
- `recent_hits INTEGER NOT NULL DEFAULT 0`
- `recent_hr INTEGER NOT NULL DEFAULT 0`
- `recent_walks INTEGER NOT NULL DEFAULT 0`

After each game, increment recent_* by the game's contribution; when recent_ab > 20, subtract a portion proportional to the oldest day's contribution (kept as a deque via additional `recent_window_json` text column storing the last ~5 games). Simpler implementation: just track last-15-games OPS via game-by-game updates in the `season_stats` row. The implementer chooses simplest path that produces a verifiable OPS-over-recent-window number.

### G10 — Send-down: "Player must have option remaining" — what is an option?
Spec assumes the concept exists. **Ruling:** Each player has `options_remaining INTEGER DEFAULT 3` (set in migration 007 to 3 for all existing players). A send-down decrements options_remaining by 1. A call-up does NOT consume an option (only the round-trip down counts). If `options_remaining = 0`, send-down triggers DFA instead per spec.

### G11 — Service time: when does the clock start?
Spec says "Service time clock starts on first call-up date." **Ruling:** On call-up, set `first_mlb_call_up_game INTEGER` if NULL. Each game spent on the 25-man increments `service_time_days` by 1. Reset on send-down? No — service time accumulates monotonically across stints.

### G12 — Service time manipulation: "analytics GMs more likely to manipulate"
Spec says "keeping top prospects in AAA ~15 extra games to delay free agency by 1 year." **Ruling:** When call-up is triggered for an analytics GM AND target player has potential A or B AND target player has service_time_days in `[157, 172)` (close to a service year), delay call-up by uniform(10, 20) games. Track via `manipulation_delay_until_game` on the player row. Old-school GMs never delay.

### G13 — In-season firing: when does the manager-firing eval run?
Spec says "Evaluated every 5 games." **Ruling:** Per-team field `last_firing_check_game INTEGER`. On each game tick, if `current_game_number - last_firing_check_game >= 5` AND `current_game_number >= 12` (G14), run eval for both GM and manager (GM check is 10-game cadence separately tracked via `last_gm_firing_check_game`).

### G14 — Firing: when is the FIRST evaluation?
Not specified. **Ruling:** Per C13 ruling — earliest possible firing is game 12 (no earlier).

### G15 — Owner-fires-manager-directly: how is this distinguished from GM-fires-manager?
**Ruling:** Both produce a `front_office_events` row. Add a new column `actor TEXT NOT NULL DEFAULT 'system'` to `front_office_events` taking values `'gm'`, `'owner'`, `'system'`. The news headline distinguishes: GM-fires uses event_type=`manager_fired`; owner-fires-directly uses event_type=`manager_fired_by_owner`. Both render in the News tab under the FRONT OFFICE filter.

### G16 — Interim manager ratings -10: -10 on what?
v0.1.0 has no manager numerical ratings. Spec assumes some. **Ruling:** Add columns to `teams`: `manager_tactics INTEGER DEFAULT 50`, `manager_motivation INTEGER DEFAULT 50`, `manager_communication INTEGER DEFAULT 50` in migration 007 (random 35-75 at worldgen). Interim ratings = MIN(originals - 10, 25). When permanent manager hired, regenerate. This is the minimum surface needed to make the "-10" spec assertion verifiable.

### G17 — News feed: when is the LLM headline filled?
Spec says "queued and filled as sim runs." **Ruling:** Background job. On each game tick (or every 10th tick for cadence), check if any `news_items` row has `is_headline_pending = 1`. Batch up to 10 and call `callNewsHeadlinesBatch` (new LLM function). On response, write `headline_text` and set `is_headline_pending = 0`. If LLM circuit-breaker open or call fails, leave pending — retry on next tick. After 30 game ticks pending, write procedural fallback headline and mark non-pending.

### G18 — News feed: filter by team — how?
Spec says `GET /api/news?team=id`. **Ruling:** The `news_items` table denormalizes `team_id INTEGER` (and `secondary_team_id INTEGER NULL` for trades involving two teams). Filter is `WHERE team_id = ? OR secondary_team_id = ?`.

### G19 — Game result events: tickers shows score only, but does the News tab show them?
Spec §"News Tab" says "Full chronological feed" but the table of "Event Types That Generate News Items" lists "Game result" with LLM=No (just score). Ambiguous whether game results appear in the News tab at all. **Ruling:** Game results DO appear in the News tab but with the GAME badge and just the score (`{home_team} {h_score}, {away_team} {a_score}`). They are stored as virtual news items synthesized at read time, not stored in `news_items` table (avoid 25,000 rows over a multi-season run). News tab query: UNION ALL between `news_items` and a SELECT from `game_log` aliased to look like news_items.

### G20 — News feed: ticker behavior during turbo sim
Turbo can produce many events per second. **Ruling:** Ticker is rate-limited client-side. New events arrive on `/api/state` poll (every 2s). Client shows last 5 by `created_at DESC`. CSS scroll animates between fetches. No throttling on server.

### G21 — Spring cuts: do release transactions get news items?
Yes per spec — release is in the news feed table. **Ruling:** Spring cuts during pre-season generate news items with `event_type='spring_release'` and badge=TRANSACTION. To prevent news flood (40 teams × ~5 releases = 200 items), use a special batch: write all spring-release news items at once with `is_headline_pending = 1`, batch-LLM-fill them in groups of 10.

### G22 — Trade deadline: what if buyer needs at position P and only sellers offering P are interim-GM teams?
**Ruling:** Per spec, interim GMs skip trade evaluation. Buyer searches next-best seller. If no non-interim seller has position P, the buyer makes no trade for P that season. Document with a console.log("[trade] No non-interim seller for position P; buyer X passes").

### G23 — Test G7 "Non-tender: at least one player non-tendered per offseason across league"
**Ruling:** Force at least one. If natural non-tender candidates < 1 across all teams, find the player league-wide with highest `annual_salary` AND `service_time_days >= 4*172` AND age ≥ 30 AND on a small/medium market team, and force non-tender. Generates news event.

### G24 — `/api/news` pagination
Spec says `?limit=50` but no offset/cursor. **Ruling:** Add `?before=<news_id>` for cursor pagination. Default limit 50, max 200. Sort by `id DESC` (proxy for created_at DESC).

### G25 — `/api/players/:id/transactions` shape
Spec doesn't say. **Ruling:** Array of `{id, season_number, transaction_type, team_id, team_name, narrative, created_at, details}` from BOTH `transactions` and `front_office_events` (if the player was mentioned as departing_person or incoming_person). Sorted by `created_at DESC`. Limit 50.

### G26 — `/api/waivers` shape
Spec test G4 says "player_id, player_name, position, overall, claim_window_games_remaining" — that defines the contract. **Ruling:** Exactly those 5 fields per entry. Plus `dfa_team_id` and `dfa_team_name` so UI shows "DFA'd by X." Array of objects, no envelope.

### G27 — Server restart mid-spring-cuts
v0.1.0 has the checkpoint pattern via `offseason_step`. Spring cuts happen AFTER offseason in the new model. **Ruling:** Add `spring_cuts_done_season INTEGER` to `leagues`. Set after cuts complete. On restart, if `phase=regular_season AND season_number > spring_cuts_done_season` (or NULL), run cuts before processing any game tick.

### G28 — Server restart mid-waiver-window
Test G10 explicitly tests this. **Ruling:** Waiver window is stored in DB row (`claim_game_window_end`). On restart, no recomputation needed — game number advances normally and expirations fire correctly. No in-memory state.

## 3. Architecture Risks

### R1 — LLM call volume could exceed daily budget on a fresh league
v0.1.0 daily budget is 2000 calls (`services/llm.ts:28`). v0.2.0 adds news headline calls. Per-season estimate:
- 500 game ticks × ~3 news events/tick avg = 1500 events
- Batched 10 per call = 150 LLM calls/season for news
- + 1 season narrative
- + 50 transactions × 0.1 = 5 transaction flavor calls (batched)
- + 20 sampled draft picks
- Total: ~180 calls/season — well under 2000

**Mitigation:** Multi-season turbo runs could exhaust the budget. Add per-feature budget caps in `getLlmStatus`. If news budget exhausted, skip news LLM and use procedural template fallback for headlines (e.g., `"${team} released ${player}"`). Test G1 says "<100 calls" per season — this requires ALL news to be batched 10 per call AND season narrative ≤ 1 AND transaction flavors batched. Architecture must enforce.

### R2 — Game-tick performance regression
v0.1.0 game ticks at 100ms fast / 0ms turbo. v0.2.0 adds per-tick checks: trade deadline (existing), call-up (every 5 games per team = 4 teams/tick avg), send-down, firing (every 5/10 games per team), waiver expirations, news item insertion, prospect development (every 10 games). Risk: turbo slows from ~50 games/sec to <10 games/sec.

**Mitigation:** Each new check has its own cadence guard. Batch all checks into one transaction per tick. Use prepared statements for all new queries (project pattern). Benchmark: turbo must sim 1 season in < 10 seconds (existing was ~8s — allow 12s budget).

### R3 — SQLite write contention during turbo with new news_items writes
v0.1.0 turbo wraps draft + game inserts in transactions. News insertion adds N writes per game (1 game result + ~2 transaction news). 500 games × 3 inserts = 1500 extra writes per season turbo.

**Mitigation:** Wrap each game tick's news inserts in the same transaction as the game_log insert. Use the existing `db.transaction()` pattern from `worldgen.ts:183`. Bulk-insert spring cuts news as a single transaction (200 rows in one tx).

### R4 — `gm_archetype` derivation could destabilize existing balanced leagues
If we derive archetypes from existing GM rows, the distribution of archetypes is determined by the v0.1.0 worldgen RNG — could skew heavily analytics or old-school.

**Mitigation:** Verify in migration script. Log distribution to console.warn during migration. The derivation in C1 should yield roughly 1/3 analytics, 1/3 old-school, 1/3 balanced given v0.1.0's uniform sampling. If lopsided in practice, fall back to: assign archetype randomly using `seedFor('archetype_backfill', league.worldgen_seed)` to keep determinism.

### R5 — Service time accumulation needs every-game write
Every game increments `service_time_days` for every player on `is_on_25man = 1`. 25 players × 20 teams × 500 games = 250,000 row updates per season. Single UPDATE-per-player would dominate runtime.

**Mitigation:** One UPDATE-per-team per game (incrementing service_time for all is_on_25man players on that team), executed inside the existing game-tick transaction. Or: defer to a batch update at game-day boundary (every 10 games per existing app-spec `current_game_date` logic). Decision: batch every 10 games. Test G5 "Service time increments by 1 game for each game spent at MLB level" remains true on average.

### R6 — Firing logic during turbo with cooperative pause race
Spec §"Known Edge Cases" "Firing during turbo sim: all firings still fire correctly in turbo — do not skip firing checks during burst sim." Existing turbo path wraps games in big transactions; firing actions write to multiple tables (teams + front_office_events + news_items).

**Mitigation:** Firing checks run INSIDE the per-tick transaction. The transaction wraps: (1) game simulation writes, (2) firing checks + writes, (3) news_items inserts. Single transaction commit ensures atomicity. Cooperative pause check (`isPaused()`) happens between game ticks, not inside transactions — unchanged from v0.1.0.

### R7 — Roster size drift across seasons
Spring cuts enforce 25-man. But during the season, call-ups + send-downs + DFA could drift the 25-man count if the implementation has off-by-one bugs.

**Mitigation:** Add a server-side invariant check at the END of every game tick: `SELECT team_id, COUNT(*) FROM players WHERE is_on_25man=1 GROUP BY team_id HAVING COUNT(*) != 25`. If any row returns, log warning and run autoBalance. The cost is one SQL query per tick — negligible.

### R8 — Existing tests may break under new schema
v0.1.0 has 178 passing tests. Adding columns and changing semantics can fail tests that select `*` or assume specific column shapes.

**Mitigation:** Build-rules.md §"Universal Lessons" prohibits `select('*')` already — verify zero violations exist before v0.2.0 work begins. Run existing test suite after migration applies; any failures must be diagnosed (not silenced) by the Developer. The Architect ruling C2 (preserving `is_on_mlb_roster` column name) is specifically to minimize this risk.

### R9 — Interim GM at draft
If an interim GM is in place at the trade deadline, spec says skip trade eval. But what about the annual draft? Spec is silent. **Mitigation:** Interim GMs DO participate in the annual draft (procedural decisions only — no LLM flavor sampling for interim teams). Document under D10.

### D4 — Waiver wire state machine
No separate `waivers` table. Add columns to `players`:
```sql
ALTER TABLE players ADD COLUMN waiver_state TEXT NOT NULL DEFAULT 'none' CHECK (waiver_state IN ('none','dfa','waivers','expired'));
ALTER TABLE players ADD COLUMN dfa_team_id INTEGER REFERENCES teams(id);
ALTER TABLE players ADD COLUMN claim_game_window_end INTEGER;
```
(SQLite can't add CHECK constraints in ALTER — must use table swap. Migration 007 does this.)

State machine:
1. `none` → `dfa` (set by DFA action; also sets `is_on_25man=0`, `is_on_mlb_roster=1` still, `dfa_team_id=current team`, `claim_game_window_end = current_game + 3`)
2. `dfa` is shown on `/api/waivers` (filtered by `waiver_state IN ('dfa','waivers')`)
3. At expiration tick: evaluate claims (G7); if claimed → `none` + transferred to new team; if unclaimed → `expired` then `team_id=NULL, is_on_mlb_roster=0, waiver_state='none'` (FA pool)

### D5 — Call-up/send-down evaluation integration point

`runRosterMaintenance` does (in order, per team):

4. Call-up eval (per-team 5-game cadence)

### D6 — Firing logic integration
**Existing offseason firing path is RETAINED.** v0.2.0 adds an IN-SEASON firing layer that runs via `runRosterMaintenance` (D5). Offseason firings continue to handle end-of-season formal cleanup. Mid-season firings produce `front_office_events` rows with `actor='gm'|'owner'`. Same data model, different trigger source.

`firings.ts` module exports:
```typescript
evaluateManagerFiring(team, currentGame, seed) → boolean (did we fire?)
evaluateGMFiring(team, currentGame, seed) → boolean
applyManagerFiring(team, firedBy: 'gm'|'owner') → void  // inserts events, swaps in interim
applyGMFiring(team, firedBy: 'owner') → void
```

Threshold formula (C13 ruling):
```typescript
const baseThreshold = 8;
const patienceMod = { meddling: 0.6, 'win-now': 0.8, moderate: 1.0, patient: 1.5, 'hands-off': 2.0 }[owner_personality];
const aggMod = { aggressive: 0.75, moderate: 1.0, conservative: 1.2 }[gm_risk_tolerance];
const managerThreshold = Math.round(baseThreshold * patienceMod * aggMod);
const gmThreshold = Math.round(baseThreshold * patienceMod);
```

Owner-direct-fire (non-meddling): threshold = `Math.round(baseThreshold * patienceMod * 2.5)`. GM job_security -= 2.

### D7 — Prospect development tick integration
Runs from `runRosterMaintenance` every 10 games (i.e., when `current_game_number % 10 === 0`). Iterate all minor leaguers age 18-25 in AA/A/Rookie. For each, compute tick probability per G7 formula. If success, log a transaction (or dedicated `dev_ticks` table — KEEP IT SIMPLE: use transactions table with `transaction_type='dev_tick'` and narrative=null to make G6 test verifiable without UI noise).

Hot streaks: tracked in `season_stats.recent_*` columns (G9). A 5+ hit streak in 5 games sets a `prospect_visible=1` flag on the player row (added in migration 007), boosting call-up priority by +5 to overall in call-up scoring.

Bust mechanic: at exactly the offseason transition (handled in `offseason.ts:development` step), check players turning 26: if `minor_level IN ('AA','A','Rookie') AND potential IN ('C','D')`, set `potential='D'` AND `overall_rating = MIN(overall_rating, 65)`.

### D9 — Market dynamics / trade deadline integration
- `teams.trade_posture TEXT` column populated at game 30 by `evaluateTradePosture(team)`:
  ```typescript
  if (games_back <= 5) return 'BUYER';
  else if (games_back >= 10) return 'SELLER';
  else return 'NEUTRAL';
  ```
- Trade execution runs once per game tick in window [30, 37]. Per-team max 2 trades. League cap 12. Skip teams with `interim_gm = 1`.
- Matching algorithm:
  1. List active buyers (sorted by `games_back ASC`).
  2. For each buyer, list its top 2 position needs (most undermanned positions).
  3. For each (buyer, need_position): find sellers with surplus at that position, archetype-compatible. Generate package: 1 veteran (≥28yo, ≤2 contract years) for 1-2 prospects from buyer's AAA/AA. Archetype evaluates fairness; analytics demands more prospects, old-school demands more proven players.
  4. Execute via existing `transactions` table with `transaction_type='trade'`.
  5. After 3 failed match attempts per buyer, break.

## 7. v0.1.0 Lessons Applied

From `build-rules.md` §"Universal Lessons Learned":

1. **`data-testid` on third-party wrapper components silently dropped — wrap in div.**
   APPLIES TO: News ticker scrolling component (if using any animation library), all chart-like minors-stats components if they use a wrapper. Developer must wrap any third-party component in a `<div data-testid=...>` parent.

2. **Return `null` on error and `[]` for genuine empty results.**
   APPLIES TO: `/api/waivers`, `/api/news`, `/api/players/:id/transactions`. Empty result = `[]`. Internal error = HTTP 500 (caught by middleware).

3. **Never use `select('*')` in API routes — explicit field list.**
   APPLIES TO: All new endpoints. Audit during Phase 5/10/12 reviews.

4. **Responsive layouts can duplicate data-testid.**
   APPLIES TO: News ticker (which may render at top of every tab). Use ONE ticker instance in App.tsx, NOT per-tab. The `new-dynasty-button` duplicate from v0.1.0 backlog MUST be fixed (D13).

5. **When a fix is applied to one call in a file, check ALL similar calls.**
   APPLIES TO: When removing `callDraftPick` from non-sampled path, check ALL invocations. Confirmed via grep (Section 8): only 1 call site exists. Same for `callSeasonNarrative` (1 site, unchanged).
