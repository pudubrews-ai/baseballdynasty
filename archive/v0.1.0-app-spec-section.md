## Startup Sequence

1. `npm run dev` starts both server (ts-node/nodemon) and client (vite) concurrently
2. Server creates SQLite DB at `./data/dynasty.db` if not exists, runs migrations
3. Client loads at http://localhost:5173, proxies /api to http://localhost:3001
4. If no league exists in DB: show "Start New Dynasty" splash screen
5. `POST /api/league/new` triggers world gen → expansion draft phase begins
6. Draft room tab auto-activates, first pick clock starts

## World Generation Detail

### City Generation
Pull from `data/cities.ts`. Cities are fictional but feel real:
- Regions: Pacific Northwest, Southwest, Mountain West, Midwest, Great Plains, South, Southeast, Mid-Atlantic, New England, Canada, Mexico
- Each city has: name, state/province, region, market_size, population_hint
- No two teams share a region initially
- Examples of the vibe (do NOT use real city names): "Valmora", "Cresthaven", "Fort Dulce", "Lake Hensley", "Port Cascadia"

### Team Nickname Generation
Pull from `data/nicknames.ts`. Nicknames are single-word, evocative, and avoid real team names:
- Pool contains ~50 unique nicknames
- Categories: animals (Tigers, Falcons, Raptors), natural phenomena (Storm, Blaze, Thunder), industry/trades (Steelworkers, Miners, Ironmen), regional (Trailblazers, Pioneers, Voyageurs)
- Each team randomly assigned one nickname from pool
- No two teams share the same nickname

### Player Name Generation
Pull from `data/names.ts`. Pool should feel like a real pro sports league:
- ~35% US-born (diverse regions, all ethnicities)
- ~30% Latin American / Caribbean (Dominican Republic, Venezuela, Cuba, Puerto Rico, Panama, Colombia)
- ~15% East Asian (Japan, South Korea, Taiwan)
- ~10% Canadian
- ~5% European (Netherlands, Germany, UK, Italy)
- ~5% Other (Australia, West Africa, Mexico)

Names must be culturally appropriate to origin. Do not mix Japanese surnames with Latino first names, etc. Each origin has a separate first name and last name pool in `data/names.ts`.

### Rating Distribution
Use a seeded random with normal distribution:
- Overall talent: mean 55, std dev 12
- Elite players (overall 85+): ~2% of pool (~16 players)
- Star players (75-84): ~8% of pool (~64 players)
- Regulars (60-74): ~25% of pool (~200 players)
- Fringe/prospects (45-59): ~40% of pool (~320 players)
- Replacement level (<45): ~25% of pool (~200 players)

### Team Financial Generation
- Mega market: revenue $180-220M, payroll budget $140-170M
- Large market: revenue $120-160M, payroll budget $90-120M
- Medium market: revenue $70-110M, payroll budget $55-80M
- Small market: revenue $40-65M, payroll budget $30-50M

Luxury tax threshold: $180M. Teams over threshold pay 50% tax on excess.

## Front Office Personality Generation

Each team's GM has a personality profile stored in the `teams` table as a JSON object with three attributes:

**Philosophy:** One of:
- "Win-now" — prioritizes veteran talent, short-term success
- "Rebuild" — prioritizes prospects, long-term asset accumulation
- "Balanced" — mix of veteran and young talent

**Risk Tolerance:** One of:
- "Conservative" — avoids high-risk moves, values proven players
- "Moderate" — balanced approach
- "Aggressive" — willing to gamble on upside

**Focus:** One of:
- "Hitting" — prioritizes offensive talent
- "Pitching" — prioritizes pitching depth
- "Defense" — prioritizes defensive metrics

**GM Personality String Format:**
When building LLM context, serialize the personality JSON as: `"Philosophy: [VALUE], Risk Tolerance: [VALUE], Focus: [VALUE]"`

Example: `"Philosophy: Win-now, Risk Tolerance: Aggressive, Focus: Pitching"`

This string is passed to all LLM prompts requiring GM decision-making.

## Expansion Draft Logic

### Structure
- 30 rounds, snake order (round 1: picks 1-20, round 2: picks 20-1, etc.)
- All 800 generated players are in the draft pool
- First 15 rounds: players rated overall 50+
- Rounds 16-30: players rated overall 30-49

### Draft Order
Random for expansion draft (no standings to reverse). Assign by coin flip simulation.

### Position-Adjusted Value Formula
Used to rank available players for LLM context:
```
position_adjusted_value = overall_rating + position_scarcity_bonus + age_bonus

position_scarcity_bonus:
  C: +5
  SS: +4
  CF: +3
  SP (overall 70+): +6
  CL: +4
  All other positions: 0

age_bonus:
  age <= 24: +3
  age 25-28: +1
  age 29-32: 0
  age 33+: -2
```

### LLM Draft Pick Flow
For each pick:
1. Build context: team's current roster needs (count by position), GM personality string, top 10 available players by position-adjusted value
2. Call Claude Haiku: returns `{ pickIndex: 0-9, reasoning: "..." }`
3. If LLM call fails, times out, or returns invalid index (out of range 0-9 or unparseable JSON): fall back to best available by position need
4. Log pick to `draft_picks` table
5. Emit pick event to frontend via state polling

**Draft pick prompt template:**
```
You are the GM of the [TEAM_NAME]. Your philosophy is [PHILOSOPHY] and your approach is [RISK_TOLERANCE], with a focus on [FOCUS].
Your current roster needs by position (positions with 0 players are critical needs): [NEEDS_JSON]
The top 10 available players are: [PLAYERS_JSON]
Each player has: name, age, position, overall_rating, potential, key_strengths.
Pick the index (0-9) of the player you would draft and give a one-sentence reasoning.
Respond ONLY with valid JSON: {"pickIndex": N, "reasoning": "..."}
```

**LLM Rate Limiting:**
- Maximum 5 concurrent Claude Haiku calls
- Minimum 100ms delay between consecutive calls
- Queue all draft pick LLM calls; process queue with rate limit enforced

### Draft Room UI
- Shows draft board as grid: rows = rounds, cols = teams
- Completed picks fill in with player name + position
- On-the-clock team row highlighted in team color
- Pick card animates in with: player name, position, age, rating (shown as stars 1-5), reasoning text
- Speed: each pick takes 1.5s at normal speed, 200ms at fast, instant at turbo

## Game Simulation Detail

### Win Probability Formula
```
base_win_prob = 0.5
+ (starting_pitcher_rating - opp_starter_rating) * 0.003
+ (batting_lineup_avg - opp_lineup_avg) * 0.004
+ (bullpen_avg - opp_bullpen_avg) * 0.002
+ (home_field ? 0.04 : 0)
- (ace_fatigue_penalty)
- (injury_roster_penalty)
```
Clamp result to 0.15-0.85 (minimum 15% win probability, maximum 85%).

### Score Generation
- Winner score: random integer 3-12, weighted toward 3-6 (use triangular distribution with mode=4)
- Loser score: random integer 0 to (winner_score - 1)
- Blowouts (winner_score >= 8): occur in approximately 15% of games (enforce this percentage over a season)

### Box Score Generation
Distribute hits, HR, RBI to batters weighted by their ratings. Distribute IP, K, BB to pitchers weighted by their ratings.

**Internal Consistency Rules:**
- Total team hits >= total team runs (you can't score more runs than hits, except via walks/errors)
- Total team RBI <= total team runs + 2 (allow small margin for scoring anomalies)
- Starting pitcher IP: 4.0 - 9.0 innings
- Total IP for both teams = 9.0 innings each (no extra innings in v0.1.0)
- Winning pitcher must have pitched in the game (IP > 0)

### Notable Events (logged to game_log.notable_events JSON array)
Log the following events when they occur:
- Home run by a player with power rating > 80
- Shutout by starting pitcher (0 runs allowed, pitched >= 6 IP)
- Player injury during game (if injury_prone roll fires)
- Walk-off win
- Player hitting milestone: 100 career HR, 200 career HR, 1000 career K (pitchers), 2000 career hits

## Error Handling

### LLM Failures
- Any LLM call that fails, times out (8s timeout), or returns unparseable JSON: log warning to console, use procedural fallback
- Never block the sim on an LLM call timeout
- Rate limiting: queue LLM calls with max 5 concurrent, 100ms between calls

### Database
- All DB writes wrapped in try/catch; on error: log error to console with operation details, sim continues without crashing
- On server restart: read current league state from DB, resume from where sim left off
- Never lose a game result — write to DB before advancing sim tick

### Frontend
- If /api/state returns error: show "Reconnecting..." banner, retry every 3s
- Never crash on missing data — all player/team lookups should handle null gracefully

## Known Edge Cases

- **Draft pool exhausted before 30 rounds complete:** should not happen with 800 players and 20 teams × 30 picks = 600 picks total. But if it does: fill remaining picks with procedurally generated replacement-level players.
- **Team payroll exceeds budget after draft:** allowed — teams start over budget in expansion year only, no luxury tax in season 1.
- **LLM drafts duplicate player:** check pickIndex against already-drafted players before assigning; if duplicate, take best available at that position need.
- **LLM returns invalid JSON or out-of-range pickIndex:** fall back to best available by position need, log warning.
- **Zero games remaining in season when trade deadline fires:** skip trade deadline phase.
- **All playoff teams from one division:** valid, proceed normally.
- **Player age reaches 40:** auto-retire at end of season. Generate career summary for timeline.
